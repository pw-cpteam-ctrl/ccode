# WORKLOG — 입고안내 이미지 자동 제작 툴 (inbound-image-composer)

> 작성 규칙은 레포 루트의 `WORKLOG-작성-규칙.md` 참고. 하루 단위로, 그날 있었던 요청과
> 처리 과정을 비개발자도 읽고 이해할 수 있게 풀어서 적는 문서다.

## 2026-07-13

### 배포가 갑자기 멈춘 문제 (다른 툴 포함, 원인 파악까지)
- **무슨 문제였나**: 새 기능을 올려도 실제 서비스 페이지에 반영이 안 됨. 배포 목록을 봐도
  5시간 넘게 아무 변화가 없었음. 사용량 초과, 서버 장애 등 여러 가능성을 하나씩 확인.
- **확인해보니**: 사용량 초과도 아니었고 서버 자체 장애도 아니었음. 실제 원인은 웹사이트를
  올리는 서비스(Vercel)와 우리 저장소(GitHub)를 연결해주는 통로 자체가 끊겨서, 코드를
  올려도 "새로 배포해야 한다"는 신호 자체가 전달이 안 되고 있었던 것.
- **어떻게 처리했나**: 이 통로 연결 문제는 다른 작업자가 같은 날 별도로 찾아서 고쳐놓음
  (연결 방식을 더 안정적인 방식으로 교체). 우리 쪽에서는 원인을 좁혀서 확인하는 역할을 함.
- **결과**: 이후 커밋부터는 정상적으로 배포에 반영되는 것을 확인함.

### 그리드(사진 칸) 인식이 특정 상황에서 실패하는 문제
- **무슨 문제였나**: 두 가지 증상이 보고됨. ① 업로드한 스크린샷 가로 폭이 표준(약 1000px)이
  아니면 사진 칸을 하나도 못 찾음. ② 표준 폭이어도 5칸 중 1칸(예: 흰 배경 스티커/아이콘류
  상품)이 계속 통째로 안 잡힘.
- **확인해보니**: ①은 칸 위치를 찾는 로직이 "가로 폭이 항상 딱 1000px일 것"이라고 가정한
  채 고정된 좌표만 보고 있었기 때문 (`sampleCols`/`cardW`/`cardH`가 1000px 기준 절대
  좌표로 고정돼 있었음). 폭이 다르면 완전히 엉뚱한 위치를 보게 됨. ②는 칸 경계를 찾을 때
  사진 한 줄 중 딱 한 지점만 확인하는 방식이었는데 (`detectColStarts`가 첫 행 시작점+40px
  딱 한 지점만 샘플링), 하필 그 위치의 색이 흰색에 가까운 상품(흰 배경 아이콘 등)이 있으면
  "여기 상품 없음"으로 잘못 판단해서 그 칸을 그냥 건너뛰어버린 것.
- **어떻게 처리했나**: ①은 고정된 숫자 좌표 대신, 실제 업로드된 이미지 폭에 비례해서
  위치를 계산하도록 바꿈(폭이 얼마든 같은 비율로 알아서 맞춰짐) (`img.width` 기준
  비율로 좌표들을 다시 계산). ②는 딱 한 지점만 보지 않고, 여러 줄 · 여러 높이를 번갈아
  확인해서 가장 많은 칸이 찾아지는 결과를 채택하도록 바꿈 (여러 행 × 여러 세로 오프셋을
  교차 시도해서 검출된 열이 가장 많은 결과를 채택) — 한 상품 사진이 흰 편이어도 다른
  줄의 같은 위치 상품은 대부분 다르게 생겼으니 잡힐 확률이 훨씬 높아짐.
- **결과**: 실제로 문제와 똑같은 상황(가로 폭이 다른 이미지, 흰 배경 상품이 낀 5칸 배치)을
  직접 만들어서 테스트해봤고, 둘 다 정상적으로 다 잡히는 것까지 확인함.

### 최종 이미지에 헤더(매장 배너)가 안 나오는 문제
- **무슨 문제였나**: 최종 페이지 만들기에서 헤더 이미지를 골랐는데도 실제로 다운로드해보면
  헤더 없이(또는 텅 빈 이미지로) 나온다는 신고가 있었음.
- **확인해보니**: 사진과 글자, 그리고 헤더를 화면에 합쳐서 그리는 부분 자체는 직접 만든
  테스트 데이터로 끝까지 돌려봐도 문제없이 정상 동작했음. 그래서 코드 자체의 버그라기보다,
  "헤더"로 쓰이는 부분이 애초에 "업로드한 스크린샷에서 첫 번째 상품 줄보다 위쪽 부분을
  그대로 오려낸 것"이라는 점이 원인일 가능성이 높다고 판단함 (`GridDetect.cropCell(원본이미지,
  0, 0, 이미지폭, 첫행시작y)`로 오려서 저장) — 만약 올린 스크린샷 자체가 매장 배너 없이
  상품부터 바로 시작하는 캡처였다면, 오려낼 배너 자체가 없으니 당연히 비어 보이게 됨.
- **어떻게 처리했나**: 정확한 원인을 말로만 설명하는 대신, 최종 페이지 화면에 "지금 선택된
  헤더가 실제로 어떤 그림인지" 바로 보여주는 미리보기를 추가함 (`updateHeaderPreview()`
  함수 추가, 헤더 드롭다운을 바꿀 때마다 썸네일 갱신). 이러면 사용자가 직접 눈으로
  "비어 있는지 아닌지" 바로 확인할 수 있어서, 원인이 코드 문제인지 원본 캡처 문제인지
  스스로 판단할 수 있게 됨.
- **결과**: 미리보기 기능은 실제로 이미지를 넣고 확인해봤고 정상적으로 화면에 뜨는 것까지
  확인함. 다만 실제로 신고된 그 헤더가 정확히 왜 비어 있었는지(원본 캡처 문제였는지 다른
  이유였는지)는 사용자가 미리보기로 직접 확인해야 최종 확정 가능.

### "등급표에 이미 있는 상품인데도 계속 미분류로 뜨는" 문제
- **무슨 문제였나**: 이미 유명해서 등급표에 등록해둔 상품(예: 특정 인기 애니메이션)인데도,
  AI로 자동 채우기를 한 뒤에는 계속 "미분류(기타)"로 표시됨.
- **확인해보니**: 이 프로그램은 "브라우저에 저장해둔 설정값"과 "기본 제공 설정값"을 합쳐서
  쓰는 구조인데, 합치는 방식에 문제가 있었음 — 브라우저에 저장된 값이 하나라도 있으면
  기본 제공 값 전체를 무시하고 그냥 덮어써버렸음 (`loadDict()`에서
  `local.gradeTable || seed.gradeTable` 식으로, 로컬 값이 있으면 시드값 전체를 통째로
  교체하는 구조였음). 그래서 나중에 기본 설정에 새로 추가된 등급 정보가 있어도, 예전에
  저장해둔 값 때문에 계속 반영이 안 되고 있었던 것. (사전 데이터를 관리하는 여러 항목—
  등급표, 분위기 클러스터, 스토어 프로필, 상품 라인명—이 다 같은 방식으로 잘못돼 있었음)
- **어떻게 처리했나**: 저장된 값과 기본 제공 값을 "덮어쓰기"가 아니라 "둘 다 합쳐서 쓰기"
  방식으로 바꿈 (`mergeGradeTable()`/`mergeMoodClusters()` 함수를 새로 만들어 배열을
  합집합으로 병합) — 같은 항목이 겹치면 저장된 값을 우선하되, 기본 제공 값 중 저장 안 된
  나머지는 그대로 살아있게 함.
- **결과**: 등급표에 있는 상품을 다시 인식시켜보니 정상적으로 자동 분류되는 것을 확인함.

### AI가 미등록 상품의 성향까지 추측하도록 기능 추가
- **무슨 요청이었나**: 등급표에 없는(신규) 상품은 매번 사람이 직접 "남성향/여성향"을
  분류해야 해서 번거로움 — AI가 이것도 좀 도와줄 수 없냐는 요청.
- **어떻게 처리했나**: 등급표에 이미 등록된 상품은 원래대로 그 정보를 그대로 쓰고(AI가
  건드리지 않음), 등록이 안 된 신규 상품에 한해서만 AI가 사진과 상품명을 보고 남성향인지
  여성향인지 추측해서 미리 채워두도록 함 (AI 응답 항목에 `genderLean` 필드를 추가하고,
  `applyAiGenderLean()` 함수가 등급표에 없는 IP에만 이 값을 반영). 다만 "얼마나 잘 팔릴
  것 같은지(우선순위 등급)"는 AI가 판단할 수 있는 영역이 아니라서, 일단 기본값으로
  채워두고 최종 확인/조정은 여전히 사람이 하도록 남겨둠.
- **결과**: 새로 만든 기능이라 실제 사진 인식 결과로는 아직 확인 못 했고, 코드 로직만
  점검한 상태 — 다음에 실제로 AI 자동 채우기를 써볼 때 결과를 같이 확인해야 함.

### 화면 단계 통합 (미리보기 화면과 정렬 화면이 겹쳐서 헷갈리던 문제)
- **무슨 문제였나**: "전체 미리보기" 화면과 "순서 정렬" 화면이 따로 있었는데, 둘 다
  "카드를 마우스로 끌어서 순서 바꾸기" 기능을 똑같이 갖고 있어서 왜 화면이 나뉘어 있는지
  헷갈린다는 지적이 있었음.
- **어떻게 처리했나**: 두 화면을 하나로 합침 (화면 두 개를 각각 그리던 렌더 함수 호출을
  하나로 합치고, HTML도 두 화면분 내용을 한 화면 안에 병합) — 이제 미리보기, 삭제, 자동
  정렬, 수동으로 끌어서 순서 바꾸기를 전부 한 화면에서 처리한다. 전체 진행 단계도 5단계에서
  4단계로 줄음.
- **결과**: 실제로 화면을 눌러가며 확인해봤고, 한 화면에서 삭제/자동정렬/수동조정이 다
  정상적으로 동작하는 것을 확인함.

### 표 정리 화면 편의 기능 추가
- **무슨 요청이었나**: ① 같은 상품(IP)끼리 구분하기 어려움. ② 여러 항목을 한 번에 골라서
  태그나 분류를 한꺼번에 바꾸고 싶음.
- **어떻게 처리했나**: ① 같은 상품명을 가진 항목끼리 카드 테두리 색을 맞춰서 한눈에 묶여
  보이게 함 (`colorForIp()` 함수가 상품명 글자를 계산해서 정해진 색상 목록 중 하나를
  자동으로 골라줌). ② 빈 공간을 마우스로 끌면 여러 항목을 한 번에 선택할 수 있게 하고,
  선택한 항목들의 태그나 분위기 분류를 한 번에 바꾸는 버튼을 추가함 (드래그 선택 기능을
  다른 화면과 공용으로 쓰게 만들고, 선택 시 나타나는 일괄편집 툴바를 새로 만듦).
- **결과**: 실제로 여러 항목을 골라서 태그를 한 번에 바꿔보고, 골랐던 개수만큼 정확히
  바뀌는 것까지 확인함.

### 마우스로 여러 개 선택하는 범위가 화면 일부에서만 되던 문제
- **무슨 문제였나**: 마우스로 드래그해서 여러 카드를 한 번에 선택하는 기능이, 특정 네모난
  영역을 벗어나면 선택이 끊기는 것처럼 느껴져서 불편하다는 지적이 있었음.
- **확인해보니**: 선택 영역을 표시하는 사각형이 특정 영역 안에서만 그려지도록 만들어져
  있어서, 그 영역을 벗어나면 화면에 안 그려질 뿐 실제 선택 자체는 되고 있었음 — 그래도
  눈에 안 보이면 사용자 입장에선 "안 되는 것"과 다름없음.
- **어떻게 처리했나**: 선택 영역 사각형이 화면 전체 어디로 끌어도 잘리지 않고 계속 보이게
  바꿈 (선택 박스를 표시하는 화면 요소를 특정 영역의 자식이 아니라 페이지 전체(`body`)에
  바로 붙이는 방식으로 교체).
- **결과**: 화면 끝까지 끌어봐도 선택 영역이 잘리지 않고 끝까지 따라오는 것을 확인함.

### "AI 전송 미리보기" 버튼 정리
- **무슨 지적이 있었나**: 예전에 추가했던 "AI 전송 미리보기" 버튼이 원래 있던 검출 결과
  화면과 똑같아 보여서 왜 따로 있는지 모르겠다는 지적이 있었음.
- **어떻게 처리했나**: 별도 버튼을 없애고 (`previewAiCrops()` 함수와 관련 버튼 삭제),
  원래 있던 검출 결과 화면 안에 같이 표시되도록 합침 — 사진이 잘리는 범위(빨간 테두리)와
  AI에게 실제로 전달되는 글자 포함 범위(초록 점선)를 한 화면에서 동시에 볼 수 있게
  정리함 (검출 결과를 그려주던 함수에 초록 점선 박스 그리는 부분을 추가).
- **결과**: 화면에서 두 범위가 겹쳐서 잘 보이는 것을 확인함.

### 이전 작업 자동 저장 기능 추가 (탭을 실수로 닫아도 복구 가능)
- **무슨 요청이었나**: 다른 도구(insta-gen)에 있는 "최근 작업 5개 자동 저장" 기능을 이
  도구에도 넣어달라는 요청. 참고용 코드도 같이 전달받음.
- **어떻게 처리했나**: 탭을 닫거나 다른 탭으로 넘어갈 때, 그 시점의 표 데이터(상품명·가격·
  태그·순서 등)와 사진을 최근 5개까지 자동으로 저장해두고 (탭이 닫히거나 화면에서 안
  보이게 되는 시점을 감지해서 저장 함수를 호출, 저장은 브라우저 안에 있는 `localStorage`
  라는 저장 공간 사용), 시작 화면에서 눌러서 그 상태로 바로 돌아갈 수 있게 함. 다만 저장
  공간을 아끼기 위해 사진은 화질을 낮춰서 저장하기 때문에, 이 저장 기능으로 되돌린 경우
  최종 결과물의 사진 화질이 원본보다 살짝 떨어질 수 있다는 점을 화면에도 안내 문구로
  남겨둠 — 정상적으로 작업 중이면 이 기능을 거치지 않고 그대로 이어서 하면 되고, 어디까지나
  "실수로 탭을 닫았을 때"를 위한 안전장치임. 저장 공간이 부족해지면 오래된 기록부터 사진을
  지우고 글자 정보만 남겨서 계속 동작하게 해둠 (저장 시도가 실패하면 오래된 순으로 사진
  데이터를 비우고 재시도하는 폴백 로직 추가).
- **결과**: 저장 → 새로고침 → 복원까지 실제로 반복해봤고, 저장된 상품 정보/사진/태그가
  다 정확히 돌아오는 것과 자동으로 표 정리 화면으로 이동하는 것까지 확인함.

### "다시 검출" 버튼이 작동을 안 하는 문제
- **무슨 문제였나**: 1단계에서 크롭을 이미 확정한 사진에 대해 "다시 검출" 버튼을 눌러도
  아무 효과가 없는 것처럼 보인다는 지적이 있었음. 같은 화면에서 최종 결과물에 헤더(매장
  배너)가 여전히 안 나온다는 신고도 같이 있었음.
- **확인해보니**: 진짜 원인을 찾았음 — 한 번 크롭을 확정하고 나면 "확인 및 크롭" 버튼이
  영구히 눌러지지 않게 잠겨 있었음 (`confirmSourceCrop()`이 이미 확정된 소스면 그냥
  아무 일도 안 하고 끝나버림, 버튼도 계속 비활성 상태). 그래서 "다시 검출"을 눌러서 칸
  위치를 새로 찾아내도, 그 결과를 실제로 적용할 방법 자체가 없었던 것 — 겉보기엔 버튼이
  고장 난 것처럼 보이지만 실제로는 "재검출 결과를 반영할 다음 버튼이 원천적으로 막혀
  있던" 구조적 문제였음.
- **어떻게 처리했나**: 이미 확정된 사진이어도 "다시 검출"을 누르면, "이 소스에서 만든
  항목이 전부 삭제되고 새로 만들어지며 2단계 이후에서 직접 고친 내용은 사라진다"는 경고
  창을 먼저 띄우고, 동의하면 확정 상태를 풀어서 "확인 및 크롭"을 다시 누를 수 있게 함.
  다시 확정을 누르면 이 소스가 예전에 만든 상품 항목·사진·헤더를 정확히 찾아서 지우고
  (항목의 고유 번호를 기준으로 지우기 때문에, 그 사이 표를 정리하거나 순서를 바꿔놨어도
  엉뚱한 항목이 지워지는 일은 없음) 새 결과로 다시 채움.
- **결과**: 실제로 확정 → 항목 하나 직접 수정 → 다시 검출(경고 문구 확인) → 재확정까지
  전체 흐름을 돌려봤고, 상품 개수·헤더 개수가 중복되지 않고 그대로 유지되면서 예전
  항목/사진은 깨끗이 지워지고 새 항목으로 교체되는 것까지 확인함. 다만 헤더가 여전히
  안 보이는 원인 자체(원본 스크린샷에 매장 배너가 없어서인지, 다른 이유인지)는 이 수정과
  별개라서, 다시 검출 후에도 헤더 미리보기가 비어 보인다면 원본 캡처 범위를 다시 확인해야 함.

### 스크린샷을 파일로 저장하지 않고 바로 붙여넣기(Ctrl+V) 요청
- **무슨 요청이었나**: 캡처한 스크린샷을 파일로 저장했다가 다시 파일 선택창에서 골라
  올리는 과정이 번거로우니, 클립보드에서 바로 Ctrl+V로 붙여넣을 수 있게 해달라는 요청.
- **어떻게 처리했나**: 1단계 화면에서 Ctrl+V를 누르면, 클립보드에 이미지가 있는지 확인해서
  있으면 파일 선택으로 올린 것과 완전히 동일하게 처리되도록 추가함 (기존 "파일 선택으로
  이미지를 받으면 처리하는 경로"를 그대로 재사용). 화면 안내 문구에도 이 사용법을 추가함.
- **결과**: 가짜 이미지를 만들어 붙여넣기 동작을 그대로 흉내내서 테스트해봤고, 파일
  선택 없이도 정상적으로 이미지가 추가되는 것을 확인함.

### "다시 검출"이 진짜 아예 먹통이라는 재신고 — 배포 파일까지 직접 대조해서 원인 확정
- **무슨 문제였나**: 앞서 "다시 검출" 버그를 고쳤다고 안내했는데도, 팝업도 안 뜨고
  아무 반응도 없다는 재신고가 있었음. 실제 서비스 주소(`inbound-image-composer.vercel.app`)가
  맞는지, 개발 중인 코드와 실제로 보이는 사이트가 다른 게 아니냐는 의심도 함께 제기됨.
- **확인해보니**: 실제 서비스 사이트에서 파일 3개(메인 화면, 상품 정리 로직, 그리드 인식
  로직)를 전부 직접 내려받아서 지금 갖고 있는 코드와 한 글자도 다른 부분이 없는지
  전부 대조해봤음 — 완전히 동일했음. 즉 배포가 안 된 게 아니라 코드는 이미 최신 상태로
  나가 있었음. 그다음 실제 화면의 개발자 도구 콘솔 로그를 확인해보니, 프로그램이
  멈추거나 깨졌다는 증거(빨간 에러)는 없고 무해한 경고 두 줄뿐이었음. 그래서 진짜 원인은
  ① 이미 확정된 소스에서 재검출하면 뜨는 경고창이 있는데, 아직 확정 전 상태에서
  "다시 검출"을 누르면 애초에 경고창이 뜰 이유가 없다는 점, ② 같은 이미지를 같은 방식
  으로 다시 스캔하면 검출 결과가 이전과 똑같이 나올 수 있는데, 그럴 때 화면에 "방금
  뭔가 실행됐다"는 표시가 전혀 없었다는 점 — 두 가지가 겹쳐서, 실제로는 정상 동작
  하고 있는데도 사람 눈엔 "버튼이 완전히 안 먹는다"로 보였던 것.
- **어떻게 처리했나**: 재검출 결과가 이전과 같든 다르든 상관없이, 버튼을 누르는 즉시
  해당 사진 영역 테두리가 잠깐 초록색으로 강조되고 버튼 글자도 "✓ 재검출 완료"로
  1.4초간 바뀌었다가 원래대로 돌아오게 함 — 눈에 보이는 확인 표시를 추가.
- **결과**: 실제로 버튼을 눌러서 테두리 강조와 문구 변경이 즉시 나타났다가 시간이
  지나면 원래대로 돌아오는 것까지 확인함.

### "1404px, 2160px짜리 사진이 전부 이상하게 크롭된다"는 반복 신고 — 실제 원인을 찾아서 고침

### "텍스트는 유지하고 사진 순서만 바꾸고 싶다" — 잠금 모드 추가
- **무슨 요청이었나**: 이미 만들어둔 결과물(입고안내_6.jpg)처럼 IP명·가격 텍스트는
  그대로 두고 사진 순서만 바꾸고 싶은데, 페이지를 옮기다 보면 텍스트가 지워지는 것
  같다는 요청. 2단계쯤에 "텍스트 수정" ↔ "순서만 변경"을 켜고 끌 수 있는 체크박스를
  만들어서, 3단계에서는 순서만 바꿀 수 있게 해달라는 구체적인 기능 요청.
- **확인해보니**: 카드를 드래그해서 순서를 바꾸는 코드(`moveItemsBeforeTarget`)를
  직접 다시 읽고, 합성 데이터로 재정렬 전/후 값을 대조하는 테스트도 돌려봤음 —
  이 코드는 배열 안에서 항목의 위치만 바꿀 뿐 IP명·가격·태그 같은 텍스트 필드는
  전혀 건드리지 않음. "자동 정렬" 버튼이나 삭제·되돌리기 로직도 마찬가지로 텍스트를
  건드리지 않음. 즉 순서를 바꾸는 동작 자체가 텍스트를 지우는 버그는 없었음 — 다만
  이미 만들어둔 결과물을 새 원본으로 다시 올리면, 그 사진에서 새로 잘라낸 항목은
  이 도구가 사진 속 글자를 직접 읽어내지 못하니 처음부터 빈 칸으로 시작하는 게
  원래 동작이라, 그걸 "텍스트가 지워졌다"로 느끼셨을 수 있음.
- **어떻게 처리했나**: 요청하신 대로 2단계에 "🔒 텍스트 잠금 (순서만 변경)" 체크박스를
  추가함. 켜면 2단계 표의 모든 입력칸(IP명/가격/태그/등급)이 비활성화되고, 3단계에서도
  항목을 클릭해 수정하는 버튼과 삭제 체크박스가 비활성화됨 — 다만 카드를 드래그해서
  순서를 바꾸는 기능은 잠금 여부와 관계없이 항상 그대로 동작하게 해서, 잠금 켠 상태로
  3단계에 가면 정말로 "순서만" 바꿀 수 있음. 3단계 화면에도 지금 잠금 중이라는 안내
  배너를 띄움.
- **결과**: 잠금 켰을 때 2단계 입력칸 전부 비활성화, 3단계 수정 버튼·삭제 체크박스
  비활성화, 카드는 계속 드래그 가능, 실제로 드래그(재정렬 함수 호출)해도 텍스트 값이
  그대로 유지되는 것, 잠금을 끄면 다시 수정 가능해지는 것까지 자동화 테스트로 확인함.

### "AI로 정렬만 돕게 해줘, 근데 1단계 AI버튼이랑 헷갈리면 안 돼" — 저비용 정렬 보조 기능 추가
- **무슨 요청이었나**: 자동 정렬을 눌러도 결과가 안 바뀐 이유(등급표/클러스터에 없는

  "그럼 텍스트는 잠가둔 채로 AI 도움만 받아서 정렬을 더 잘 시키고 싶은데, 이걸 1단계
  AI버튼처럼 사진을 다시 보내는 방식으로 할지, 3단계에 새로 만들지, 어느 쪽이 토큰(비용)을

- **어떻게 판단했나**: 1단계 AI버튼(사진을 다시 읽어서 IP명·가격까지 새로 뽑아내는 방식)을
  그대로 3단계에 새로 만드는 것도, 1단계 걸 재사용하는 것도 둘 다 사진(이미지) 데이터를
  다시 보내야 해서 비용이 큼 — 게다가 이미 확정한 텍스트를 다시 덮어쓸 위험도 있음. 근데
  IP명은 이미 텍스트로 다 알고 있으니, 사진 없이 "IP명 글자만" 보내서 성향·분위기만
  판단받으면 이미지 토큰이 아예 안 들어서 훨씬 저렴하고, 응답에 IP명/가격/태그 자체가
  없으니 텍스트를 실수로 바꿀 방법도 없음 — 이 방식을 추천하고 그대로 구현함.
- **어떻게 처리했나**: 텍스트만 보내는 새 API(`/api/classify-ip`)와 3단계 "🏷️ AI 분류로
  정렬 보조" 버튼을 추가함. 이미 등급표(S/A)에 있거나 이미 분위기 클러스터에 들어있는
  IP는 물어볼 필요가 없으니 대상에서 빼고, 중복 IP도 한 번만 물어봐서 호출량을 최소화함.
  응답을 받으면 성향(subGrade)과 분위기 클러스터만 갱신하고 자동으로 정렬까지 적용함.
  1단계 버튼과 헷갈리지 않도록: 1단계 안내문에 "3단계에는 사진 재인식 없이 정렬만 돕는
  더 저렴한 버튼이 따로 있다"는 문장을 추가하고, 3단계에는 버튼 바로 아래 회색 설명
  문구로 "1단계 버튼과 다르다"는 걸 바로 보이게 적고, 버튼에 마우스를 올리면 나오는
  설명(툴팁)도 붙임.
- **결과**: 합성 데이터 + 가짜 API 응답으로 자동화 테스트를 돌려서, 등급표에 이미 있는
  IP는 정확히 호출 대상에서 빠지고, 응답에 따라 성향만 반영되고, IP명·가격·태그는
  정렬 전후로 완전히 그대로 유지되는 것까지 확인함.

### "잠금 상태에서 3단계 갔더니 AI 분류할 IP가 없다고 뜬다" — 안내 문구가 원인을 잘못 짚던 문제
- **무슨 문제였나**: 텍스트 잠금을 켠 상태로 3단계에서 "AI 분류로 정렬 보조"를 눌렀더니
  "새로 분류할 IP가 없다"는 팝업만 뜨고 아무것도 안 바뀐다는 신고. 사용자가 스스로
  "잠금 상태에서 IP명이 비어있어서 그런 것 같다"고 정확히 원인을 짚어줌.
- **확인해보니**: 정확히 그 말대로였음. 크롭을 확정하면 그 항목들의 IP명은 원래
  빈 칸으로 시작하는데(1단계에서 사진 속 글자를 자동으로 읽어내는 게 아니라서), 아직
  아무것도 안 채운 상태에서 텍스트 잠금부터 켜버리면 2단계에서 IP명을 입력할 방법이
  없어져서 계속 빈 칸으로 남음. 근데 AI 분류 버튼을 누르면 "새로 분류할 IP가 없습니다
  (전부 이미 등급표/클러스터에 등록돼 있어요)"라는, 실제 원인과 전혀 다른 문구가
  떠서 혼란스러웠던 것 — 진짜 이유는 "등록이 이미 다 돼서"가 아니라 "IP명 자체가
  비어서"였음.
- **어떻게 처리했나**: 후보가 0개일 때 이유를 두 가지로 구분해서 안내하도록 고침 —
  IP명이 비어있는 항목이 있으면 그 개수를 알려주고, 지금 텍스트 잠금 중이면 "2단계에서
  잠금을 먼저 해제하고 IP명을 채운 뒤 다시 시도하라"고 구체적으로 안내함. 전부 채워져
  있는데 정말 이미 등록된 경우에만 기존 문구를 그대로 씀.
- **결과**: 크롭 확정 직후 곧바로 잠금부터 켠 뒤 3단계에서 버튼을 누르는 순서로 실제
  재현해봤고, 새 안내 문구("IP명이 비어있는 항목이 5개라서... 잠금을 먼저 해제하고
  IP명을 채운 뒤 다시 시도하세요")가 정확히 뜨는 것까지 확인함.

### "이미 만든 이미지 순서만 바꾸고 싶은데 상품 사진 인식은 필요 없잖아" — 라벨 전용 재인식 추가
- **무슨 문제였나**: 바로 앞서 고친 "IP명이 비어서 AI 분류를 못 쓴다" 문제의 근본
  원인을 짚은 뒤, 사용자가 정확한 지적을 함: "결국 IP명을 읽으려면 토큰이 드는 거
  아니냐, 그럼 이미 만들어진 이미지에서 순서만 바꾸고 싶은 경우엔 상품 사진(캐릭터가
  뭔지 알아맞히는 것) 인식은 필요 없고 'IP명 라벨' 부분만 인식하면 되는 거 아니냐"는
  구체적인 아이디어를 제시함.
- **어떻게 처리했나**: 정확히 그 아이디어대로 새 기능을 만듦. 이 도구가 이미 만든
  결과물(사진 아래 IP명·가격 글자가 이미 렌더링돼 있는 카드)을 다시 소스로 올렸을
  때 전용으로, 사진 자체는 아예 잘라서 빼버리고(`cropLabelStripRegion`) 그 아래
  텍스트 라벨 줄만 잘라서 신규 API(`/api/ocr-label`)로 보냄. 이미 깨끗하게 렌더링된
  글자라 원본 스크린샷처럼 "글자가 흐리면 사진 보고 캐릭터 추측" 같은 복잡한 규칙이
  필요 없어서, 프롬프트도 "보이는 글자 그대로 옮겨 적기"로 훨씬 짧고 저렴함. 1단계에
  "🏷️ 라벨만 재인식" 버튼으로 추가함.
  이제 총 AI 버튼이 3개(1단계 AI로 채우기 / 1단계 라벨만 재인식 / 3단계 AI 분류로
  정렬 보조)가 됐는데, 처음 쓰는 사람이 헷갈리지 않도록 1단계 상단에 세 버튼의
  "언제 쓰는지·비용이 어느 정도인지"를 한눈에 비교하는 표를 추가함.
- **결과**: 합성 이미지로 두 크롭 방식의 실제 전송 이미지 크기를 직접 재봤음 —
  같은 한 줄(행) 기준으로 사진까지 포함한 기존 방식은 높이 560px, 라벨만 자른 새
  방식은 208px로 약 63% 줄어드는 것을 확인함. 버튼을 실제로 눌러서(가짜 API 응답
  사용) IP명·가격·태그가 정확히 채워지고, "AI 추정" 배지 없이 곧바로 확정 상태로
  들어가는 것까지 확인함.

### "드래그가 안 된다, 4단계엔 다 없어졌다" — 드래그 방식을 통째로 교체
- **무슨 문제였나**: 라벨 재인식으로 텍스트를 다 채운 뒤 3단계에서 순서를 바꾸려는데
  드래그가 아예 안 먹힌다는 신고. 4단계(최종 내보내기) 화면 캡처도 같이 보내면서
  "여기 다 없어졌다"고 함.
- **확인해보니**: 먼저 코드 리뷰만으로 판단하지 않고, 실제 마우스로 누르고-이동하고-
  떼는 동작을 그대로 흉내내서 재현해봤음. 지금까지 쓰던 방식(브라우저 기본 제공
  `draggable` 속성 + dragstart/dragover/drop 이벤트)은, 자동화 도구의 마우스 시뮬레이션
  으로도 재현에 실패할 만큼 입력 방식에 따라 시작 자체가 안 잡히는 경우가 있다는 걸
  확인함 — 반대로 마우스를 누르고(mousedown) 움직이고(mousemove) 떼는(mouseup) 저수준
  이벤트로 직접 만든 방식은 안정적으로 재현됨. 이 김에 코드를 다시 훑다가 완전히
  별개의 버그도 하나 더 발견함 — 3단계의 "빈 배경 드래그로 여러 카드 한번에 선택"
  기능이 내부 함수 호출에서 필수 인자 하나가 통째로 빠져 있어서 처음 만들어질 때부터
  단 한 번도 실제로는 작동한 적이 없었음. 4단계의 "다 없어졌다"는 건, 원래부터 4단계는
  순서 변경이 아니라 최종 이미지 생성/다운로드 전용 화면이라 3단계식 카드 미리보기/드래그
  자체가 없는 게 맞는 설계였음 — 다만 그 사실이 화면에 안내돼 있지 않아서 "사라졌다"로
  오해하기 쉬웠음.
- **어떻게 처리했나**: 카드 드래그를 브라우저 기본 드래그 API에서 마우스 이벤트 기반의
  직접 구현으로 완전히 바꿈. 항목이 아주 많을 때(예: 155개) 화면이 매우 길어지는데,
  기존 브라우저 기본 드래그가 공짜로 주던 "화면 끝까지 마우스를 가져가면 자동으로
  스크롤되는" 기능이 없어지지 않도록 그 동작도 직접 만들어 넣음. 마퀴 다중선택 버그도
  같이 고침. 4단계 화면 위쪽에 "여기는 순서 바꾸는 화면이 아니라 최종본을 만드는
  화면"이라는 안내 문구를 추가함.
- **결과**: 실제 마우스 누르고-이동하고-떼는 동작으로 단일 카드 드래그, 여러 페이지
  박스 경계를 넘는 장거리 드래그(자동 스크롤 포함), 텍스트 잠금 중에도 드래그는 되고
  수정 버튼은 막히는 것, 빈 배경 드래그로 여러 카드 선택되는 것까지 전부 재현 테스트로
  확인함.

### "다시 검출 눌러도 효과가 없다, 이 버튼이 대체 무슨 의미냐" + "헤더가 계속 사라진다"
- **무슨 문제였나**: 실제 매장 스크린샷(1463×7904px, 95개 셀)을 올렸는데 그리드가
  이상하고, "다시 검출"을 눌러도 전혀 효과를 못 본다는 신고. 없애지 말고 실제로 값어치를
  하게 고쳐달라는 요청. 4단계 헤더가 계속 없어진다는 지적도 같이 있었음.
- **확인해보니**: 이번엔 스크린샷을 붙잡고 추측하는 대신 코드부터 다시 봤음 — 그리고
  "다시 검출" 버튼의 근본적인 설계 결함을 코드만으로 확정할 수 있었음. `detectGrid()`는
  오직 이미지 픽셀만 보고 판단하는 완전히 결정론적인 함수라서, 같은 이미지에 대고
  몇 번을 다시 눌러도 항상 정확히 똑같은 결과만 나올 수밖에 없는 구조였음 — 즉 지금까지
  이 버튼은 "정상적으로 작동은 하지만, 애초에 아무것도 다르게 시도해볼 방법이 없어서
  다시 눌러봐야 의미가 없는" 상태였던 것. 합성 이미지(배경을 살짝 회색(232,232,232)으로
  만든 것)로 실제 재현해보니, 기본 판정 기준(흰색 임계값 240)으로는 그리드를 아예 하나도
  못 찾는데, 이 상태에서 "다시 검출"을 눌러봐야 100번을 눌러도 여전히 0개인 게 당연했음.
  헤더가 비어 보이는 문제도 비슷한 성격 — 헤더로 자를 높이를 자동 검출값(`rows[0]`)에만
  의존하는데, 이 값이 이미지마다 잘못 잡히면 사용자가 직접 고칠 방법이 아예 없었음.
- **어떻게 처리했나**: "다시 검출"을 누를 때마다 판정 기준(흰색 임계값, 사진 한 칸으로
  인정하는 세로/가로 길이 범위)이 다른 3가지 방식을 순환하며 시도하도록 바꿈 — 기본값이
  안 맞는 이미지에서는 이제 재검출을 누를 때마다 실제로 다른 결과를 시도해볼 수 있음.
  버튼 문구도 "다른 방식으로 재시도 (2/3)"처럼 몇 번째 방식을 시도했는지 보이게 바꿈.
  헤더는 소스 카드에 "헤더 높이(px) 직접 조정" 숫자 입력을 추가해서, 자동 검출이 틀려도
  사람이 직접 값을 넣어 파란 박스 미리보기로 확인하며 고칠 수 있게 함.
- **결과**: 일부러 기본 방식으로는 그리드를 아예 못 찾게 만든 합성 이미지로 재현해서,
  1회 재검출 후 다른 방식(임계값 225)으로 넘어가면서 정상적으로 그리드가 검출되는 것,
  변형이 0→1→2→0으로 순환하는 것, 헤더 높이를 직접 입력하면 실제 크롭 결과도 그 값
  그대로 나오는 것까지 전부 실제로 돌려서 확인함.

### "완전히 동일한 데이터는 갱신되지 않게 해줘" + "AI로 채운 것도 캐시에 남아?"
- **무슨 요청이었나**: 이전 작업 슬롯 목록에 내용이 똑같은 항목이 여러 개 쌓여 보이는 걸
  막아달라는 요청("완전히 동일한 데이터는 갱신되지 않게"), 그리고 AI로 채우기든 라벨만
  재인식이든 AI로 채운 데이터도 이 캐시에 같이 남는 게 맞는지 확인 요청.
- **확인해보니**: 탭을 닫거나 다른 탭으로 넘어갈 때마다 자동 저장이 실행되는데, 그 사이
  실제로 아무 작업도 안 했어도 매번 새 스냅샷을 만들어 목록 맨 앞에 추가하고 있었음 —
  내용은 완전히 같고 저장 시각만 다른 항목이 계속 쌓이는 구조적 문제였음. AI로 채운
  데이터 쪽은 코드로 직접 확인함 — "AI로 채우기"든 "라벨만 재인식"이든 결과는 전부
  `item.ip`/`price`/`tag`/`subGrade` 같은 일반 표 데이터에 그대로 쓰이고, 슬롯 저장
  함수(`snapshotState`)는 그 표 데이터 전체를 통째로 복사해서 저장하므로 AI로 채운
  값도 예외 없이 캐시에 포함됨.
- **어떻게 처리했나**: 새 스냅샷을 저장하기 직전에 저장 시각만 빼고 나머지 내용을 가장
  최근 슬롯과 비교해서, 완전히 똑같으면 새로 추가하지 않고 그냥 건너뛰도록 고침.
- **결과**: 똑같은 상태로 저장을 3번 연달아 시도해도 슬롯이 1개만 남는 것, 실제로 값을
  하나 바꾼 뒤 저장하면 정상적으로 슬롯이 하나 늘어나는 것, AI로 채운 IP명·가격·성향
  값이 저장된 슬롯 안에 그대로 들어있는 것까지 실제로 돌려서 확인함.

### "라벨인식 했는데 뭐야 이거, 우리식으로 바꿔야 할 거 아냐" — 지저분한 원문 그대로 나오던 문제
- **무슨 문제였나**: 원본 매장 스크린샷에 "라벨만 재인식"을 썼더니, IP명 자리에
  "[예약] GMG 콜렉션 17 지구연방군 세이라 마스 제복 버전ㅣ기동전사 건담"처럼 상품명
  전체가 그대로 들어가버렸다는 신고. 우리 방식(깨끗한 IP명)으로 나오게 고쳐달라는 요청.
- **확인해보니**: 이 버튼을 처음 만들 때 "이 도구가 이미 만든 결과물(사진 아래 이미
  깨끗한 IP명만 렌더링된 카드)을 재수입할 때 전용"이라고 전제하고, 그래서 "해석하지
  말고 보이는 글자 그대로 옮겨 적기"만 하도록 만들었었음. 근데 실제로는 지저분한
  원본 상품명이 그대로 있는 진짜 매장 스크린샷에도 이 버튼을 쓰고 있었음 — 그러면
  "그대로 옮겨 적기"가 곧 "상품명 전체를 그대로 베껴오기"가 돼서 전혀 쓸모가 없는
  게 당연했음.
- **어떻게 처리했나**: 1단계 "AI로 채우기"가 쓰는 것과 똑같은 IP명 추출 규칙(사전
  매칭, "|" 구분자 뒤 작품명 판단, 상품 라인명 걸러내기, VTuber 규칙, 태그/분위기
  클러스터/성향 판단)을 라벨 인식에도 그대로 적용하도록 바꿈. 사진이 없어서 "글자가
  흐리면 캐릭터 그림을 보고 유추해라"는 지시만 뺐고, 그 경우엔 확인 필요(⚠) 표시로
  남게 함. 결과적으로 사진만 안 보낼 뿐 인식 품질은 "AI로 채우기"와 동등해지고,
  원본 스크린샷/재수입 어느 쪽에 써도 깨끗한 IP명이 나오게 됨.
- **결과**: 가짜 API 응답(지저분한 원문 + 깨끗하게 추출된 IP명, 클러스터 매칭 1건,
  확신 없는 항목 1건 포함)으로 실제로 돌려서, 지저분한 원문이 그대로 안 남고 추출된
  IP명이 반영되는 것, 클러스터 자동 편입되는 것, 등급표에 이미 있는 IP(예: 에반게리온)
  는 성향 추측을 건너뛰고 사전 값을 우선하는 것까지 전부 확인함.
