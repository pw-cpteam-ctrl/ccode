## 1. GitHub Actions로 서버 단 시크릿 스캔 추가
지금 있는 `.claude-hooks/pre-commit`은 **로컬 훅**이라 `--no-verify`로 우회 가능하다.
GitHub Actions는 GitHub 서버에서 도니까 로컬에서 우회해도 소용없음. push할 때마다
diff를 다시 한번 스캔해서, 뭔가 새어나갔으면 자동으로 알림(이슈 생성 등)을 보내게
만들면 "로컬 훅 = 1차 방어선, GitHub Actions = 2차 방어선(뚫려도 잡힘)" 구조가 됨.

## 2. 과거 커밋 히스토리 전체 시크릿 스캔 (1회성)
`pre-commit` 훅은 **오늘 이후 커밋만** 막는다. 혹시 이전에 실수로 커밋됐다가 지금은
지워진 것처럼 보이는 시크릿이 git 히스토리 어딘가에 남아있을 수 있음(한 번이라도
push됐으면 히스토리에 흔적이 남음). 전체 히스토리를 한 번 훑어서 확인해두면 안심할 수 있음.

## 3. 허브 페이지(index.html) 링크 상태 점검
루트 `index.html`에 등록된 카드들의 링크(GitHub Pages URL, Vercel URL 등)가 실제로
살아있는지 점검. Vercel 프로젝트는 배포 URL이 바뀌거나 무료 티어 한도 초과로 죽어있을
수 있는데, 직접 매번 클릭해보지 않는 이상 모름.

## 4. 프로젝트 상태판 추가
`sns-report-scraper`처럼 "킵" 상태인 프로젝트가 있는데, 시간이 지나면 이게 죽은
건지 잠깐 멈춘 건지 헷갈릴 수 있음. 루트 README나 별도 파일에 "프로젝트별 현재
상태(운영중/킵/실험중)"만 한눈에 보이는 표를 만들어두면, 나중에 다른 세션이 재개할
때 맥락 파악이 빨라짐.

## 5. 문서-코드 드리프트 점검 (CHANGELOG/README 최신화 여부)
각 프로젝트의 CHANGELOG/README가 실제 코드 상태와 맞는지, git log와 대조해서 놓친
게 없는지 점검. 문서가 낡으면 나중에 "이 기능이 있었나 없었나" 헷갈리는 원인이 됨.

## 6. 프로젝트별 누수 점검
특히 `product-sns-formatter`처럼 **서버 코드 + 외부 API 호출**이 있는 프로젝트는
"쓸모없는 반복 호출, 불필요하게 큰 프롬프트, Vercel 배포 낭비" 같은 누수가 생기기
쉬운 유형이라 우선순위로 점검할 만함.
