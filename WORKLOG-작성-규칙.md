# WORKLOG.md 작성 규칙

`CHANGELOG-작성-규칙.md`를 기반으로 하되, 목적이 다르기 때문에 형식과 문체를 더 강하게
조정한 규칙입니다.

**CHANGELOG와의 차이**
- CHANGELOG: "버전 단위"로 묶어서, 무슨 기능이 추가/변경됐는지 짧게 요약하는 문서
- WORKLOG: "날짜 단위"로, 그날 무슨 요청이 있었고 무엇을 어떻게 처리했는지 하루 흐름 그대로
  더 자세히 풀어서 적는 문서 — **보고용**이므로 비개발자가 처음부터 끝까지 읽고 "무슨 일이
  있었는지, 왜 그랬는지, 지금은 어떻게 됐는지" 다 이해할 수 있어야 함

---

## 📋 기본 포맷

```markdown
## YYYY-MM-DD

### [주제 한 줄 — 예: 그리드 인식 오류]
- **무슨 문제/요청이 있었나**: (사용자가 실제로 겪은 상황이나 요청한 내용을 있는 그대로)
- **확인해보니**: (원인이 뭐였는지, 왜 그런 문제가 생겼는지 — 쉬운 말로)
- **어떻게 처리했나**: (뭘 바꿨는지 — 쉬운 말로)
- **결과**: (지금은 어떻게 달라졌는지, 확인은 어떻게 했는지)

### [다음 주제]
- ...
```

같은 날짜 안에서도 주제(요청 건)별로 소제목을 나눠서 적는다 — 하루에 여러 건이 있었으면
그만큼 소제목을 늘린다.

---

## 🎯 문체 규칙 (CHANGELOG보다 강화)

### 원칙: 쉬운 설명을 먼저 쓰고, 코드 용어는 괄호로 병기한다
CHANGELOG는 "1줄 쉬운 설명 + (기술: ...)" 2줄 구조지만, WORKLOG는 **한 문장 안에서
"쉬운 설명 (코드 용어 그대로)" 형태로 병기한다.** 순서는 항상 쉬운 설명이 먼저, 코드
용어는 맨 뒤 괄호 안에 원문 그대로(함수명/변수명 등 축약 없이) 붙인다. "기술:" 같은
라벨은 붙이지 않는다 — 괄호 자체가 "이게 그 코드 용어다"라는 표시 역할을 한다.

**형식**: `쉬운 말로 풀어쓴 설명 (코드에서 실제로 쓰인 용어)`

**예시**
- 예전에 저장해둔 설정이 하나라도 있으면, 새로 추가된 항목이 있어도 무시하고 예전
  것만 쓰던 문제 (`loadDict()`가 로컬 저장값 있으면 시드 전체를 덮어씀)
- 사진 한 줄 중 딱 한 군데만 보고 칸 위치를 판단하던 방식 (`detectColStarts`가 한
  지점만 샘플링)
- AI가 답변할 때 "성향(남성향/여성향)"도 같이 대답하도록 항목을 하나 추가함
  (`RESULT_SCHEMA`에 `genderLean` 필드 추가)
- 저장 공간이 부족하면 오래된 기록부터 사진을 빼고 글자만 남겨서 계속 저장되게 함
  (localStorage 용량 초과 시 폴백 처리)

**주의**
- 쉬운 설명 문장만 읽어도 무슨 얘기인지 100% 이해되게 쓴다 — 괄호 안 코드 용어는
  "덧붙이는 참고"일 뿐, 이해에 필수인 정보를 괄호 안에만 숨기면 안 된다
  (괄호를 통째로 가리고 읽어도 뜻이 통해야 함)
- 괄호 안 코드 용어는 사실 확인용이니 정확하게 — 실제 코드에 있는 이름 그대로 쓰고,
  대충 지어내지 않는다

### 그 외 CHANGELOG 규칙은 그대로 적용
- 음슴체/명사형으로 끝내기 ("~함", "~됨", "~수정", "~추가"), 존댓말 금지
- 각 줄 마침표 없음
- "그냥", "음...", "뭔가" 같은 표현 금지
- 심볼: `→`(흐름), `—`(설명 연결) 자유롭게 사용

## 📐 소제목 구조 규칙

- 소제목은 그 날 있었던 "요청 건" 단위로 나눈다 (기능 단위가 아니라 **대화/요청 단위**)
- 소제목 이름은 사용자가 겪은 증상이나 요청 문구에 가깝게 붙인다
  (예: "5번째 칸이 안 잡히는 문제", "이전 작업 캐시 기능 추가")
- 한 요청 건 안에서 여러 개를 고쳤으면 그 안에서 다시 하위 항목(bullet)으로 나눈다
- 실제로 재현/검증까지 했으면 "결과"에 그 사실을 반드시 남긴다 (예: "실제로 다시 만들어서
  테스트해보니 정상 작동함" 등) — 재현 못 했거나 확인 못 한 부분이 있으면 그것도 숨기지
  않고 적는다 ("직접 확인은 못 했고 코드상으로만 점검함" 등)

## 🔽 접기(토글) 규칙

문서가 길어질수록 스크롤이 늘어나므로, `<details>`/`<summary>` 태그로 날짜 단위를
접어서 관리한다. 전체 문서 안에 토글은 항상 딱 2개만 있다 — 개별 날짜마다 따로
접지 않는다.

- **토글1 (금일 날짜, 펼친 상태로 시작)**: 오늘 작성 중인 날짜 섹션 하나만 담는다.
  `<details open>`로 열어둬서 방금 작업한 내용이 클릭 없이 바로 보이게 한다.
- **토글2 (금일 이전 전체, 접은 상태)**: 오늘보다 이전 날짜는 전부 하나로 묶어서
  이 토글 안에 넣는다. `<details>`(open 속성 없음)로 기본 접힌 상태로 둔다.

**형식**
```markdown
<details open>
<summary>2026-07-17 (오늘)</summary>

## 2026-07-17

### [주제]
- ...

</details>

<details>
<summary>이전 기록 (2026-07-16 이전)</summary>

## 2026-07-16

### [주제]
- ...

## 2026-07-15
...

</details>
```

**날짜가 바뀔 때(다음날 새로 작성 시)**: 어제까지 토글1 안에 있던 날짜 섹션을
토글2(이전 기록) 안으로 옮기고, 오늘 날짜로 새 토글1을 새로 만든다. 즉 토글1은
항상 "오늘 날짜 단 하나"만 담고 있어야 한다.

`<summary>` 다음 줄은 반드시 빈 줄을 하나 넣는다 — 안 그러면 그 안의 마크다운
(리스트, 굵게 등)이 렌더링 안 되고 그냥 텍스트로 나온다.

## 📝 작성 순서

1. 날짜 헤더 작성
2. 그날 있었던 요청/이슈를 소제목 단위로 나열
3. 각 소제목마다 "무슨 문제 → 확인해보니 → 어떻게 처리 → 결과" 4단 구성으로 작성
4. 코드 용어(함수명/변수명 등)가 등장하는 곳마다, 그 앞에 쉬운 설명이 먼저 오고
   코드 용어는 맨 뒤 괄호 안에 병기돼 있는지 다시 훑어보고 정리
5. 처음부터 끝까지 다시 읽어보면서, 비개발자가 이해 못 할 부분이 있는지 마지막 점검
6. 오늘 날짜 섹션이 토글1(펼침) 안에 있는지, 이전 날짜들이 전부 토글2(접힘)
   안으로 들어가 있는지 확인 — 날짜가 바뀐 첫 작성이면 어제 섹션을 토글2로 옮기기

## 🎨 다른 프로젝트 적용 시

파일명은 `WORKLOG-{프로젝트명}.md` 형태로 프로젝트 폴더 안에 둔다 (예:
`inbound-image-composer/WORKLOG-inbound-image-composer.md`) — CHANGELOG와 같은 이유로
프로젝트명을 파일명에 붙여서 나중에 검색/구분이 쉽게 한다. 이 규칙 문서(`WORKLOG-작성-규칙.md`)
자체는 레포 루트에 하나만 두고 모든 프로젝트가 공유한다.
