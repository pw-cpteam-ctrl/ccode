# WORKLOG.md 작성 규칙

`CHANGELOG-작성-규칙.md`를 기반으로 하되, 목적이 다르기 때문에 형식과 문체를 더 강하게
조정한 규칙입니다.

**CHANGELOG와의 차이**
- CHANGELOG: "버전 단위"로 묶어서, 무슨 기능이 추가/변경됐는지 짧게 요약하는 문서
- WORKLOG: "날짜 단위"로, 그날 무슨 요청이 있었고 무엇을 어떻게 처리했는지 하루 흐름 그대로
  더 자세히 풀어서 적는 문서 — **보고용**이므로 비개발자가 처음부터 끝까지 읽고 "무슨 일이
  있었는지, 왜 그랬는지, 지금은 어떻게 됐는지" 다 이해할 수 있어야 함

---

## 📋 기본 포맷

```markdown
## YYYY-MM-DD

### [주제 한 줄 — 예: 그리드 인식 오류]
- **무슨 문제/요청이 있었나**: (사용자가 실제로 겪은 상황이나 요청한 내용을 있는 그대로)
- **확인해보니**: (원인이 뭐였는지, 왜 그런 문제가 생겼는지 — 쉬운 말로)
- **어떻게 처리했나**: (뭘 바꿨는지 — 쉬운 말로)
- **결과**: (지금은 어떻게 달라졌는지, 확인은 어떻게 했는지)

### [다음 주제]
- ...
```

같은 날짜 안에서도 주제(요청 건)별로 소제목을 나눠서 적는다 — 하루에 여러 건이 있었으면
그만큼 소제목을 늘린다.

---

## 🎯 문체 규칙 (CHANGELOG보다 강화)

### 원칙: 코드 용어를 원칙적으로 쓰지 않는다
CHANGELOG는 "1줄 쉬운 설명 + (기술: 함수명/변수명 등 기술 용어)" 2줄 구조를 쓰지만,
WORKLOG는 **그 괄호 안의 "기술 설명"조차 일상 언어로 다시 풀어써야 한다.** 코드를 모르는
사람이 괄호까지 포함해서 전부 읽어도 막히는 부분이 없어야 한다.

**금지**: 함수명, 변수명, 파일명, 라이브러리/기술 스택 이름, HTTP/API/스키마 같은 용어를
본문에 그대로 노출하는 것. 정 언급해야 하면 아주 짧게, 비유나 쉬운 표현으로 바꿔서 쓴다.

**변환 예시**

| ❌ 코드 용어 그대로 | ✅ WORKLOG용 표현 |
|---|---|
| `loadDict()`가 로컬 저장값 있으면 시드 전체를 덮어씀 | 예전에 저장해둔 설정이 하나라도 있으면, 새로 추가된 항목이 있어도 무시하고 예전 것만 쓰던 문제 |
| `detectColStarts`가 한 지점만 샘플링 | 사진 한 줄 중 딱 한 군데만 보고 칸 위치를 판단하던 방식 |
| `RESULT_SCHEMA`에 `genderLean` 필드 추가 | AI가 답변할 때 "성향(남성향/여성향)"도 같이 대답하도록 항목을 하나 추가 |
| localStorage 용량 초과 시 폴백 처리 | 저장 공간이 부족하면 오래된 기록부터 사진을 빼고 글자만 남겨서 계속 저장되게 함 |

### 그 외 CHANGELOG 규칙은 그대로 적용
- 음슴체/명사형으로 끝내기 ("~함", "~됨", "~수정", "~추가"), 존댓말 금지
- 각 줄 마침표 없음
- "그냥", "음...", "뭔가" 같은 표현 금지
- 심볼: `→`(흐름), `—`(설명 연결) 자유롭게 사용

## 📐 소제목 구조 규칙

- 소제목은 그 날 있었던 "요청 건" 단위로 나눈다 (기능 단위가 아니라 **대화/요청 단위**)
- 소제목 이름은 사용자가 겪은 증상이나 요청 문구에 가깝게 붙인다
  (예: "5번째 칸이 안 잡히는 문제", "이전 작업 캐시 기능 추가")
- 한 요청 건 안에서 여러 개를 고쳤으면 그 안에서 다시 하위 항목(bullet)으로 나눈다
- 실제로 재현/검증까지 했으면 "결과"에 그 사실을 반드시 남긴다 (예: "실제로 다시 만들어서
  테스트해보니 정상 작동함" 등) — 재현 못 했거나 확인 못 한 부분이 있으면 그것도 숨기지
  않고 적는다 ("직접 확인은 못 했고 코드상으로만 점검함" 등)

---

## 📝 작성 순서

1. 날짜 헤더 작성
2. 그날 있었던 요청/이슈를 소제목 단위로 나열
3. 각 소제목마다 "무슨 문제 → 확인해보니 → 어떻게 처리 → 결과" 4단 구성으로 작성
4. 코드 용어가 하나라도 남아있는지 다시 훑어보고 전부 일상 언어로 치환
5. 처음부터 끝까지 다시 읽어보면서, 비개발자가 이해 못 할 부분이 있는지 마지막 점검

## 🎨 다른 프로젝트 적용 시

파일명은 `WORKLOG-{프로젝트명}.md` 형태로 프로젝트 폴더 안에 둔다 (예:
`inbound-image-composer/WORKLOG-inbound-image-composer.md`) — CHANGELOG와 같은 이유로
프로젝트명을 파일명에 붙여서 나중에 검색/구분이 쉽게 한다. 이 규칙 문서(`WORKLOG-작성-규칙.md`)
자체는 레포 루트에 하나만 두고 모든 프로젝트가 공유한다.
