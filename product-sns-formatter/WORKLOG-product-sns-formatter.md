## 2026-07-24

### 상품 1건씩만 처리하던 화면을, 하루치 여러 상품을 한 화면에서 다루는 구조로 통째로 개편
- **무슨 문제/요청이 있었나**: 실제로는 하루에 여러 상품을 처리하는데, 화면은 한 번에 상품
  1개만 붙여넣고 변환하는 구조였음. 페이지를 옮기지 않고 그날 상품 전체(정보 입력 → 원고
  생성 → 캐러셀 이미지 제작)를 한 화면에서 끝낼 수 있게 다시 짜달라는 요청. 실제 구현 전에
  레이아웃 후보 여러 개를 눈으로 먼저 비교하고 확정한 뒤에 진행함.
- **확인해보니**: 레이아웃 후보 3가지(세로로 쌓기/카드로 나열/좌측 목록+우측 작업창)를
  실제 화면처럼 만들어서 비교해본 결과, "왼쪽에 그날 상품 목록, 오른쪽에 고른 상품의 작업
  화면" 방식이 제일 낫다고 확정됨.
- **어떻게 처리했나**: 화면을 좌측 상품 목록 / 우측 작업 화면 구조로 새로 짬. 상품마다
  ①상품정보(공식정보+캐릭터 메모, 기존에 쌓아둔 실제 게시물 157건 자료에서 검색해서 바로
  가져다 쓸 수 있음) → ②원고(기존 비교 화면·교정 내역·글자수 표시 그대로 재사용) →
  ③캐러셀 이미지 순서로 배치. 원고는 실제 판매 링크가 들어간 것을 기본으로 하고, "인스타용
  복사" 버튼을 누르면 그 링크 부분만 안내 문구로 자동으로 바꿔서 복사되게 함 (`insVersion()`
  함수가 정규식으로 링크 줄만 치환). 상품 목록과 입력 내용은 브라우저에 자동 저장되게 함
  (`localStorage`), 사진 원본은 용량이 커서 별도 저장소에 둠 (`IndexedDB`).
- **결과**: 실제 화면을 자동으로 조작해보는 방식(상품 추가/전환/입력/검색)으로 검증함,
  정상 동작.

### 캐러셀 이미지, 실제로 만들어보니 주황 틀이 상품을 너무 많이 가리는 문제
- **무슨 문제/요청이 있었나**: 사진을 올리면 정해진 비율(1080x1350, 4:5)로 꽉 채워 자동으로
  잘라주고, 미리 만들어둔 주황색 틀 이미지를 맨 위에 얹어 완성하는 기능을 새로 만들었는데,
  실제로 써보니 틀 위쪽(로고 들어간 부분)이 생각보다 두꺼워서 상품 사진 윗부분이 많이
  가려짐.
- **확인해보니**: 사진을 자를 때 기준을 "캔버스 전체 크기"로 잡고 있어서, 어차피 안 보이는
  (틀에 가려지는) 부분까지 사진에 억지로 채워 넣느라 필요 이상으로 확대되고 있었음. 실제
  틀 이미지 파일을 픽셀 단위로 분석해서 진짜로 보이는 영역(투명한 부분)의 정확한 좌표를
  알아냄.
- **어떻게 처리했나**: 사진을 자르는 기준을 "캔버스 전체"가 아니라 "틀에서 실제로 사진이
  보이는 영역"으로 바꿈 (`FRAME_INNER` 좌표 기준으로 `cover-fit` 계산). 추가로 여러 장 중
  대표로 지정한 1장에만 어두운 그라데이션과 글자를 넣고, 나머지 장은 필요하면 틀 자체를
  꺼서 사진이 안 가려지게 하는 기능도 넣음.
- **결과**: 일부러 사진 맨 위에 표시가 되는 테스트 이미지를 만들어서, 가려지는 비율이
  기존 방식보다 실제로 줄어드는 것을 직접 재봐서 확인함 (기존 0% 노출 → 개선 후 2.66%
  노출). 완전히 안 가려지는 건 아니라서, 그래도 가리면 사진 위치를 손으로 옮겨서 조정
  가능.

### 사진 여러 장 순서를 못 바꾸는 문제 → 마우스로 직접 끌어서 옮기는 방식으로 추가
- **무슨 문제/요청이 있었나**: 캐러셀 사진을 여러 장 올렸을 때 순서를 바꿀 방법이 없었음.
  처음엔 화살표 버튼(◀▶)으로 만들었는데, 써보니 너무 불편하다며 마우스로 직접 끌어서
  순서를 바꾸는 방식으로 바꿔달라는 요청 (참고할 기존 코드도 같이 전달받음).
- **어떻게 처리했나**: 화살표 버튼을 없애고, 사진을 마우스로 누른 채 다른 사진 위로 끌어다
  놓으면 그 자리로 순서가 바뀌는 방식으로 새로 만듦. 순서를 실제 화면의 이동 칸 위치가
  아니라 사진마다 가진 고유 값으로 계산해서 (`slide.key` 기준 이동), 대표로 지정해둔 사진이나
  지금 편집 중인 사진이 순서를 바꿔도 엉뚱한 걸 안 가리키게 같이 맞춰줌.
- **결과**: 마우스로 직접 끌어서 옮기는 걸 자동으로 재현해서, 순서가 실제로 바뀌는 것까지
  확인함.

### "나머지 사진 틀 제거" 버튼이 사진을 통째로 지워버리는 것으로 오해해서 잘못 만듦
- **무슨 문제/요청이 있었나**: "대표 사진 빼고 나머지는 틀 제거" 기능을 요청받았는데, 처음엔
  "나머지 사진 자체를 지우는 것"으로 잘못 이해해서 그렇게 만들어버림. 실제 의도는 사진은
  그대로 두고 틀(액자 이미지)만 안 보이게 해달라는 것이었음.
- **어떻게 처리했나**: 사진을 지우는 대신, 사진마다 "틀 있음/없음" 표시만 따로 갖게 해서
  (`slide.noFrame` 값), 버튼을 누르면 대표 사진 빼고 나머지는 이 표시만 켜지도록 고침. 사진
  자체는 전혀 건드리지 않음.
- **결과**: 버튼을 눌러도 사진 개수는 그대로 유지되고, 대표가 아닌 사진만 틀이 빠지는 것을
  픽셀 단위로 직접 확인함 (틀이 있어야 할 자리의 색을 검사).

### 틀을 한 번 제거하면 다시 켜지지 않는 버그
- **무슨 문제였나**: 위 기능으로 틀을 제거한 뒤, 대표 사진을 다른 걸로 바꾸면 새로 대표가 된
  사진에 틀이 안 생기고 계속 빈 채로 남는 문제. 그리고 특정 사진 하나만 다시 틀을 켜고 싶어도
  그럴 방법이 없었음.
- **확인해보니**: 대표 사진을 바꾸는 버튼이 "대표"라는 값만 바꾸고, 그 사진의 "틀 있음/없음"
  표시는 그대로 내버려두고 있었음. 그래서 예전에 꺼져 있던 사진이 새로 대표가 돼도 꺼진
  채로 남아있었던 것.
- **어떻게 처리했나**: 대표로 지정할 때 그 사진의 틀을 무조건 다시 켜지도록 고치고, 추가로
  사진마다 개별적으로 틀을 껐다 켰다 할 수 있는 버튼을 새로 만듦.
- **결과**: 대표를 바꿔도 새 대표에 틀이 정상적으로 생기는 것, 개별 버튼으로 틀을 껐다가
  다시 켰을 때도 정상적으로 돌아오는 것까지 화면을 직접 조작해서 확인함.

### B2B 사이트에서 상품을 자동으로 가져오는 기능 착수
- **무슨 문제/요청이 있었나**: 상품정보를 사람이 일일이 손으로 옮겨 적는 게 아니라, 회사
  B2B 사이트에서 자동으로 긁어와달라는 요청. 화면엔 "B2B에서 가져오기" 버튼만 있고 실제
  연동은 안 돼 있던 상태였음.
- **확인해보니**: 이 사이트는 로그인이 필요하고, 로그인 자체는 사람이 직접 눈으로 보고
  하는 방식이 안전하다는 걸 이전에 만든 비슷한 도구(다른 사이트 자동 수집 도구)에서 이미
  확인해둔 상태였음. 그리고 이 로그인 과정은 화면이 있는 컴퓨터에서만 할 수 있어서, 지금
  작업하는 원격 서버에서는 대신 해줄 수 없다는 것도 확인함.
- **어떻게 처리했나**: 새 프로젝트(`b2b-scraper`)를 만들어서, 로그인 세션 저장, 사이트 구조
  확인용 정찰 스크립트, 사진 다운로드, 결과 정리까지 필요한 부분을 미리 짜둠. 그리고
  실제 사이트 연동이 되기 전에도 확인해볼 수 있게, 화면 쪽 "B2B 가져오기" 버튼을 안내
  문구만 뜨던 상태에서 실제로 파일(사이트에서 긁은 결과 + 사진)을 골라서 상품 목록에
  채워 넣는 기능으로 먼저 완성함.
- **결과**: 실제 사이트 로그인·정찰은 사람이 직접 컴퓨터에서 진행해야 하는 부분이라 아직
  못 함(다음 순서로 넘김). 그 앞단인 "가져온 결과를 화면에 채워 넣는 기능"은 가짜 결과
  파일로 직접 넣어보는 방식으로 확인함, 정상 동작 (상품 제목·정보·사진 개수까지 전부
  정확히 반영됨).

### B2B 스크래퍼, 실제 쓸 사람(비개발자 팀원) 기준으로 실행 방식 다시 정리
- **무슨 문제/요청이 있었나**: 위 B2B 스크래퍼는 처음엔 "터미널 열어서 로그인 스크립트 먼저
  실행 → 엔터 → 스크래핑 스크립트 실행" 방식으로 짜여 있었는데, 실제로 쓸 사람은 코딩을
  전혀 모르는 팀원이라 이 방식은 안 된다는 요청. 매번 검은 창 열어서 명령어 치는 방식(예전
  메가하우스 스크래퍼 방식)은 금지, 목표는 "바탕화면 아이콘 더블클릭 → 크롬에서 로그인만 →
  끝."
- **어떻게 처리했나**:
  - 로그인 스크립트와 스크래핑 스크립트를 하나로 합침 — 이제 스크래핑 스크립트 하나가
    "로그인 세션이 남아있나 확인 → 없거나 만료됐으면 크롬 로그인 창을 띄우고 기다림 → 있으면
    바로 스크래핑"까지 한 흐름으로 처리함. 로그인 완료도 터미널에서 엔터 치는 방식 대신,
    로그인 후 페이지 주소가 바뀌는 걸로 자동 감지하게 바꿈.
  - 아이콘 더블클릭 하나로 "Node 설치 확인 → (최초 1회만) 필요한 파일 자동 설치 →
    스크래퍼 실행"까지 되는 실행 파일(`run.bat`, 맥은 `run.command`)을 추가함. 팀원은 이
    파일을 더블클릭만 하면 됨 — 타이핑 없음.
  - 최초 설치를 가볍게 하기 위해, 브라우저까지 통째로 내려받는 방식(`playwright`) 대신
    팀원 컴퓨터에 이미 깔려 있는 진짜 크롬을 그대로 쓰는 방식(`playwright-core`)으로
    바꿈 — 처음 설치할 때 수백MB를 추가로 받을 필요가 없어짐.
- **결과**: 팀원이 겪을 과정이 "더블클릭 → (필요할 때만) 크롬에서 로그인 → 완료 문구 확인"
  세 단계로 줄어듦. 다만 진행상황을 보여주는 검은 콘솔 창 자체는 뜨는데(입력은 필요 없음),
  이걸 완전히 없애려면 별도 프로그램 포장이 필요해서 이번 규모엔 과하다고 판단해 지금
  방식으로 마무리함. 실제 사이트 정찰·연동은 여전히 로컬 컴퓨터에서 사람이 진행해야 하는
  부분이라 그대로 대기 중.

### 상품 설명에 섞인 잡담을 "오타"라고 잘못 설명하던 문제, 알고 보니 대사였던 경우
- **무슨 문제/요청이 있었나**: "어때, 예쁘게 나와?" 같은 상품 설명이 아닌 말이 입력에 섞여
  있으면 결과에서는 잘 빠지는데, 그 이유를 "의미 없는 오타라서 지웠다"고 잘못 설명함. 근데
  알고 보니 이 문장은 잡담이 아니라 피규어 캐릭터가 실제로 하는 대사였고, 이런 대사를 홍보
  글 맨 앞에 넣는 게 원래 흔한 방식이었음.
- **확인해보니**: AI에게 "오타 교정"과 "상품과 무관한 문장 제외"를 구분해서 알려주는 내용
  자체가 없어서 아무 이유나 갖다 붙인 것으로 보였음. 그리고 실제 계정에 올라간 과거 게시물
  157건을 받아서 살펴보니, 캐릭터 대사를 큰따옴표로 시작하는 게 드물지 않은 관행(약 17%
  정도)이라는 것도 확인됨.
- **어떻게 처리했나**: 오타 교정과 무관한 문장 제외를 구분해서, 무관한 문장을 뺄 땐 정확한
  이유를 적도록 정리함. 그리고 맨 앞에 큰따옴표로 감싼 문장이 있으면 잡담이 아니라 대사이니
  그대로 살리도록 규칙에 새로 추가함. 대사가 문장부호 없이 어정쩡하게 끝나면 마침표를 붙여
  마무리하도록도 같이 정리함.
- **결과**: 이후 테스트에서 이유 설명이 정확해졌고, 대사도 잘 살아남는 것 확인함.

### 브랜드를 미리 고르게 해달라는 요청
- **무슨 문제/요청이 있었나**: 원래는 세 브랜드(부시로드/메가하우스/굿스마일) 규칙을 한꺼번에
  넣어주고 AI가 알아서 맞는 걸 고르게 했는데, 정확도가 떨어지니 사용자가 입력 전에 브랜드를
  직접 고르게 해달라는 요청.
- **확인해보니**: 규칙 세 개를 한 번에 다 넣어주다 보니 서로 헷갈릴 여지가 있었음.
- **어떻게 처리했나**: 화면에 브랜드 선택 버튼 3개를 추가하고, 고른 브랜드의 규칙만 골라서
  넘기도록 바꿈. 버튼 모양도 처음엔 작고 어색해서, 화면 폭에 맞춘 큼직한 버튼으로 바꿨다가
  나중엔 가운데 정렬 + 각각 독립된 박스 모양으로 다시 다듬음. 이 중 건수가 제일 많은
  굿스마일을 기본값으로 미리 선택해둠.
- **결과**: 실제로 화면에서 버튼 눌러가며 확인함, 정상 동작.

### 실제 계정 게시물 데이터로 굿스마일 규칙을 다시 만든 일
- **무슨 문제/요청이 있었나**: 굿스마일 브랜드 규칙이 예전에 추측으로 만들어둔 거라 실제
  게시물이랑 많이 다르다는 지적이 있었고, 실제 계정에서 뽑은 과거 게시물 목록(엑셀 표)을
  받아서 그 기준으로 다시 정리해달라는 요청.
- **확인해보니**: 실제 게시물을 살펴보니 여러 차이가 드러남 — 글자수 제한이 실제보다 훨씬
  빡빡하게 잡혀있었고, 해시태그 개수/조합도 실제와 다르게 정해져 있었고, "아직 판매는 안
  하고 공개 발표만 하는 글" 양식이 아예 빠져있었음. 이모지는 게시물 표 자체에서 대부분
  빠져있어서, 이건 표를 뽑아준 도구가 놓친 것뿐이고 실제로는 담당자가 알아서 넣는 것으로
  확인됨.
- **어떻게 처리했나**: 실제 게시물 패턴에 맞춰 규칙을 통째로 다시 작성함 (글자수 제한 완화,
  해시태그 개수 실제에 맞춤, 발표 전용 양식 추가, 이모지는 AI가 손대지 않고 입력에 있는
  그대로 두도록 정리).
- **결과**: 이후 테스트에서 실제 게시물과 비슷한 형태로 나오는 것 확인함. 다만 세부적으로
  안 맞는 부분은 이후에도 계속 발견돼서 아래 "반복 테스트" 항목에서 계속 다듬어나감.

### 예전 광고 초안들로 반복 테스트하며 잡은 문제들 (여러 건)
- **무슨 문제/요청이 있었나**: 오타도 많고 지금 양식과도 다른, 1년 전쯤 실제로 썼던 초안을
  여러 개 넣어보면서 결과가 이상한 부분을 하나씩 짚어주는 방식으로 여러 차례에 걸쳐 진행됨.
- **확인해보니 + 어떻게 처리했나** (건별로):
  - 장식용 기호(이모지)가 스펙 줄 말고 다른 자리(작품 소개 줄, 설명 문장 안)에서는 자꾸
    사라짐 → 결과 전체에 기호 보존 원칙이 적용되도록 다시 정리
  - 제목 칸이 비어있을 때 캐릭터 이름만 겨우 채워 넣거나, 반대로 이미 제대로 된 제목인데도
    단어를 하나 더 붙여서 이상하게 중복시킴 → 제목이 진짜로 비어있을 때만 보강하고, 이미
    내용이 있으면 절대 손대지 않도록 정리
  - 문장이 이미 자연스러운데도 억지로 "이름 (버전)" 줄과 "~로 등장!" 줄, 이렇게 두 줄로
    어색하게 쪼개던 문제 → 문장을 억지로 안 쪼개고 자연스럽게 이어쓰도록 정리하되, 모바일
    에서 읽기 좋게 줄바꿈은 하도록 함
  - 마무리 문장이 "~만나보세요", "~등장합니다" 등 제각각으로 끝나던 문제 → 실제 계정은
    예외 없이 "~등장!"으로 끝난다는 걸 확인하고 그렇게 고정
  - 구매 링크나 스펙(높이/발매일) 앞에 붙는 기호가 한 줄로 합칠 때 사라지거나 다른 항목
    기호로 바뀌던 문제, 링크 앞 콜론 표시가 통째로 사라지는 경우도 있었음 → 항목마다 원래
    기호를 각자 그대로 유지하고 콜론도 항상 남기도록 정리
  - 입력에 있지도 않은 대사를 "더 생생해 보이려고" 마음대로 지어내서 넣은 심각한 사례
    발견 → 대사는 입력에 있는 것만 옮기고 절대 새로 짓지 않도록 강하게 금지
  - 원래 한국어로 된 대사를 갑자기 일본어로 바꿔서 써버린 사례 발견 (정확한 원인은 알 수
    없으나, 이 분야 콘텐츠 자체가 일본 원작이 많다 보니 AI가 "원래 일본어였을 것"이라고
    착각한 것으로 추정) → 지침으로 강하게 금지하는 것과 별개로, 없던 일본어 글자가 결과에
    나오면 아예 화면에 내보내지 않고 자동으로 막아버리는 이중 안전장치를 추가함
  - 원문에 마침표로 문장이 나뉘어 있으면 무시하고 억지로 한 문장으로 붙이거나, 반대로
    마침표가 하나도 없는데 새로 마침표를 만들어서 쪼개버리는 문제가 둘 다 있었음 → 원문에
    있는 문장 경계는 존중하되, 없는 경계는 새로 만들지 않도록 양쪽 다 정리
  - "정보를 지우지 말고 문법만 고쳐라"고 너무 좁게 정해놔서, 정보 손실 없이 표현만
    다듬는 것까지 막혀버림 → 정보는 그대로 유지하되 표현이나 문장 구조는 자유롭게 다듬어도
    되도록 완화
  - 같은 조사가 겹쳐서 어색한 문장(예: "~모습으로 ~시리즈로 등장")이 계속 안 고쳐짐 →
    "시리즈" 뒤에서만 다른 조사를 쓰도록 구체적으로 못박음
  - 복잡한 설명에서 같은 내용을 두 번 반복해서 쓰고, 두 캐릭터의 역할까지 뒤바꿔 쓴 심각한
    사례 발견 → 실패했던 사례를 그대로 예시로 넣어서 최대한 방지하도록 정리했지만, 문장이
    복잡할 때 AI가 헷갈려서 생기는 문제라 완전히 막힌다고 장담은 못 함
- **결과**: 매번 실제로 재테스트해가며 하나씩 확인함. 이 과정에서 나온 문제들은 한 번에 다
  안 잡히고 여러 차례 되풀이해서 조금씩 잡아나갔음. 특히 마지막 "중복 서술" 문제는 규칙
  문구만으로 100% 방지된다고 보장하기 어려운 유형이라, 앞으로도 비슷한 사례가 보이면
  계속 알려주기로 함.

### 원문이랑 결과를 나란히 비교해서 보여달라는 요청
- **무슨 문제/요청이 있었나**: 뭐가 어떻게 바뀌었는지 한눈에 보기 힘드니, 위키나 코드
  비교하듯 원문과 결과를 좌우로 놓고 빠진 건 빨간색, 추가된 건 초록색으로 표시해달라는
  요청. 교정 내역도 항목이 많으면 안 읽히니 핵심만 굵게 표시해달라는 것도 같이 요청.
- **어떻게 처리했나**: 변환하면 원문/결과를 좌우로 놓고 색으로 비교해서 보여주게 만들고,
  필요하면 색을 꺼서 볼 수 있는 버튼도 추가함. 교정 내역은 "무엇을 무엇으로 고쳤는지" 핵심
  부분만 자동으로 굵게 표시되게 함.
- **결과**: 실제 화면에서 확인함, 정상 동작.

### AI가 왜 그런 판단을 했는지 기록으로 남겨달라는 요청
- **무슨 문제/요청이 있었나**: 일본어를 마음대로 지어내는 것처럼 AI가 규칙을 어긴 경우,
  그냥 막기만 하지 말고 왜 그랬는지 기록을 남기고 화면에서도 바로 확인할 수 있게
  해달라는 요청.
- **어떻게 처리했나**: 규칙 위반이 감지되면 그 순간의 원문, 막힌 결과, AI가 스스로 설명한
  이유까지 전부 별도 기록으로 남기도록 만들고, 화면 맨 아래에 접었다 펼 수 있는 "이상 감지
  로그" 칸을 추가해서 그 기록을 바로 볼 수 있게 함.
- **결과**: 화면에 칸은 정상적으로 뜸. 다만 기록이 실제로 잘 쌓이는지는 아래 항목 문제로
  아직 확인 못 함.

### 변환 기록이 실제로 저장 안 되는 문제 (미해결)
- **무슨 문제였나**: 오늘 수십 번 변환 테스트를 했는데도, 그 기록이 쌓여야 할 저장소를
  확인해보니 단 한 건도 남아있지 않음. 접속에 필요한 값들을 다시 등록했는데도 여전히
  그대로임.
- **확인해보니**: 접속에 필요한 값 세 가지 중 최소 하나가 여전히 비어있거나 틀렸을
  가능성, 또는 값은 맞는데 "읽기"만 되고 "쓰기"는 안 되는 권한으로 등록돼 있을 가능성,
  이렇게 두 가지로 좁혀둔 상태. 정확히 어느 쪽인지는 아직 확정 못 함.
- **어떻게 처리했나**: 원인이 확정되지 않아 별도 조치는 하지 않음. 확인에 필요한 정보를
  요청해둔 상태.
- **결과**: 미해결. 다음에 이어서 확인 필요.
