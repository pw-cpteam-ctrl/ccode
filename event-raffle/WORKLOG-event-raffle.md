  `@anthropic-ai/sdk`로 `claude-opus-4-8` 호출, `output_config.format`으로 JSON 스키마
  강제). 문서에 남아있던 옛 환경변수 이름도 전부 정정함 (`GEMINI_API_KEY` →
  `ANTHROPIC_API_KEY`).
- **결과**: 전환 자체는 코드 변경 확인 수준이었고, 이후 실제 필터링 동작은 다음 항목들에서
  반복적으로 검증됨.

### 비용을 줄이기 위한 코드+AI 하이브리드 필터링 재설계
- **무슨 요청/문제였나**: 참가자 조건 판단을 매번 AI에게 전부 맡기면, 참가자가 많아질수록
  (예: 2천 명) AI에게 보내는 데이터량이 커져서 비용과 응답 시간이 같이 늘어나는 구조적
  문제가 있었음.
- **어떻게 처리했나**: 조건 텍스트를 AI에게 한 번만 보여주고, "코드만으로 정확히 판단
  가능한 규칙"과 "AI의 주관적 판단이 필요한 나머지"로 미리 나누게 함 (`api/plan-filter.js`
  신규 추가 — 참가자 데이터는 보내지 않고 조건 텍스트만 분석). 코드로 판단 가능한 규칙은
  브라우저가 참가자 전원에게 무료로 적용해서 먼저 추리고, AI에게는 그렇게 추려진 소수만
  보내 실제 판단(비용 발생)을 맡김. 조건이 전부 객관적이면 AI 호출 자체가 생략돼 비용이
  0에 가까워짐. 코드 규칙이 없거나 실행 중 오류가 나면 안전하게 기존 방식(전체를 AI로
  판단)으로 자동 전환됨.
- **결과**: Playwright로 세 가지 상황(조건 전체가 코드로 처리됨 / 코드로 1차 추림 + AI로
  나머지 판단 / 코드 규칙 오류 시 전체를 AI로 폴백)을 각각 만들어서 실제로 동작하는지
  확인함.

### 모델 교체 + 비용 안전장치 추가
- **무슨 요청이었나**: 단순 필터링 작업에 비싼 모델을 쓸 필요가 없다는 판단, 그리고 실수로
  대량의 인원이 AI에게 넘어가 비용이 나가는 상황을 막아달라는 요청.
- **어떻게 처리했나**: 필터링에 쓰는 모델을 더 가벼운 것으로 바꿈 (`claude-opus-4-8` →
  `claude-sonnet-5`, `output_config.effort`를 `medium`으로 명시). 필터링 버튼을 실행 중에
  다시 못 누르게 막고, AI 판단에 넘어가는 인원이 200명이 넘으면 예상 비용을 미리 보여주고
  확인을 받은 뒤에만 진행하도록 함 (`ROW_CONFIRM_THRESHOLD=200`, `confirm()` 다이얼로그) —
  코드 규칙이 실패해서 전체 인원이 AI에게 넘어가는 상황에서도 사용자 확인 없이는 비용이
  나가지 않게 함.
- **결과**: 코드 로직 확인 수준으로 반영, 이후 실제 대량 파일 테스트에서 확인 다이얼로그가
  정상적으로 뜨는 것까지 함께 검증됨.

- **어떻게 처리했나**: 당첨자 명단과 조건 텍스트만 GitHub 레포에 자동 커밋하는 기능을 추가함

  (`sample-data/participants-sample.csv`). CSV 인코딩 문제는 `.csv` 파일을 읽을 때 텍스트로
  직접 읽어 UTF-8로 명시 디코딩하도록 고쳐서 해결함.
- **결과**: 실제로 업로드→필터링→추첨→저장 전체 흐름을 Playwright로 돌려서 한글이 안 깨지고
  레포에 정상 저장되는 것까지 확인함.

### 실제 409명 파일을 올렸더니 "참가자 데이터가 비어 있어요" 에러가 난 문제
- **무슨 문제였나**: 실제로 스크래퍼로 모은 409명짜리 파일을 올려서 필터링을 돌렸더니
  "참가자 데이터(rows)가 비어 있어요"라는 서버 에러가 반복해서 남.
- **확인해보니**: 컬럼명 자체가 `css-1jxf684`, `__EMPTY` 등으로 뒤섞여 있는 실제 스크래퍼
  원본 데이터였는데(이건 정상 — 스크래퍼로 수집한 데이터라 원래 이런 형태), 1차 코드 규칙
  적용 결과 후보가 0명이 되면 그 빈 배열이 그대로 AI 판단 API(`/api/filter`)로 넘어가면서
  서버가 정상적으로 "데이터가 비어 있다"고 거부하고 있었음. 문제는 그 에러를 화면에서
  받아본 코드가 원인과 상관없이 항상 "API 키 미설정"이라는 엉뚱한 메시지로만 안내하고
  있었던 것 — 실제 원인 파악을 방해하고 있었음.
- **어떻게 처리했나**: 코드 규칙 적용 후 후보가 0명이면 AI를 호출하지 않고 바로 "0명
  선정됨" + 적용된 규칙을 화면에 보여주도록 함. 네트워크 자체가 안 되는 경우(`TypeError`)와
  서버가 실제로 응답한 오류를 구분해서 정확한 메시지를 보여주도록 고침. 또한 컬럼명이
  의미 없는 문자열일 수 있다는 점과, 확신이 없으면 코드 규칙을 무리하게 만들지 말고 AI
  판단으로 넘기라는 지침을 plan-filter 프롬프트에 추가하고, 참고용 예시 데이터도 3개에서
  8개로 늘려 컬럼 의미를 더 정확히 추론하게 함.
- **결과**: 실제 409명 파일로 재현해서 0명 케이스가 에러 없이 정상적으로 표시되는 것까지
  확인함.

### 조건을 여러 칸으로 분리 + AND/OR 명시 선택 + 본문 컬럼 직접 지정
- **무슨 문제였나**: 같은 409명 파일로 계속 테스트하는 과정에서, 조건 문장 하나에 여러
  조건을 자연어로 섞어 쓰면(예: "사진 첨부하고 정성스럽게 쓴 사람") 그리고/또는이 정확히
  어디에 걸리는지 애매해지는 문제, 그리고 컬럼명만 보고는 어떤 컬럼이 진짜 참가자 댓글
  본문인지 코드도 AI도 확신할 수 없는 문제가 확인됨(이 파일엔 참가자 댓글과 무관한 캐릭터
  랭킹 집계표까지 같은 시트에 섞여 있었음).
- **어떻게 처리했나**: 조건 입력을 텍스트 한 칸에서 "조건1/조건2/조건3..." 여러 칸으로
  나누고, 각 조건 사이에 그리고(AND)/또는(OR)을 사용자가 직접 고르게 함. 왼쪽부터 순서대로
  묶어서 "(조건A) AND (조건B) OR (조건C)" 형태의 명확한 문장으로 조합해 서버에 보내도록
  바꿈(자연어 문장이 가진 연산자 우선순위 모호함 자체를 없앰). 업로드 직후 "댓글/후기 본문"
  컬럼과 닉네임 컬럼을 사용자가 직접 지정하게 하고(값이 평균적으로 가장 긴 컬럼을 본문으로
  자동 추정해 미리 골라둠), 이후 필터링/AI 판단은 오직 이 컬럼 내용만 기준으로 판단하도록
  변경 — 무관한 컬럼(랭킹 집계표 등)은 애초에 API에 보내지 않으므로 전송 데이터도 줄고
  비용도 낮아짐.
- **결과**: 실제 파일로 Playwright 검증 — 자동 컬럼 추정이 실제 댓글/닉네임 컬럼을 정확히
  찾아냄, 조건 3개+AND/OR 조합이 서버에 올바른 문장으로 전달됨, 당첨자 카드에 실제 닉네임이
  표시됨을 확인. 기존 회귀 테스트(코드/AI 세 시나리오 + 0명 처리)도 모두 재검증 통과.

### "나히아 OR 정성스럽게"로 조건을 걸면 항상 전체 인원이 AI로 넘어가는 문제
- **무슨 문제였나**: "나히아(히로아카 동의어)" 단독 조건은 39명만 추려지는데, 여기에
  "또는 정성스럽게 쓴 사람"을 더하면 409명 전체가 AI 판단으로 넘어간다는 재신고. 처음엔
  "OR로 묶인 조건 중 하나라도 순수 주관적 판단(정성스러움)이면, 그 조건만으로도 통과할 수
  있는 사람이 있을 수 있어서 코드가 미리 걸러낼 수 없는 구조적 한계"라고 설명했으나, 이
  설명에 대한 반박(비용을 줄일 방법을 코드로 직접 제안)이 있었음.
- **확인해보니**: 구조적 한계 설명 자체는 틀리지 않았음(OR + 주관적 조건은 전체를 봐야
  하는 게 맞음) — 하지만 "정성스러움"을 무조건 AI 판단으로만 취급해야 하는 건 아니라는
  지적이 맞았음. 짧은 참여 인증성 댓글("참여요" 등)과 실제 이유를 풀어 쓴 긴 댓글은 글자
  수만으로도 상당히 잘 구분됨.
- **어떻게 처리했나**: plan-filter가 "정성스럽게/진심이 담겼는지/성의있게 썼는지" 같은
  조건을 만나면 기본적으로 글자 수 기준 코드 규칙(`row["본문"].length >= N`)으로 처리하도록
  프롬프트를 바꿈. 이러면 "나히아 OR 정성스럽게" 같은 조합도 양쪽 다 코드로 판단 가능해져
  AI 호출 자체가 필요 없어짐. 완벽한 정확도보다 비용 절감을 우선하는 트레이드오프라는 점을
  프롬프트에 명시함.
- **결과**: 실제 조건으로 다시 확인 — "(나히아) OR (정성스럽게 쓴 사람)" 조합이
  `(row) => /나히아|나히야|나희아|nahia/i.test(row["본문"]) || row["본문"].length >= 20`
  같은 코드 규칙 하나로 만들어져 AI 호출 없이(비용 0) 처리되는 것을 실제 API 호출로
  확인함.

### 글자 수 기준을 직접 정할 수 있는 "글자 수 조건" 칸 추가
- **무슨 요청이었나**: 위 글자 수 휴리스틱을 AI가 알아서 추측하게 두지 말고, 파일을 넣을
  때마다 실제 글자 수 분포를 통계로 보여주고 사용자가 직접 기준을 정하게 해달라는 요청.
  전체 400명 중 절반(200명)이 걸리는 기준이면 "정성 조건"으로 의미가 있는지 의문이라는
  지적도 있었음. 구체적인 구간 예시(0~10자/10~20자/20~40자/40~100자 이상)를 직접
  제시했으나, 그 구간을 그대로 쓰지 말고 파일마다 실제 분포에 맞게 자동으로 나누는 방식을
  생각해보라는 요청이 뒤따름.
- **어떻게 처리했나**: 파일 업로드 시 본문 글자 수를 5개 구간으로 자동 분할하되, 구간
  경계는 고정값이 아니라 이번 파일의 실제 최대 글자 수에 맞춰 매번 다시 계산하도록 함
  (`computeLengthStats` — 최소/중앙값/평균/최대까지 함께 계산). 이 통계를 화면에 보여주고,
  그 옆에 사용자가 직접 숫자를 입력하는 "글자 수 조건" 칸을 새로 추가함(기본값은 중앙값으로
  미리 채워둠). 이 조건은 100% 코드로만 판단되어 AI 호출이 전혀 없고, 기존 자유 텍스트
  조건과 그리고/또는으로 섞어 쓸 수 있음. 기존에 자유 텍스트로 "정성스럽게"라고만 입력했을
  때 AI가 알아서 글자 수 기준을 추측해주던 폴백 동작은 그대로 남겨둠(둘 중 골라 쓸 수
  있게).
- **결과**: 실제 업로드 파일로 확인 — 자동 통계가 실제 분포(중앙값 19자 등)를 정확히
  계산하고, 글자 수 조건 기본값이 중앙값으로 채워지며, "(나히아) OR (본문 글자 수가 50자
  이상인 사람)" 같은 조합이 코드로만(AI 비용 0) 처리되는 것까지 확인함.

### 표 셀이 잘려서 안 보이고, 당첨자 원본 데이터도 안 보이는 문제
- **무슨 문제였나**: 스크린샷으로 확인한 결과, 미리보기/후보 표에서 긴 댓글이 잘려 보이는데
  전체 내용을 볼 방법이 없었음. 또한 추첨이 끝난 뒤 당첨자 카드에는 닉네임과 판단 근거만
  보이고, 그 사람이 실제로 어떤 댓글을 남겼는지(원본 데이터)는 확인할 방법이 없었음.
- **어떻게 처리했나**: 표의 행을 클릭하면 그 행만 줄바꿈이 풀리며 전체 내용이 펼쳐지고,
  다시 클릭하면 원래대로 접히도록 함. 당첨자 카드에는 "원본 데이터 보기" 토글을 추가해서
  클릭하면 그 사람의 전체 컬럼 데이터를 볼 수 있게 함 (`<details>`/`<summary>` 사용).
- **결과**: 실제 업로드 파일로 Playwright 검증 — 표 행 클릭 시 펼쳐짐, 당첨자 카드 토글
  클릭 시 전체 원본 데이터가 표시됨을 확인. 기존 회귀 테스트 전체도 재검증 통과.

### 표 기본 펼침 전환 + xlsx 다운로드 + 결과 누적 저장 + 난수 화면 표시
- **무슨 요청이었나**: 리뷰 후 이어서 들어온 요청 묶음 — ① 표를 기본으로 펼친 상태로 두고
  접기/펼치기 버튼을 따로 넣을 것(클릭해서 개별 행 접는 기존 동작은 유지), ② 결과
  다운로드를 JSON 대신 엑셀로, ③ 당첨자 명단을 회차마다 파일 하나씩 받는 대신 서버에 계속
  쌓이게(누적) 저장, ④ 추첨에 실제로 쓰인 난수 값을 화면에도 표기.
- **어떻게 처리했나**: 표 기본 상태를 펼침으로 뒤집고(`.collapsed` 클래스를 붙였을 때만
  접히도록 CSS 반전), 표 위에 "전체 접기/펼치기" 버튼을 추가하면서 기존 행 클릭 토글도
  그대로 유지함. 다운로드는 이미 불러와져 있던 SheetJS로 엑셀 파일을 만들어 브라우저에서
  바로 받도록 변경함 (`XLSX.utils.json_to_sheet` + `XLSX.writeFile`). 저장은 회차별 파일
  대신 `event-raffle/results/history.json` 하나에 계속 이어붙이는 방식으로 바꾸고, 그
  사이 다른 저장 요청이 먼저 반영돼 충돌 나는 경우(sha 낡음)에는 최신 내용을 다시 받아와
  합친 뒤 최대 3번까지 재시도하도록 함(과거 기록이 덮어써져 사라지는 것 방지). 추첨 시마다
  실제 난수 값과 그 시점 후보 수, 몇 번째로 뽑혔는지를 기록해서 화면과 다운로드/저장 파일에
  모두 남기도록 함.
- **결과**: 누적 저장 로직은 실제 파일을 만들지 않고 가짜 GitHub API로 신규 생성/누적
  append/충돌 시 재시도 3가지 경우를 모두 검증함(진짜 저장 코드를 그대로 불러와서 검증 —
  재구현 아님). 그 외 UI 변경은 Playwright로 기본 펼침 상태, 전체 토글 버튼, 난수 값 화면
  표시, xlsx 다운로드 모두 확인함. 기존 회귀 테스트도 전부 재검증 통과.

### 본문 컬럼이 눌려서 표가 이상해진 문제 (필터 이후 표에도 적용 요청)
- **무슨 문제였나**: 스크린샷 확인 결과 댓글(본문) 컬럼이 갑자기 매우 좁고 어색하게
  줄바꿈되어 보임 — 조금 전까진 안 그랬는데 왜 이렇게 됐는지 문의. 그리고 제목 행만 있고
  그 이후로 값이 거의 없는(전체의 1% 이하) 컬럼은 표에서 아예 빼달라는 요청, 이어서 이
  컬럼 제거를 "미리보기 표"뿐 아니라 "필터링 이후 후보 표"에도 똑같이 적용해달라는 요청이
  추가됨.
- **확인해보니**: 두 가지가 겹쳐 있었음. ① 실제 파일 컬럼별 채움 비율을 직접 계산해보니
  스크래퍼가 남긴 무관한 랭킹 집계 컬럼(`__EMPTY_1~9`)들이 1%가 아니라 4.6~7.6% 정도
  채워져 있어서, 애초에 정해뒀던 "1% 이하만 제거" 기준(`getMeaningfulColumns`의
  `threshold`)이 이 컬럼들을 걸러내지 못하고 있었음. ② 컬럼 선택 드롭다운을 채우는 코드
  (`populateColumnSelectors`)가 표를 그리는 코드보다 나중에 실행되고 있어서, 본문 컬럼이
  아직 정해지지 않은 상태로 표가 먼저 그려져 폭 힌트가 적용되지 않고 있었음.
- **어떻게 처리했나**: 컬럼 제거 기준을 실측값에 맞춰 1%에서 10%로 올림. 두 함수의 실행
  순서를 바로잡아 컬럼이 먼저 정해진 뒤에 표를 그리도록 함. 필터링 이후의 후보 표는 전체
  참가자 기준이 아니라 후보 집합 자체를 기준으로 다시 계산해서 컬럼을 제외하도록 별도
  처리함. 본문으로 지정된 컬럼 헤더에는 최소 폭을 줘서 다른 빈 컬럼들 때문에 좁게 눌리지
  않게 함.
- **결과**: 실제 파일로 재검증 — 미리보기가 13개에서 5개 컬럼으로 줄어듦, 본문 컬럼에 폭
  힌트가 정상 적용됨, 후보 표도 후보 집합 기준으로 별도 계산되는 것까지 확인. 기존 회귀
  테스트 전부(세 시나리오 + 0명 처리 + 조건 칸 + 이전 UI 개선) 재검증 통과.

### 참가자 파일을 이 PC에만 캐싱해서 다음 방문 시 자동으로 이어서 쓰기
- **무슨 요청이었나**: 다른 프로젝트(inbound-image-composer)에서 쓰던 "슬롯 캐싱" 방식을
  참고해서, 파일을 업로드하면 기본적으로 실행되고 나중에 다시 들어왔을 때 저번 파일이
  캐싱돼서 보이도록 해달라는 요청(단 이 PC에서만). 캐시 항목에 삭제 버튼이 당연히 있어야
  한다는 확인도 함께 있었음.
- **어떻게 처리했나**: 참가자 파일을 업로드하면 이 PC의 브라우저에만(서버·깃허브에는 전혀
  안 올라감) 최근 5개까지 저장하고, 다음에 페이지를 다시 열면 가장 최근 파일을 자동으로
  불러오도록 함 (`localStorage` 기반 슬롯 캐시, `SLOT_MAX=5`). 내용이 완전히 같은 파일은
  중복 저장하지 않고, 저장 용량이 부족하면 오래된 슬롯부터 무거운 참가자 데이터를 빼고
  다시 저장을 시도함. 업로드 직후·컬럼 선택 변경 시·탭을 닫거나 전환할 때(안전망) 자동으로
  저장되도록 하고, 각 캐시 항목마다 불러오기 버튼과 삭제(✕) 버튼을 넣음. 참가자 데이터
  저장 범위가 달라진 만큼 소개 카드 문구도 "이 PC 브라우저에만 최근 5개까지 남는다"로
  정정함.
- **결과**: Playwright로 전체 흐름 검증 — 첫 방문엔 캐시 없음 → 업로드 직후 바로 캐싱됨 →
  새로고침해도 자동으로 복원됨 → 다른 파일을 올리면 캐시가 2개로 늘어남 → 삭제 버튼으로
  캐시 제거됨 → 남은 캐시를 클릭하면 다시 정상적으로 불러와짐, 전부 확인함. 기존 회귀
  테스트 5종도 모두 재통과함. 커밋 직후 다른 세션이 먼저 올린 커밋과 겹쳐 `main` 푸시가
  한 번 거절됐으나, 강제 푸시 없이 `git fetch` + `rebase` 후 재푸시로 정상 반영함.
