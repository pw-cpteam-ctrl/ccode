# 이벤트 추첨기 — 기획 정리 (v1)

> 이벤트 참가자 원본 데이터(엑셀/CSV)를 업로드하면, 그때그때 자유롭게 입력한
> 조건으로 필터링한 뒤 그중에서 실제 랜덤 추첨까지 해주는 도구.
> `text-gradient/`, `llm-product-formatter-plan.md`, `sns-report-scraper/`와는
> 완전히 별개 프로젝트 — 폴더 분리해서 진행 예정.

## 목적
이벤트 댓글 참가자 중 조건에 맞는 사람을 추리는 작업을 매번 손으로 하나하나
읽으면서 걸러내는 대신 자동화. 조건은 이벤트마다 다르므로 고정 규칙이 아니라
**매번 자유텍스트로 입력**하는 방식.

예시 조건:
1. 사진 첨부한 사람
2. 나루토 꼽은 사람
3. 정성스럽게 쓴 사람

1·2번은 데이터만 보면 기계적으로 판단 가능하지만, 3번("정성스러움")은 주관적
판단이 필요 — 조건 조합이 매번 달라지고 자유텍스트로 들어오는 이상, 조건
형식을 미리 고정하는 규칙 엔진은 만들지 않는다.

**v2: LLM은 "필터 역할"이 아니라 "필터 틀을 짜는 역할"로 전환.** 처음엔
참가자 데이터 전체 + 조건을 LLM에 통째로 보내 판단시켰는데(v1), 참가자 수가
많아지면(예: 2천 명) 호출 1번에 비용이 크게 뛰는 문제가 있었음. 조건 텍스트만
LLM에게 보내서 "코드(정규식 등)로 정확히 판단 가능한 부분"과 "AI 주관 판단이
필요한 나머지"로 미리 나누게 하고, 실제 대량 데이터 검수는 브라우저의 코드가
무료로 처리 → 코드로 못 거른 소수만 그때 AI에게 넘겨 판단. 조건이 전부
객관적이면 AI 판단 호출 자체가 생략되어 비용이 0에 가까워짐. (다른 프로젝트에서
"Claude Code가 필터 코드를 짜주고 실제 대량 처리는 파이썬이 무료로 수행"했던
패턴을 이 웹 도구 구조에 맞게 이식한 것 — 자세한 배경은 대화 기록 참고.)

## 핵심 설계 원칙
- **필터링(LLM) ≠ 추첨(코드).** 조건을 만족하는 사람을 골라내는 건 LLM의
  판단 영역이지만, 그중 최종 당첨자를 뽑는 건 **코드가 실제 난수(예: 배열
  셔플/`crypto.getRandomValues` 등)로 처리**한다. LLM한테 "이 중에 골라줘"까지
  맡기면 진짜 무작위가 아니게 되고, 왜 그 사람이 뽑혔는지 설명하기 애매해져서
  공정성 시비가 생길 수 있음. 필터링 결과(후보 명단)는 화면에 근거와 함께
  보여주고, 그 다음 추첨 버튼을 눌러야 코드가 랜덤으로 최종 당첨자를 뽑는
  2단계 흐름으로 간다.
- 프로젝트 B(언어모델 웹 구현)와 동일하게 **stateless**: 회차마다 완전히 새
  요청. 지난 회차 조건이나 데이터가 이번 판단에 섞이지 않음.

## 결정된 사항

### Raw data 입력
- **엑셀/CSV 파일 업로드** (참가자를 수동으로 정리해둔 파일)
- 정확한 컬럼 구성(닉네임/댓글 내용/사진 첨부 여부 등)은 실제 샘플 파일을
  받아서 확정 필요 — 아래 "아직 안 정한 것" 참고

### 조건 입력
- **매번 자유텍스트**로 입력 ("사진 첨부한 사람 + 나루토 꼽은 사람 + 정성스럽게
  쓴 사람"처럼 한 번에 여러 조건을 자연어로)
- 조건 해석과 후보 필터링은 LLM이 한 번에 처리 (위 "핵심 설계 원칙" 참고)

### 당첨자 추첨
- 필터링 통과자 명단이 나오면, **코드가 진짜 랜덤으로 최종 당첨자를 추첨**
  (LLM 관여 안 함 — 공정성·투명성 확보)

### 실행 환경
- **웹사이트, Vercel** (프로젝트 B와 동일 패턴: 브라우저에서 파일 업로드 →
  Vercel 서버리스 함수가 LLM 호출 → 결과 반환. LLM 키가 브라우저에 노출되지
  않도록 백엔드를 거침)

### 결과 저장
- **결과만 기록** — 당첨자 명단 + 그때 사용한 조건 텍스트만 남기고, 원본
  댓글·참가자 개인정보는 저장하지 않음 (매번 업로드 → 필터링 → 추첨 → 끝,
  원본 데이터는 휘발성)
- 나중에 "지난달 누가 당첨됐었지" 확인용 최소 기록만 남기는 목적
- 저장 위치(깃허브 재사용 여부, 공개/비공개)는 아직 미정 — 아래 참고

## 진행 상황 (막힌 부분 빼고 미리 코딩함)

- [x] `index.html`: 파일 업로드(엑셀/CSV, SheetJS로 파싱) → 미리보기 표 →
  조건 입력 → 후보 명단 표 → 추첨(실제 난수) → 결과 다운로드까지 전체 화면
  완성. 컬럼 구조를 미리 몰라도 헤더 줄을 그대로 읽어 처리하는 방식이라 실제
  엑셀 샘플 없이도 동작하게 만들어둠.
- [x] `api/filter.js`: Claude API(Anthropic SDK, `claude-opus-4-8`) 호출 뼈대
  완성 (Vercel 서버리스 함수). 참가자 데이터 + 조건을 프롬프트로 구성해서
  보내고, `output_config.format`(구조화 출력)으로 `{"matches":[{index,
  reason}]}` 형식을 강제함.
- [x] `api/plan-filter.js` + `index.html` 3단계 필터링으로 재설계 (v2, 위
  "핵심 설계 원칙" 참고): ① 조건 텍스트만 Claude에게 보내 코드 규칙/AI 주관
  조건으로 분해 → ② 코드 규칙을 브라우저에서 전체 참가자에게 무료로 적용 →
  ③ 남은 소수만 `api/filter.js`로 AI 판단 (조건이 전부 객관적이면 ③ 생략,
  AI 비용 0). 코드 규칙이 없거나 실행 중 오류나면 안전하게 v1 방식(전체를
  AI에게 보냄)으로 자동 대체. Playwright로 세 시나리오(전체 코드 처리 /
  코드+AI 혼합 / 코드 오류 시 폴백) 모두 정상 동작 확인함.
- [x] 모델을 `claude-opus-4-8` → `claude-sonnet-5`로 변경, `effort: medium`
