# CHANGELOG.md 작성 규칙

메가하우스 마토메 프로젝트의 CHANGELOG를 기반으로 한 통일된 작성 규칙입니다.  
**목적**: 비개발자도 이해할 수 있는 보고 자료이면서, 개발자는 기술 용어로 학습할 수 있도록 함.

---

## 📋 기본 포맷

```markdown
## YYYY-MM-DD (VX.X 버전 제목)
- **섹션명**
  - 사용자 관점의 설명 (명사형/음슴체)
    (기술: 기술 용어 및 상세 기술 설명)
  - 다른 항목
    (기술: 해당 기술 용어)

- **다른 섹션명**
  - 항목
    (기술: ...)
```

---

## 📐 구조 규칙 (3단계)

### 1단계: 버전 헤더
```markdown
# YYYY-MM-DD (VX.X 제목)
```
- 제목1 사용 (이전: 제목2)
- 날짜: ISO 형식 (2026-07-03)
- 버전: `V1.0`, `V1.4` 등 (초기 버전은 V1.0, 첫 업데이트는 V1.1)
- 제목: 해당 버전의 주요 테마 (예: "2페이지 지원", "팀 공유 기능")
- 마침표 없음

### 2단계: 섹션 (기능 분야)
```markdown
### 섹션명
```
- 제목3 사용 (이전: `- **섹션명**`)
- 버전 헤더 다음에 빈 줄 1개
- 섹션 다음에 빈 줄 1개
- 기능 분야별 분류:
  - 페이지 관리
  - 셀(사진 칸) 조작
  - 도움말
  - 저장 기능 개선
  - 안정성 개선
  - 입력 편의성
  - 버그 수정
  - 배포
  - 기타

### 3단계: 상세 항목 (2줄 구조)
```markdown
- **강조할 부분** 나머지 설명
  (기술: 구체적 기술 용어)
```

**규칙:**
- 강조할 부분은 **굵게** 처리 (동작, 주요 변경점)
- 1줄: 비개발자도 이해할 수 있는 설명 (명사형/음슴체)
- 2줄: 괄호 안에 기술 용어 + 상세 기술 설명
- 각 줄은 마침표 없음
- 2칸 들여쓰기

## 🎯 문체 규칙

### ✅ 음슴체/명사형으로 끝내기
- "추가", "수정", "개선", "제거", "지원", "변경"
- "~함", "~됨", "~가능"
- 예: "사진 드래그 기능 추가", "프로그램 느려짐 현상 수정"

### ❌ 피해야 할 문체
- "~요" (존댓말 금지)
- "~했어요", "~할 수 있어요"
- "그냥", "음...", "뭔가"

### 📝 실제 변환 예시

**좋은 예:**
```markdown
- 여러 번 드래그했을 때 프로그램 느려짐 현상 수정
  (기술: window 이벤트 리스너 누적으로 인한 메모리 누수 버그 해결)

- 저장 용량 대폭 증가 (이전 5MB → 무제한)
  (기술: localStorage → IndexedDB 마이그레이션)

- 달마다 다른 작업을 분리해 저장 가능
  (기술: 월별 저장 슬롯 시스템, YYYY_MM 단위 상태 분리 관리)

- 클립보드에서 이미지 직접 붙여넣기 지원
  (기술: Ctrl+V 이벤트 핸들러 구현, 마지막 페이지 자동 배치)
```

**나쁜 예:**
```markdown
- localStorage 저장 실패(용량 초과) 시 조용히 무시되던 문제 → 경고 토스트 표시
- 이미지 압축률 조정, 저장 방식을 localStorage → IndexedDB로 전환 (5MB 용량 제한 해제)
- 월별 저장 슬롯 기능을 추가할 수 있어요 (기술 설명 부족)
```

## 🏷️ 심볼 및 이모지 (선택)

자주 쓰이는 심볼:
- `→`: 변경 흐름 ("A → B로 변경", "이전 5MB → 무제한")
- `—`: 설명 연결 ("기능명 — 설명")
- `📤`: 내보내기
- `📥`: 불러오기

## 🔄 버전 번호 규칙

### 초기 릴리스
```
V1.0 (초기 버전)
```

### 이후 업데이트
- **V1.1 ~ V1.9**: 마이너 업데이트 (기능 추가, 버그 수정)
- **V2.0**: 메이저 업데이트 (구조 변경, 대규모 리팩토링)

### 다중 버전 담기
같은 날짜에 여러 버전을 한 항목으로:
```markdown
## 2026-07-03 (V1.4 ~ V1.9 안정성 개선 및 신규 기능)
```

---

## 📝 작성 순서

1. **버전 헤더** 작성 (날짜 + 버전 + 제목)
2. **섹션** 구분 (기능 영역별)
3. **상세 항목** 작성 (1줄: 사용자 설명, 2줄: 기술 용어)
4. **문체 통일** (명사형/음슴체, "~요" 금지)
5. **기술 용어 괄호에 담기** (2줄 구조 유지)

---

## 📦 실제 예시 (마토메 기준)

```markdown
## 2026-07-03 (V1.4 ~ V1.9 안정성 개선 및 신규 기능)
- **저장 기능 개선**
  - 저장 실패 시 경고 토스트 표시
    (기술: localStorage 용량 초과 예외 처리, 사용자 알림 추가)
  - 저장 용량 대폭 증가 (이전 5MB → 무제한)
    (기술: localStorage → IndexedDB 마이그레이션)
  - 달마다 다른 작업을 분리해 저장 가능
    (기술: 월별 저장 슬롯 시스템, YYYY_MM 단위 상태 분리 관리)

- **오픈용 헤더 (신규 기능)**
  - 검정 그라데이션 오버레이 헤더 추가
    (기술: 2계층 오버레이 구조, position:absolute 활용)
  - 그라데이션 농도/퍼짐 조절 슬라이더 추가
    (기술: 6단계 ease-in 커브, 동적 CSS 그라데이션)
  - 스토어명 박스를 알약형(pill shape)으로 변경
    (기술: border-radius:999px, padding 조정)
  - 타이틀 폰트 변경 및 레이아웃 3열로 재구성
    (기술: font-family 프리텐다드 적용, CSS Grid 3열 배치)

- **안정성 개선**
  - 여러 번 드래그했을 때 프로그램 느려짐 현상 수정
    (기술: window 이벤트 리스너 누적 버그 해결, 전역 _cellDrag 상태 변수 적용)
  - 그리드 빈 공간 드롭 시 중복 실행 버그 제거
    (기술: renderGrid() 호출 시 dragover/drop 이벤트 중복 등록 제거, _dndBound 플래그 활용)

- **입력 편의성**
  - 클립보드에서 이미지 직접 붙여넣기 지원
    (기술: paste 이벤트 핸들러, DataTransfer API, 마지막 페이지 자동 배치)

- **기타**
  - 폴더 구조 통일 (megahouse.html → megahouse-matome/index.html)
    (기술: 디렉토리 이동, 다른 도구들과 구조 통일)
```

## ✨ 팁

1. **날짜는 실제 작업 완료 날짜**로 기록 (커밋 날짜 X)
2. **상세 항목은 항상 2줄**: 1줄 사용자 설명 + 2줄 기술 용어
3. **기술 용어는 괄호 안**에만 들어감 (본문에 섞지 말 것)
4. **비개발자도 1줄만 읽으면 이해**되도록 작성
5. **섹션당 항목 수 제한 없음** (3개~10개 정도 권장)
6. **신규 기능은 섹션명에 "(신규 기능)" 표기** (예: "오픈용 헤더 (신규 기능)")
7. **여러 버전 담을 땐 하이픈으로 연결** (V1.4 ~ V1.9)

## 🎨 다른 프로젝트 적용 시 커스터마이징

이 규칙은 **megahouse-matome** 기준이므로, 다른 프로젝트는 섹션 이름만 조정:

예시 (insta-gen):
```markdown
## 2026-07-04 (V1.2 모바일 최적화)
- **UI 반응성 개선**
  - 모바일 화면(< 480px)에서 패널 자동 숨김 지원
    (기술: media query, display:none 처리)
  
- **성능 최적화**
  - 이미지 로드 중 placeholder 표시
    (기술: blur-up 기법, 점진적 이미지 로딩)
```
