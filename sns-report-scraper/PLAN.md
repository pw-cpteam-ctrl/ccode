# SNS 성과 리포트 자동화 — 기획 정리 (v1, 진행 중 킵)

> 매달 수동으로 작성하던 SNS 성과 리포트(자사·경쟁사 비교 + 판매데이터 연동)를
> 자동화하는 프로젝트. 컴퓨터 필요한 작업이라 잠시 킵 — 재개 시 이 문서부터 확인.
> `llm-product-formatter-plan.md`(언어모델 웹 구현), `text-gradient/`(그라데이션 도구)와는
> 완전히 별개 프로젝트.

## 목적
- 자사/경쟁사 SNS 게시물의 좋아요·댓글·팔로워 등 공개 지표를 링크만 붙여넣으면 자동 수집
- 자사 vs 경쟁사 비교표 자동 생성 (몇 % 우세인지 비율 계산)
- 판매데이터와 연결해서 SNS 성과가 판매에 얼마나 기여했는지까지 (2단계, 후속)
- AI가 글 써주는 리포트는 불필요 — 순수 수집·집계 자동화만

## 결정된 사항

### 수집 지표
- 좋아요/댓글/팔로워수 등 **공개 지표만** (로그인 없이도 보이는 값)
- 경쟁사 계정은 로그인 불가하니 애초에 공개 지표 기준으로 설계

### 기간 필터
- 수집 범위 지정 가능해야 함 (예: 7/1~7/2, 또는 6/30 하루만)
- SNS 수집 + 판매데이터 양쪽 다 같은 기간 기준으로 맞춰서 비교

### 로그인 방식 (인스타/X)
- 본인 인스타그램/X 계정 로그인 세션으로 브라우징하며 수집 (예전에도 이 방식으로 해옴)
- Playwright `storageState` 파일(쿠키 기반)로 세션 전달 — 비밀번호는 절대 코드에 안 남음
- 세션 파일 생성법 (기본, `playwright codegen` 방식):
  ```
  npx playwright install chromium   # 최초 1회
  npx playwright codegen --save-storage=instagram-session.json https://www.instagram.com
  npx playwright codegen --save-storage=x-session.json https://x.com
  ```
  브라우저 창 뜨면 직접 로그인 → 창 닫으면 세션 파일 자동 생성
  - ⚠️ **실제로 시도해보니 X/인스타 양쪽 다 "일시적으로 로그인 제한됨"으로 막힘.** playwright가
    새로 내려받은 낯선 크로미움 + 자동화 흔적을 봇으로 의심해서 차단하는 것으로 추정
    (계정 2개 다 동일하게 막혀서 계정 문제가 아니라 브라우저 지문 문제로 판단).
    `--channel=chrome`(설치된 크롬 그대로 사용)로도 시도했으나 마찬가지로 막힘.
- **대안 경로 A-1 (개발자도구 cURL 복사) — ⚠️ 최신 크롬은 보안상 Cookie 헤더를 빼고 복사함,
  실제로 시도했으나 쿠키가 아예 없이 복사돼서 이 방식은 사실상 못 씀 (도구는 만들어뒀지만
  크롬 버전에 따라 동작 안 할 수 있음, 참고용으로만 남김):**
  1. F12 → Network 탭 → 새로고침 → 맨 위 문서 요청 우클릭 → Copy → Copy as cURL
  2. `node curl-cookies-to-storage-state.js <파일> x-session.json https://x.com`
- **대안 경로 A-2 (개발자도구 Application 탭 쿠키 표 복사) — 확장 없이 가능, 채택된 방식:**
  1. F12 → **Application** 탭 → Storage → Cookies → `https://x.com` 클릭
  2. 쿠키 표 안 클릭 → **Ctrl+A**(전체선택) → **Ctrl+C**(복사)
     — 실제로 해보니 **헤더 줄(Name, Value...)은 안 딸려오고 표 본문만 복사됨** (헤더는 고정
     영역이라 스크롤 영역 선택에 안 포함되는 듯). `table-cookies-to-storage-state.js`가 헤더 유무를
     자동 판별해서 헤더 없으면 크롬 고정 열 순서(Name/Value/Domain/Path/Expires/Size/HttpOnly/
     Secure/SameSite/PartitionKey/CrossSite/Priority)로 처리하도록 대응함
     — 만료일이 "세션"(한국어 로케일)으로 표시되는 것도 처리함
  3. `node table-cookies-to-storage-state.js <붙여넣은 텍스트 파일> x-session.json https://x.com` 실행
     → 표에 있는 실제 httpOnly/secure/SameSite/만료일 속성을 그대로 반영 (curl 방식보다 정확).
     Playwright 컨텍스트에 실제 로드되는 것까지 검증 완료 (모킹 값 + 실제 계정 쿠키 둘 다 확인)
  4. 인스타그램도 동일하게 origin만 `https://www.instagram.com`으로
  - ⚠️ **가상 스크롤 때문에 Ctrl+A가 화면에 보이는 근처 행만 선택할 수 있음.** 필요한 쿠키가 안
    보이면 패널 상단 필터(돋보기) 입력창에 쿠키 이름(`auth_token`, `ct0` 등)을 검색해서 그 행만
    복사하는 걸 추천
  - ⚠️ **실제 로그인 쿠키 값(`auth_token` 등)은 비밀번호와 동급으로 민감함 — 채팅에 붙여넣지 말고
    본인 컴퓨터에서 직접 `table-cookies-to-storage-state.js`를 돌려서 로컬에만 `x-session.json`을
    만들어 쓸 것.** (Claude Code 웹 실행 환경은 격리된 원격 샌드박스라 브라우저의 실제 네트워크
    접속 자체가 막혀있어서 — `example.com`조차 `ERR_CONNECTION_RESET` — 이 환경에서 실제
    x.com/인스타그램 접속 테스트도 못 함. `twitter.js`/`instagram.js` 실전 검증은 어차피
    PLAN대로 로컬 컴퓨터에서만 가능)
- **대안 경로 B (확장 설치, Cookie-Editor)** — A가 안 되면 시도:
  1. "Cookie-Editor" 확장 설치 → 정상 로그인 → "Export as JSON"으로 내보내기
  2. `node cookies-to-storage-state.js cookies-x-raw.json x-session.json https://x.com` 실행
  - 둘 다 로그인 자체를 평소 쓰던 정상 브라우저에서 하므로 자동화 탐지에 안 걸림
- ⚠️ 세션 파일(그리고 원본 쿠키 export 파일)은 로그인된 상태 그 자체. 유출 주의, 만료되면 재생성 필요.
  둘 다 `.gitignore`에 포함(`*-session.json`, `cookies-*.json`)돼서 실수로 git에 안 올라감.

### 판매데이터 (네이버 스마트스토어)
- 지금은 개인 개발 크롬 확장프로그램으로 "실시간 재고"만 조회 가능 (기간별 소급 조회 불가)
- 우선순위 추천:
  1. **네이버 커머스API센터 공식 API** 신청 가능한지 먼저 확인 (제일 안정적, 장기적으로 더 간단함)
  2. 안 되면 확장프로그램 내부 뜯기보다 **판매자센터 화면 자체를 로그인 세션으로 자동 열람** (원리는 SNS 스크래핑과 동일, 더 안전하고 유지보수 쉬움)
  3. 최후 수단으로만 확장프로그램 구조 분석
- 재고가 "현시점 조회만" 가능하다는 제약 있음 → 기간별 판매량 추정하려면 앞으로 **직접 스냅샷을 주기적으로 남겨둬야** 함 (과거 소급 불가)

## 스크래퍼 (이 폴더에 포함, 실제 계정으로 검증 완료 — 아래 "다음 단계" 참고)
- `twitter.js` — X(트위터) 프로필 게시물 수집. 예전 수동 방식(브라우저 JS 주입 + 대화창 누적)을
  Playwright 스크립트로 이식. 스크롤하며 파싱, 범위 시작일 이전 게시물 연속 3건 나오면 자동 중단.
- `instagram.js` — 인스타그램 프로필 게시물 수집. 링크 먼저 다 모으고(날짜 기반 조기 종료 포함)
  → 게시물별 개별 파싱. **좋아요/댓글수를 화면 좌표(x>700, y 400~580) 기준으로 찾는 부분,
  실제 계정으로 화면과 대조해서 정확히 일치함을 확인함.** 캡션 추출 정규식은 계정명 파라미터화 완료.
- 둘 다 계정명·기간·세션파일 경로를 파라미터로 받도록 일반화해둠 (원본은 자사 계정 한정 코드였음)

## 취합/저장 로직
- `aggregate.js` — `buildComparisonReport()`: 자사/경쟁사 수집 결과를 받아 플랫폼별(트위터/인스타)로
  게시물수·총/평균 좋아요·리트윗(트위터)·댓글(인스타)을 집계하고, 자사 대비 경쟁사 비율(%)과
  우세/열세 %p를 계산. 경쟁사가 여러 계정이면 계정별 비교 + 경쟁사 평균 비교를 함께 제공.
  - `parseCount()`: "1.2만", "3천", "1.2K", "3.4M", "3,412" 등 표기를 숫자로 변환. 파싱 실패(예:
    인스타 좌표 파싱이 잘못 잡은 값)는 죽지 않고 `null` 처리 후 계정별 `parseFailures` 건수로 노출.
  - **`buildProductComparison()` (상품별 비교):** run.js로 첫 실전 실행해봤더니 전체 합산
    요약만 있어서 "상품(피규어)별로 자사 vs 경쟁사 비교"를 보고 싶다는 요청 받음.
    - 1차 시도(해시태그 교집합 매칭)는 폐기함 — 해시태그 없이 상품명을 그냥 본문에 나열하는
      게시물(예: 고정 게시물의 "헌터x헌터 원피스 나루토...")을 다 놓쳐서 매칭률이 낮았음.
    - **최종: 본문 템플릿 안 고정된 위치에서 상품명을 직접 추출.** 자사/경쟁사 둘 다 게시물
      작성 템플릿이 일정하다는 걸 확인해서(사용자 확인) — **자사는 본문 첫 줄**, **경쟁사는
      "바로가기"/링크가 있는 줄 바로 위 줄(✔️ 표시)** 이 상품명이라는 규칙으로 `extractOwnProductName()`
      / `extractCompetitorProductName()` 구현. 추출한 상품명에서 키워드를 뽑아(브랜드/시리즈/판매
      상용구는 `GENERIC_KEYWORDS`로 제외) 하나라도 겹치면 같은 상품으로 그룹화(Union-Find).
      상품명을 못 뽑았거나 겹치는 키워드가 없는 게시물은 "매칭 안 됨" 목록으로 투명하게 노출.
    - **한계**: 순수 키워드 매칭이라 (1) 표현이 아예 다르면 매칭 실패, (2) 흔한 단어가 겹쳐서
      상관없는 상품이 잘못 묶일 가능성 있음 — `GENERIC_KEYWORDS` 목록은 지금까지 나온 실제
      캡션 기준으로 임시로 채운 것이라, 실제 결과 보고 이상한 매칭/과도한 미매칭 있으면
      계속 다듬어야 함.
    - **실제 계정으로 첫 실행 결과, 심각한 오매칭 발견+수정:** 자사 트위터 32건 중 21건이
      "원피스 액션 피규어" 하나로, 인스타 30건 중 29건이 "디지몬 어드벤처 GEM" 하나로
      통째로 뭉쳐버림. 원인 2가지 — (1) **"26년", "7월" 같은 날짜 토큰이 키워드로 잡혀서**,
      거의 모든 게시물에 공통으로 등장하는 날짜 하나 때문에 서로 다른 상품들이 사슬처럼
      전부 연결돼버림(Union-Find 특성상 A-B, B-C만 겹쳐도 A-B-C가 통째로 한 그룹이 됨).
      (2) **`GENERIC_KEYWORDS`가 정확히 일치(exact match)하는 단어만 걸러내서**, 한국어는
      띄어쓰기 없이 붙여쓰는 경우가 많아 "예약시작"처럼 상용구가 포함된 복합 단어는 못 걸러냄.
      → 숫자가 포함된 토큰은 전부 키워드에서 제외, `GENERIC_KEYWORDS` 필터링을 부분 포함
      (substring) 방식으로 변경, "세컨드"(재판/리뉴얼 버전을 뜻하는 상용구)도 목록에 추가.
      `verify-mock.js`에 날짜 공유 회귀 테스트 추가해서 재발 방지.
    - **표 구조를 사용자가 예전에 수동으로 쓰던 엑셀 형식에 맞춰 통째로 다시 짬.** 기존엔
      자사 요약 블록 따로, 경쟁사 요약 블록 따로 나열하는 방식이었는데, **한 행 = 한 상품**으로
      자사(PW)/경쟁사(BH) 값을 나란히 놓고 차이·배수·게시 시각·시각차이·결과(우세/약세/경합)까지
      한 번에 보는 방식으로 교체. 사용자가 준 예전 참고표를 역산해서 규칙 확정:
      - **차이/배수**: `PW-BH` 차이값 + `(배수)` — PW가 크면 `PW/BH`, BH가 크면 `-(BH/PW)`
        (예: 336/14=24배). `formatDiffWithMultiplier()`
      - **시각차이**: PW/BH 중 각자 제일 이른 게시 시각(KST) 차이를 분 단위로. **버그 수정:**
        표시가 "시:분"만 있어서 PW/BH가 하루 넘게 차이나는 날 각각 게시한 경우
        (예: "1496분") 왜 저렇게 크게 나오는지 헷갈렸음 — `formatKstTime()`이 날짜("M/D")도
        같이 표시하도록 수정 (`verify-mock.js`에 회귀 테스트 추가)
      - **결과**: 표시 지표(리트윗+좋아요 또는 좋아요+댓글) 둘 다 자사가 크면 우세, 둘 다
        작으면 약세, 엇갈리면 경합 — 참고표 12개 행 전부 이 규칙과 정확히 일치함을 확인
      - **IP/시리즈 분리**: `KNOWN_PRODUCT_LINES`(룩업/GEM/메가캣/테노히라/컬렉션 등 알려진
        상품 라인명) 목록에서 매칭되는 걸 "시리즈"로 빼고 나머지를 "IP"로. **새로운 라인명이
        나오면 이 목록에 계속 추가해야 함** — 목록에 없으면 시리즈 칸이 빈 채로 나감
      - 맨 아래 "계" 합계 행도 추가(시각/결과는 상품마다 달라서 합계 의미 없어 비워둠)
    - **실제 데이터로 재검증했더니 트위터 32건 중 21~23건, 인스타 30건 중 29~30건이 여전히
      하나로 뭉쳐짐 (사용자가 캐시 파일을 직접 공유해줘서 실제 원본 캡션으로 정밀 진단).**

      2. **naver.com 링크의 16진수 해시 조각이 우연히 겹침** — 숫자만 지우고 남은 "ca","ef","db"

         "구매는 프로필 링크를 참고해 주세요!" 같은 반복 CTA 문구도 같은 문제라 제외 목록에 추가
      - **근본적으로 판단 기준도 강화**: 키워드 **1개만 겹쳐도 매칭** → 제외 목록에 없는 흔한
        단어 하나(예: "SET", "버전")가 우연히 겹치는 것만으로도 매칭 사고가 반복됨 →
        **2개 이상 겹쳐야 매칭**(`MIN_SHARED_KEYWORDS`)으로 변경
      - 재검증 결과: 트위터 17개 상품으로 정확히 분리(매칭 안 됨 자사 5/경쟁사 6, 원래
        11/25였음), 인스타 18개 상품으로 분리(매칭 안 됨 자사 1/경쟁사 4, 원래 4/6이었지만
        매칭된 것도 대부분 하나로 뭉쳐있었음) — 실제 캡션 데이터로 직접 검증 완료, 이상한
        오매칭 사례 안 보임
      - `verify-mock.js`에 이번에 찾은 3가지 오매칭 패턴(브라켓/URL해시/흔한단어 1개) 회귀
        테스트 추가
- `excel.js` — `saveReportToExcel()`: 취합 결과를 엑셀에 저장. **기존 시트는 절대 삭제/덮어쓰지
  않고 새 시트만 추가**(히스토리 누적), 같은 기간으로 재실행해도 시트 이름이 자동으로 분리됨.
  쓰기는 임시 파일 → 원자적 rename 방식이라 도중에 프로세스가 죽어도 기존 파일은 안전.
  파싱 실패 건수가 있으면 시트에 ⚠ 경고 문구로 표시. **저장 1회당 시트 2개 생성**: 요약 시트
  (`YYYYMMDD_YYYYMMDD`) + 상품별 비교 시트(`YYYYMMDD_YYYYMMDD-상품별`, 매칭 안 된 게시물 목록 포함).
- `run.js` — 위 두 로직 + `twitter.js`/`instagram.js`를 잇는 전체 실행 스크립트. `CONFIG`에
  원칙적으로 운용하는 자사/경쟁사 계정(X: megahousestore/megahouse_BH, 인스타:
  megahouse_store/megahouse_korea_dt_bh) 고정 반영됨. **실제로 한 번 실행해서 요약 비교표까지는
  정상 동작 확인됨.** 상품별 비교 기능은 이번에 추가된 것이라 아직 실제 데이터로 재실행 전.
  - 수집한 원본 게시물을 `reports/_last-collection.json`에 캐시로 저장함 (수집은 몇 분 걸리는데
    리포트 포맷만 고칠 때마다 매번 재수집시키는 게 비효율적이라는 피드백 받고 추가).
- `rebuild-report.js` — 위 캐시를 다시 읽어서 취합+엑셀+HTML 저장만 재실행 (브라우저/세션
  필요 없어서 몇 초 안에 끝남). `aggregate.js`/`excel.js`/`html-report.js` 포맷만 고쳤을 때
  이걸로 재생성, 계정/기간을 바꿔서 새로 수집해야 하면 `run.js`를 다시 돌려야 함.
- `html-report.js` — **엑셀은 보기 불편하다는 피드백으로 추가.** 사용자의 이사님이 쓰던
  판매 비교 리포트(완전히 다른 판매량/매출 도구, HTML) 스타일(상단 요약 카드 + 순위
  테이블 + 1~3위 강조 + 결과 배지)을 그대로 참고해서 우리 상품별 비교에 입힘.
  `saveHtmlReport()`가 `reports/sns-report.html`로 저장 — **히스토리 누적 안 하고 매번
  덮어씀**(과거 데이터 보존은 엑셀이 계속 담당, 이 HTML은 최신 결과를 보기 좋게 보는
  용도). `run.js`/`rebuild-report.js` 둘 다에서 엑셀과 함께 자동 생성됨. 실제 캐시 데이터로
  생성해서 Playwright로 스크린샷 찍어 직접 확인함(이모지가 IP명 앞에 지저분하게 남는 것도
  발견해서 `splitIpAndLine()`에 앞뒤 이모지/기호 제거 추가).
  - **게시물 임베드 토글 추가**: 각 상품 행의 "▶ 보기" 버튼을 누르면 그 상품에 속한 PW/BH
    게시물들을 2열(PW 왼쪽/BH 오른쪽)로 트위터/인스타 공식 oEmbed 마크업으로 렌더링,
    다시 누르면 접힘. 성능 때문에 각 위젯 스크립트(twitter widgets.js / instagram embed.js)
    호출은 **처음 열 때만** 1번 실행(`row.dataset.rendered`로 체크) — 모든 상품을 페이지
    로드 시점에 한꺼번에 처리하지 않음. **인터넷 연결된 브라우저에서 열어야 실제 카드로
    보임** — Claude Code 웹 실행 환경(원격 샌드박스)은 네트워크가 막혀있어 여기서는
    폴백(링크 텍스트)까지만 확인 가능, 실제 임베드 렌더링은 사용자가 직접 확인 필요.
    토글 열기/닫기 자체는 Playwright로 동작 확인함.
- `manual-matches.json` — **자동 매칭이 놓친 게시물을 수동으로 지정하는 목록 (사용자와
  상의해서 결정한 방식).** "자동 분류 → 미매칭만 사람이 지시" 흐름: 자동 매칭 후 남은
  "매칭 안 됨" 목록에 `PW #1`, `BH #1`처럼 번호가 붙어있는데(엑셀/HTML 둘 다), 사용자가
  채팅으로 "PW 3번 BH 1번 매칭해줘"라고만 말하면 Claude가 그 번호에 해당하는 실제
  링크를 찾아서 이 파일에 추가하고 `rebuild-report.js`로 재생성 — **사용자가 직접 JSON을
  안 만져도 됨.** 형식:
  ```json
  { "twitter": [{ "pw": ["링크1"], "bh": ["링크2"], "label": "표시할 상품명(선택)" }], "instagram": [] }
  ```
  `aggregate.js`의 `extractManualMatches()`가 자동 매칭보다 먼저 이 목록을 확정하고, 지정된
  게시물은 자동 매칭 대상 풀에서 빠짐. `label`을 주면 자동 정리(상용구 제거 등) 없이 그대로
  IP명으로 씀. 실제 캐시 데이터로 테스트해서 정상 동작(매칭 안 됨 개수 정확히 줄어듦) 확인함.
- `verify-mock.js` — 브라우저/로그인 세션 없이 `aggregate.js`+`excel.js` 내부 로직만 모킹
  데이터로 검증하는 스크립트 (`npm run verify`). 트위터.js/instagram.js가 실제로 반환할 형태를
  흉내낸 값으로 합계·비율 계산, 파싱 실패 집계, 엑셀 히스토리 누적(재실행 시 덮어쓰기 안 됨)까지 확인함.
- `test-collect.js` — `twitter.js`/`instagram.js`를 계정 하나만 빠르게 테스트하는 CLI
  (`node test-collect.js twitter <계정> <시작일> <종료일>`). `headless:false` 고정이라 브라우저
  창을 직접 보면서 확인 가능. `run.js` 전체 CONFIG 채우기 전 사전 점검용.
