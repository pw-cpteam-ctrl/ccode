## 2026-07-06 — 상품별 자동 매칭이 계속 크게 뭉침 (3단계 진단)

**배경**: 게시물을 상품(피규어)별로 자동으로 짝짓는 기능을 처음 만든 뒤, 실제 계정 데이터로
돌려볼 때마다 "완전히 다른 상품들이 하나로 뭉치는" 문제가 3번에 걸쳐 다른 원인으로 재발함.

**1차: 날짜 토큰 공유**. 자사 트위터 32건 중 21건이 "원피스 액션 피규어" 하나로, 인스타 30건
중 29건이 "디지몬 어드벤처 GEM" 하나로 뭉쳐버림. **원인**: "26년", "7월" 같은 날짜 토큰이
키워드로 잡혀서, 대부분 게시물에 공통으로 등장하는 날짜 하나 때문에 서로 다른 상품이 사슬처럼
전부 연결됨(Union-Find 특성상 A-B, B-C만 겹쳐도 A-B-C가 통째로 한 그룹). 여기에 더해
`GENERIC_KEYWORDS`가 정확히 일치(exact match)하는 단어만 걸러내서 "예약시작"처럼 상용구가
포함된 복합 단어(띄어쓰기 없이 붙여쓴 한국어)는 못 걸러내는 문제도 겹침. **해결**: 숫자 포함
토큰은 전부 키워드에서 제외, `GENERIC_KEYWORDS` 필터링을 부분 포함(substring) 방식으로 변경.

**2차: 라인(시리즈) 미분리**. "원피스"라는 IP 공통점만으로 룩업/스케일/컬렉션/메가캣 등
완전히 다른 상품 라인이 다 뭉쳐서 나옴. **원인**: `detectProductLine()`으로 라인을 감지하긴
했지만 "둘 중 하나만 미감지면 통과"로 느슨하게 판정해서, 라인 미감지 게시물이 다리 역할을 해
다른 라인끼리 다시 전이적으로 합쳐짐. **해결**: 감지된 라인이 정확히 같아야만(둘 다 미감지인
경우도 "같음") 매칭되도록 엄격화. 트레이드오프로 매칭 안 됨 개수는 늘었음(트위터 기준
7+13건→12+18건) — 잘못 합치는 것보다 투명하게 "매칭 안 됨"으로 보이는 게 안전하다고 판단
(수동 매칭 기능으로 바로 보완 가능).

**3차: 라인 제외 목록 누락 + URL 해시 우연 일치 + 브라켓 문구 반복**. 라인을 엄격화한 뒤에도
실제 데이터 재검증 결과 트위터 32건 중 21~23건, 인스타 30건 중 29~30건이 여전히 하나로 뭉침
(사용자가 캐시 파일을 직접 공유해줘서 실제 원본 캡션으로 정밀 진단). 원인 3가지 — (1)
`KNOWN_PRODUCT_LINES`(룩업/GEM/컬렉션 등)가 시리즈 분리용으로만 쓰이고 매칭 제외 목록엔
없어서 "컬렉션" 같은 라인명 하나로 완전히 다른 프랜차이즈가 엮임 (2) naver.com 링크의 16진수
해시 조각(숫자만 지운 나머지, 예: "ca","ef","db")이 알파벳 6개(a~f)뿐인 16진수라 무관한
게시물끼리도 우연히 겹쳐서 전부 연결됨 (3) "[예약시작]", "[채색원형 최초공개]" 같은 브라켓
태그 문구가 여러 게시물에 반복 등장해서 그 문구 자체로 무관한 상품이 연결됨. **해결**:
`extractKeywords()`가 `GENERIC_KEYWORDS`와 `KNOWN_PRODUCT_LINES`를 같이 제외하도록 수정,
naver.com/http/순수 16진수 줄을 통째로 제거하는 `stripUrlNoise()` 추가, 브라켓 통째 제거 +
인스타 반복 CTA 문구도 제외 목록에 추가. 추가로 판단 기준 자체도 강화 — 키워드 1개만 겹쳐도
매칭하던 것을 2개 이상(`MIN_SHARED_KEYWORDS`)으로 올림. 재검증 결과 트위터 17개 상품, 인스타
18개 상품으로 정확히 분리됨을 확인.

**재발 방지**: 텍스트 매칭 로직에 새 필드/소스(URL, 날짜, 반복 상용구 등)가 섞여 들어갈 때마다
"이게 우연히 겹칠 수 있는 값인가"를 먼저 의심할 것. 특히 16진수처럼 표현 가능한 문자 집합이
좁은 값(해시 조각, 짧은 코드)은 일반 키워드와 같은 방식으로 비교하면 위험 — 애초에 제거
대상으로 분류. 매칭 결과가 "소수 그룹에 몰림"으로 보이면 개별 케이스를 고치기 전에 공통으로
등장하는 토큰(날짜, 정형 문구, URL 조각)부터 의심할 것.

## 2026-07-06 — 트위터 수집 스크롤이 조기 종료됨 (2단계)

**증상**: 실제 계정(`megahousestore`, 2026-07-01~07-02) 검증 중 원본 49건 중 필터링 후 32건이
정상이어야 하는데, 처음엔 3건에서 스크롤이 멈춰버림.

**1단계 원인**: `<time>`이 리트윗/고정 시점이 아니라 원본 게시물의 원래 작성 날짜를 보여줘서,
최근 리트윗한 옛날 글이 섞이면 스크롤 시작 직후 "범위 벗어남"으로 착각해 즉시 멈춤. **해결**:
`[data-testid="socialContext"]` 라벨(예: "OO님이 리트윗함")이 붙은 게시물은 수집 자체에서
제외(좋아요/RT수도 이 계정 성과가 아니라 원본 게시물 것이라 집계 관점에서도 제외가 맞음).

**2단계**: 그래도 원본 6건에서 조기 중단. 진단 로그로 원인 게시물을 특정한 결과 **"끌올"**
(인용/멘션으로 예전 글을 오늘 다시 피드 위로 끌어올리는 방식)이 원인 — 리트윗과 마찬가지로
오래된 작성일이 스크롤 최상단에 등장함. 리트윗→인용→멘션처럼 패턴이 계속 늘어날 걸로 보여서
패턴별 개별 예외처리 대신 **판단 기준 자체를 변경**: 오래된 게시물 1건만으론 안 멈추고, 새로
발견된 순서(피드 순서) 기준 연속 3건이 전부 범위 시작일 이전이어야 진짜로 과거로 넘어갔다고
판단. 재검증해서 32건으로 정상 수집 확인.

**재발 방지**: SNS 플랫폼의 "피드에 옛날 글이 다시 등장하는" 패턴(리트윗/고정/인용/끌올 등)은
계속 새로운 변형이 나올 수 있다는 전제로 설계할 것 — 패턴 이름으로 하나씩 예외처리하기보다
"연속 N건" 같은 통계적 판단 기준을 기본으로 채택. 스크롤 기반 수집기가 예상보다 일찍 멈추면
먼저 "가짜 조기 종료 신호(오래된 타임스탬프를 가진 비정상 위치의 게시물)"부터 의심할 것.

## 2026-07-06 — 인스타그램 수집이 날짜 무시하고 프로필 전체를 다 긁음

**증상**: 2026-07-01~07-02만 요청해도 프로필 그리드를 5월까지 계속 스크롤함.

**진단**: 트위터와 달리 인스타 수집은 1단계(그리드에서 링크만 모으는 단계)에 날짜 판단이 전혀
없어서 "스크롤해도 새 링크가 안 늘어남"(=프로필 전체) 조건까지 무조건 다 모으고, 날짜 필터링은
2단계(게시물 개별 파싱)에서야 적용되는 구조였음.

**해결**: 그리드 스크롤 중 새로 발견한 링크 중 가장 나중 것(가장 오래됐을 가능성이 큰 것)
하나를 별도 탭으로 슬쩍 열어 날짜만 확인(`peekPostDate()`, 메인 페이지 스크롤 위치는 안
건드림), 범위 시작일보다 오래되면 스크롤 중단하도록 수정. 좋아요/댓글 좌표 파싱(2단계) 로직은
안 건드리고 그대로 둠 — 재검증에서 정상 동작 확인.

**재발 방지**: 여러 단계(수집→필터링)로 나뉜 파이프라인을 만들 때, "필터 조건이 실제로 어느
단계에서 적용되는지"를 설계 시점에 명시적으로 확인할 것 — 늦은 단계에만 필터가 있으면 이른
단계가 불필요하게 전체 데이터를 훑는 비효율/오동작이 생길 수 있음. 비슷한 구조를 가진 다른
수집기(twitter.js)와 대조해서 "이 수집기만 다르게 동작하는 부분"이 있는지 확인하는 습관.

## 2026-07-06 — 시각차이가 "1496분"처럼 나와서 원인 파악 어려움

**증상**: PW/BH 게시 시각차이 계산 결과가 "1496분"처럼 비정상적으로 크게 나와서 혼란.

**근본 원인**: `formatKstTime()`이 "시:분"만 표시하고 날짜("M/D")를 표시하지 않아서, PW/BH가
하루 넘게 차이나는 날 각각 게시한 경우 계산 자체는 맞는데(실제로 1496분 차이) 눈으로 봤을 때
왜 이렇게 큰지 판단할 근거가 화면에 없었음.

**해결**: `formatKstTime()`이 날짜("M/D")도 같이 표시하도록 수정. `verify-mock.js`에 회귀
테스트 추가.

**재발 방지**: 시간/기간 관련 값을 화면에 표시할 때는 "계산이 맞는지"와 별개로 "그 값이 왜
그렇게 나왔는지 표시만 보고 판단 가능한지"를 항상 같이 확인할 것 — 특히 날짜 경계를 넘나드는
값은 날짜 정보 없이 절대 표시하지 않기.

## 2026-07-07 — 자사 게시물 "다시 리트윗"이 원본 게시물 자체를 통째로 누락시킴

**증상**: 사용자가 매출 1위 상품의 PW 링크를 줬는데 캐시(`_last-collection.json`)에 아예 없음.
BH 쪽 대응 게시물은 정상 수집됨 — 매칭 실패가 아니라 수집 단계에서 통째로 빠진 것.

**진단 과정**: 트위터 status ID에 타임스탬프가 인코딩돼 있어서(snowflake) 디코딩해보니 실제
게시 시각은 수집 기간(7/1~7/2) 안이었음. PW 수집 타임라인상 그 시각대에 정확히 이 게시물이
있어야 하는데 통째로 없어서, 날짜 밖이라 걸러진 게 아니라 뭔가 이 한 건만 특이하게 누락됐다고
판단.

**근본 원인**: `twitter.js`의 `collectOnce()`에서 리트윗/고정 게시물 제외 로직이
`socialContext` 라벨(예: "OO님이 리트윗함")이 있으면 통째로 스킵하면서 그 링크를
`_seenLinks`에 추가해버림. 그런데 이 계정은 고정이 아니라 **주요 상품을 자기가 다시 리트윗하는
습관**이 있음 — 자사 원본 게시물을 다시 리트윗하면 피드 맨 위에 이 부스트된 카드(socialContext
라벨 있음, 원본 링크 그대로)로 먼저 마주치는데, 이때 링크를 `_seenLinks`에 넣어버리면 스크롤이
내려가서 이 게시물의 자연스러운(라벨 없는) 위치에 도달했을 때도 "이미 처리함"으로 오인해서 두
번 다 놓쳐버림 — 원본 데이터를 한 번도 저장 못 하고 완전히 사라짐.

**해결**: `socialContext`(리트윗/고정) 및 인용 트윗 중첩 카드 제외 분기에서
`_seenLinks.add(link)` 제거 — 스킵만 하고 링크는 "본 것"으로 마킹 안 함. 그래야 스크롤 중 이
카드가 화면에 남아있는 동안엔 매 tick마다 계속 스킵되지만(부작용 없음), 나중에 자연스러운
(라벨 없는) 위치에 도달하면 정상적으로 수집됨.

**재발 방지**: "이 게시물은 건너뛴다"와 "이 링크는 이미 처리했다고 기록한다"를 항상 분리해서
생각할 것 — 같은 링크가 여러 형태(부스트된 카드/자연스러운 위치)로 반복 등장할 수 있는
무한스크롤 수집기에서는, 스킵 사유가 있다고 곧바로 seen-set에 넣으면 나중에 나올 정상 케이스를
막아버릴 수 있음. "캐시에 아예 없는" 누락은 매칭 문제가 아니라 수집 단계 문제일 가능성을 먼저
의심.

## 2026-07-08 — naver-stock.js 순수 fetch가 HTTP 429로 막힘

**증상**: 로그인 세션 없이 네이버 공개 상품 페이지를 읽는 `naver-stock.js` 1차 버전을 순수
`fetch()` GET으로 구현했는데, 샌드박스에서 실행 시 네이버 엣지(`server: nfront`)가 정적 429
에러 페이지를 반환.

**진단 과정**: 처음엔 "데이터센터 IP 차단"으로 추정했으나, 사용자가 본인 로컬(주거용 IP)
컴퓨터에서 `node naver-stock.js <URL>`을 실행해봐도 똑같이 HTTP 429가 나서 이 가설이 틀렸음을
확인.

**근본 원인**: IP 평판이 아니라 네이버 프론트가 TLS/브라우저 지문(fingerprint)으로 봇을
거르는 것으로 판단 — Node의 fetch/TLS 스택은 실제 크로미움과 지문이 달라서 걸림. `twitter.js`/
`instagram.js`가 처음부터 Playwright로 간 것과 같은 이유.

**해결**: `naver-stock.js`도 순수 fetch 대신 Playwright(실제 크로미움)로 전환. 로그인 세션은
여전히 불필요(공개 페이지라 새 컨텍스트로 그냥 열면 됨). 사용자 로컬 컴퓨터에서 실제 검증

## 2026-07-09 — 인스타가 정상 세션인데도 계속 재로그인 요구

**증상**: 사용자 질문 — "세션이 왜 자꾸 로그인하라고 뜨는거야?"

**진단 과정**: 먼저 확인해서 답변한 것 — Playwright가 띄우는 브라우저 창은 북마크바/확장
프로그램/저장된 자동완성이 하나도 없는 완전 새 프로필이라 겉모습이 시크릿창과 비슷해 보이는

**근본 원인**: 진짜 원인으로 의심되는 건 `navigator.webdriver` — Playwright/Selenium 등
자동화 브라우저는 이 값이 기본적으로 `true`라서, 메타(인스타)가 이걸 감지하면 쿠키가 유효해도
"의심스러운 기기"로 판단해 재로그인/체크포인트를 요구할 수 있음. 트위터(X)는 이 부분에 덜
엄격해서 지금까지 잘 됐던 것으로 보임.

**해결**: `chromium.launch()`에 `args:['--disable-blink-features=AutomationControlled']`
추가 + `context.addInitScript()`로 `navigator.webdriver`를 `undefined`로 덮어씀 —
`instagram.js`와 `login-session.js`(로그인 자체를 하는 브라우저이니 여기서부터 자동화 흔적을
지워야 함)에 적용. `twitter.js`는 지금 잘 되고 있어서 안 건드림. ⚠️ 100% 해결책인지는 검증
못 함(메타의 탐지는 이 값 하나만 보는 게 아닐 수 있음) — 사용자가 실제로 세션을 다시 만든 뒤
며칠 지나서 재확인 필요한 상태로 남겨둠(정직하게 불확실성 표기).

**재발 방지**: 자동화 브라우저 특유의 탐지 신호(`navigator.webdriver` 등)는 로그인을 직접
수행하는 스크립트(`login-session.js`)에도 처음부터 같이 적용해둘 것 — 세션을 "만드는" 시점과
"쓰는" 시점 중 한쪽만 흔적을 지우면 의미가 없음. 확실하지 않은 해결책은 확실하지 않다고 그대로
남겨두고, 사용자의 실사용 재확인을 다음 액션으로 명시해둘 것(검증 없이 "해결됨"으로 단정하지

## 2026-07-10 — 재고 추이 그래프가 "변화 없이 일자로 보인다" — 3단계 시행착오

**1차: 원본 재고 수량을 그대로 그림**. PW/BH 둘 다 재고 수량(예: PW 9999개, BH 100개)을
한 축에 겹쳐 그렸는데, 사용자가 "그래프를 열어봤는데 변화 없이 일자다"라고 지적.

**진단(1차 원인)**: PW/BH가 서로 다른 상품이라 초기 판매한도 자체가 다름 — 재고는 그 큰
고정값에서 조금씩 줄어드는 구조라, 그 큰 값이 축을 지배해버려서 실제 변화폭이 작아 보임.

**2차: 최초 관측 시점=100 지수화로 전환**. "PW/BH 각자 시작점 대비 몇 %인지"로 정규화하면
스케일 차이 문제가 해결될 거라 판단해서 적용. 그런데 사용자가 "지수화 이런 거 필요 없고
그냥 총판매추정 개수 그대로 그리면 되잖아"라고 반려 — 사용자는 원본 값 그대로를 보고
싶어했음.

**3차: 총판매추정(개) 값을 그대로 그림**. 지수화를 걷어내고 `estimateInitialCap(stock) -
stock`(총판매추정치)을 그대로 그리도록 변경. 이번엔 축이 큰 고정값에 안 눌릴 거라 예상함.

**진단(2차, 더 깊은 원인)**: 그런데도 사용자가 "여전히 일자인데?"라고 재차 지적. 사용자가
올려준 실제 HTML 파일을 직접 열어서 데이터를 뜯어본 결과: PW 총판매추정치가 999→994→996개
처럼 **5개 안팎의 아주 좁은 범위**에서만 움직이고 있었는데, 이걸 BH(386개, 실제로 변화
없음)와 **같은 축**(0부터 시작하는 축)에 그리다 보니 그 5개짜리 변화가 축 전체(0~1000+)의
0.5%도 안 돼서 또 안 보인 것. 원인은 "축을 0부터 강제로 시작한 것" — 총판매추정으로
바꿔도 여전히 존재하던 문제였음.

**최종 해결**: PW/BH를 각자 축을 가진 두 개의 작은 그래프(위아래로 쌓음, small
multiples)로 분리. 각 축은 그 시리즈 자체의 최소~최대 범위로 확대하고 0을 강제하지 않음
(주가 차트처럼). 실측 데이터(PW 994~999, BH 386 고정)로 재현 테스트까지 거쳐서 확인함 —
BH는 실제로 안 팔린 게 맞고(진짜 데이터, 버그 아님), PW의 미세한 움직임만 이제 제대로
보임.

**재발 방지**: 두 시리즈의 스케일(값 크기, 변동폭)이 다를 가능성이 있으면 공유축부터
의심할 것 — 지수화가 아니라 각자 축(small multiples)을 먼저 검토할 것. 사용자가 "여전히
이상하다"고 재차 지적하면 새 가설을 세우기 전에 실제 데이터를 직접 추출해서 숫자로 확인할
것(이번에 실제 HTML을 뜯어봐서 999→994→996 vs 386이라는 근거를 확보하고서야 진짜 원인을
찾음).

## 2026-07-10 — PW/BH 재고 상품 매칭률이 너무 낮음 (51개 중 14쌍)

**증상**: 사용자가 "SNS 표에서는 30쌍 나왔는데 재고 종합표는 14쌍밖에 안 된다, 매칭률이
너무 안 좋다"고 지적.

**진단 과정**: 처음엔 SNS 매칭 로직(aggregate.js의 extractKeywords/detectProductLine,
Union-Find 그룹화)을 그대로 재사용하면 될 거라 판단하고 그렇게 구현했었음. 사용자가 실제
업로드한 리포트의 PW 51개/BH 57개 상품명 전체를 직접 추출해서, 같은 알고리즘으로 돌려보며
근본 원인을 찾음.

**근본 원인**: Union-Find는 전이적으로 묶는 방식이라(A-B, B-C가 연결되면 A-C까지 한
그룹), 같은 프랜차이즈의 여러 변형 상품(예: "리바이 단품", "엘런 단품", "엘런+리바이
세트")이 프랜차이즈명("진격", "거인")만 겹쳐도 서로 연결돼서 PW 여러 개 + BH 여러 개가
뒤섞인 큰 그룹이 됨. 코드가 "그 그룹 안에 PW 1개, BH 1개일 때만 매칭"하도록 안전장치를
걸어놨었기 때문에, 이런 뒤섞인 그룹은 통째로 버려짐 — 정작 "리바이 단품"끼리는 명백히
서로 제일 잘 맞는 진짜 짝이었는데도 놓친 것. SNS 매칭(게시물 본문 텍스트, 정보량이 많음)엔
문제가 안 됐지만, 재고 상품명(짧고 반복되는 문구가 많음)엔 이 방식이 안 맞았음.

**해결**: Union-Find(그룹화) 대신 PW-BH 쌍마다 직접 자카드 유사도 점수(겹친 키워드 수 ÷
두 상품명 중 키워드가 더 많은 쪽 개수)를 매기고, 서로가 서로를 1순위로 고를 때만("상호
최선") 확정 매칭하는 방식으로 교체. 점수가 완전히 동률인 경우(예: BH가 같은 상품을
"박스 구성"/"단품 랜덤" 두 SKU로 중복 등록해서 PW 하나가 BH 둘과 똑같이 겹치는 경우)는
여전히 매칭하지 않음(잘못 짝짓는 것보다 안전).

**결과**: 실제 데이터 기준 14쌍 → 27쌍으로 개선. 남은 ~24개 불일치는 대부분 BH가 상품을
2개 SKU로 중복 등록한 구조적 차이(진짜 애매해서 의도적으로 안 묶은 것) + 일부 번역 차이
(PW "테노히라"/BH "손바닥", PW "에렌"/BH "엘런")로, 후자는 필요시 별칭 목록을 추가하면
더 개선 가능.

**재발 방지**: 짧고 반복되는 텍스트(상품명, SKU명 등)를 매칭할 때는 Union-Find류 전이적
그룹화보다 쌍별 점수 기반 매칭("상호 최선")을 먼저 검토할 것 — 전이적 그룹화는 정보량이
많은 자유 텍스트(게시물 본문)에는 맞지만, 짧고 공통 토큰(브랜드명 등)이 많은 텍스트에서는
쉽게 오염됨. 같은 매칭 로직이라도 입력 텍스트의 성격(길이, 반복도)이 다르면 그대로
재사용하지 말고 적합성부터 재검토할 것. 매칭률이 다른 유사 기능(SNS 매칭) 대비 크게
낮으면 임계값 조정이 아니라 알고리즘 구조 자체를 의심하고, 실제 전체 데이터로 재현해서
확인할 것.

## 2026-07-13 — 테스트하느라 여러 번 재실행했더니 엑셀 시트가 20~30개로 무한 증식

않고 항상 새 시트를 추가"하도록 설계돼 있었음(`20260710_20260711`, `20260710_20260711 (2)`,

방침으로 의도적으로 만든 동작이었는데, 실제로는 **진짜 다른 기간을 여러 번 수집하는 경우**보다

**해결**: 시트 이름을 수집 기간(`startDate_endDate`) 하나로 고정하고, `replaceWorksheet()`를

커녕 더 나빠졌다는 건, "가끔 뜨는 모달이라 다시 열면 없어진다"는 가설 자체가 틀렸다는
뜻 — 실제로는 (a) 세션 자체가 그 사이 더 나빠졌거나 (b) 애초에 좋아요 숫자가 그 좌표/형식
으로 안 나오는 상태(레이아웃 변경, 계정의 좋아요 수 비공개 설정 등)라 몇 번을 다시 열어도
안 나올 근본적인 문제일 가능성이 큼. 재시도는 실패하는 케이스에서도 매번 페이지를 새로
열고 2초씩 기다리므로, 100% 실패 상황에서는 순수하게 실행 시간만 3배로 늘리는 부작용만
남음.

**해결(수정)**: 1차 시도(재시도 조건 확장 + Esc 키)를 되돌림 — 검증 안 된 가설로 시간만
잡아먹는 코드를 그대로 두지 않고, 원인을 다시 좁힐 때까지 원래 동작(1회 시도, 실패하면
그대로 null)으로 복구. 다음 확인 필요 사항: 사용자가 문제의 게시물 URL을 본인 계정으로
직접 열어서 "좋아요 숫자가 화면에 실제로 보이는지"부터 확인 필요 — 안 보이면(계정이 좋아요
수를 비공개로 설정했거나 인스타가 그 계정엔 숫자를 안 보여주는 경우) 코드로 고칠 수 있는
문제가 아니고, 보이는데도 스크래퍼가 못 읽으면 좌표/텍스트 패턴이 바뀐 것이라 그 위치 기준
자체를 다시 잡아야 함.

**재발 방지**: 가설(특히 "가끔 나타나는 문제라 재시도하면 될 것") 기반으로 수정할 땐, 실제
재검증 결과를 받기 전까지 "이걸로 해결됐다"고 단정하지 말 것. 이번처럼 재시도가 오히려
전보다 더 나쁜 결과(100% 실패)를 보이면, 가설 자체가 틀렸다는 강한 신호로 보고 즉시
되돌릴 것 — 근거 없이 재시도 횟수/대기 시간만 계속 늘리면 실행 시간만 낭비하고 문제는 안
풀림. 여러 필드를 한 번에 파싱하는 코드에서 재시도 여부를 "결과 객체가 있는지"로만 판단하지
않는 것 자체는 여전히 유효한 개선 방향이지만, "왜 실패하는지"를 먼저 확정하지 않은 채
재시도부터 넣는 순서가 잘못이었음 — 코드 수정보다 먼저 "사람이 직접 그 페이지를 보면 뭐가
보이는지" 확인하는 게 우선이었어야 함.
