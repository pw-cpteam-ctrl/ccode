## 2026-07-10 — totalSold(총판매추정치)가 1개/6개/8개처럼 터무니없이 작게 나옴

**증상**: 재고 스냅샷 기반 총판매추정치가 실제 예약판매 규모에 비해 비정상적으로 작게
나옴(재고 8,616개짜리 상품이 "6개 판매추정"으로 표시됨).

**진단 과정**: 당시 로직은 "이 상품을 우리가 실제로 스냅샷 찍어본 적 있으면(hasRealBaseline)
그 최초 관측 재고 − 현재재고"를 우선 계산하고 있었음. 처음엔 이게 버그로 보였는데, 계산 자체는
로직대로 맞게 동작하고 있었음 — 문제는 재고 스냅샷 기능을 쓴 지 며칠 안 돼서 "우리가 추적한
이후" 구간 자체가 짧다는 점이었음.

**근본 원인**: 메가하우스 피규어는 예약판매가 보통 몇 달째 진행 중인데, 우리 관측 시작점이
그보다 훨씬 늦게 시작됐다는 이유만으로 총판매추정치가 실제보다 작게 나온 것. 알고리즘 버그가
아니라 "관측 시작 시점"을 기준으로 삼은 설계 자체의 결함.

**해결**: 실제 과거 관측 기록 유무와 무관하게, **항상** 메가하우스 예약판매 관행(9999/10000
같은 "깔끔한" 숫자로 판매 한도를 걸어두는 것)을 이용해 현재 재고 위의 가장 가까운 1000단위를
"초기 한도"로 가정하고 역산(`estimateInitialCap(stock) - stock`)하는 방식으로 통일. 항상
추정치이므로 `totalSoldIsEstimated`는 항상 true, 화면엔 "*"로 표시.

## 2026-07-10 — 재고 스냅샷 종합표: 어떤 시각화가 맞는지 논의만으로 여러 번 방향 전환

**배경**: 상품이 30개쯤 되니 PW/BH의 SNS+매출 성과를 그래프로 보고 싶다는 요청. 시작 전에
사용자가 먼저 상의를 요청해서, 프로토타입 이미지를 여러 개 만들어 비교함.

**1차 시도 — 산점도(안A)**: x축=SNS 지표, y축=매출 추정치로 30개 상품을 점으로 흩뿌리는
안을 "추천안"으로 제시했는데, 사용자가 "하나도 못 알아보겠다, 추천한 의미가 없다"고 반려.
평소 산점도를 보지 않는 대상에게는 정보 밀도보다 친숙함이 더 중요하다는 걸 놓친 판단이었음.

**2차 — 막대+꺾은선 콤보(안D)**: 사용자가 참고 이미지(정부 통계 인포그래픽)를 보여주며
막대+꺾은선 이중축 콤보를 원했는데, 처음엔 "이중축 차트는 데이터 시각화 안티패턴"이라는
이유로 지레 만들지 않고 다른 대안만 제시함 → 사용자가 "내가 만들라는 것만 쏙 빼놓고
만들었다"고 지적. **교훈**: 원칙적으로 권장하지 않는 방식이라도, 사용자가 명시적으로
요청했으면 일단 만들어서 보여주고 트레이드오프를 설명하는 게 맞음 — 선제적으로 판단해서
빼면 안 됨.

**해결**: 안D(막대+꺾은선, PW 매출 왼쪽축/BH SNS 오른쪽축)를 실제로 만들어 제시, 이후
30개 항목이 빽빽하다는 문제는 16개 단위로 페이지 분할하는 것으로 합의.

## 2026-07-10 — 재고 스냅샷 종합표에 매출순 정렬이 안 되어 있었음

**증상**: 사용자가 실제 리포트를 열어보고 "많이 팔린 순 정렬이 안 돼 있다"고 지적.

**근본 원인**: `buildIntegratedStockRows`에서 PW/BH 매칭 페어를 만든 뒤, 정렬 로직 자체를
빼먹음 — union-find/매칭 결과가 나온 순서(사실상 입력 순서) 그대로 반환하고 있었음. 단순
누락 버그.

**해결**: PW+BH 총판매추정 합산 기준 내림차순 정렬 추가.

## 2026-07-10 — 재고 추이 그래프가 "변화 없이 일자로 보인다" — 3단계 시행착오

**1차: 원본 재고 수량을 그대로 그림**. PW/BH 둘 다 재고 수량(예: PW 9999개, BH 100개)을
한 축에 겹쳐 그렸는데, 사용자가 "그래프를 열어봤는데 변화 없이 일자다"라고 지적.

**진단(1차 원인)**: PW/BH가 서로 다른 상품이라 초기 판매한도 자체가 다름 — 재고는 그 큰
고정값에서 조금씩 줄어드는 구조라, 그 큰 값이 축을 지배해버려서 실제 변화폭이 작아 보임.

**2차: 최초 관측 시점=100 지수화로 전환**. "PW/BH 각자 시작점 대비 몇 %인지"로 정규화하면
스케일 차이 문제가 해결될 거라 판단해서 적용. 그런데 사용자가 "지수화 이런 거 필요 없고
그냥 총판매추정 개수 그대로 그리면 되잖아"라고 반려 — 사용자는 원본 값 그대로를 보고
싶어했음.

**3차: 총판매추정(개) 값을 그대로 그림**. 지수화를 걷어내고 `estimateInitialCap(stock) -
stock`(총판매추정치)을 그대로 그리도록 변경. 이번엔 축이 큰 고정값에 안 눌릴 거라 예상함.

**진단(2차, 더 깊은 원인)**: 그런데도 사용자가 "여전히 일자인데?"라고 재차 지적. 사용자가
올려준 실제 HTML 파일을 직접 열어서 데이터를 뜯어본 결과: PW 총판매추정치가 999→994→996개
처럼 **5개 안팎의 아주 좁은 범위**에서만 움직이고 있었는데, 이걸 BH(386개, 실제로 변화
없음)와 **같은 축**(0부터 시작하는 축)에 그리다 보니 그 5개짜리 변화가 축 전체(0~1000+)의
0.5%도 안 돼서 또 안 보인 것. 원인은 "축을 0부터 강제로 시작한 것" — 총판매추정으로
바꿔도 여전히 존재하던 문제였음.

**최종 해결**: PW/BH를 각자 축을 가진 두 개의 작은 그래프(위아래로 쌓음, small
multiples)로 분리. 각 축은 그 시리즈 자체의 최소~최대 범위로 확대하고 0을 강제하지 않음
(주가 차트처럼). 실측 데이터(PW 994~999, BH 386 고정)로 재현 테스트까지 거쳐서 확인함 —
BH는 실제로 안 팔린 게 맞고(진짜 데이터, 버그 아님), PW의 미세한 움직임만 이제 제대로
보임.

## 2026-07-10 — PW/BH 재고 상품 매칭률이 너무 낮음 (51개 중 14쌍)

**증상**: 사용자가 "SNS 표에서는 30쌍 나왔는데 재고 종합표는 14쌍밖에 안 된다, 매칭률이
너무 안 좋다"고 지적.

**진단 과정**: 처음엔 SNS 매칭 로직(aggregate.js의 extractKeywords/detectProductLine,
Union-Find 그룹화)을 그대로 재사용하면 될 거라 판단하고 그렇게 구현했었음. 사용자가 실제
업로드한 리포트의 PW 51개/BH 57개 상품명 전체를 직접 추출해서, 같은 알고리즘으로 돌려보며
근본 원인을 찾음.

**근본 원인**: Union-Find는 전이적으로 묶는 방식이라(A-B, B-C가 연결되면 A-C까지 한
그룹), 같은 프랜차이즈의 여러 변형 상품(예: "리바이 단품", "엘런 단품", "엘런+리바이
세트")이 프랜차이즈명("진격", "거인")만 겹쳐도 서로 연결돼서 PW 여러 개 + BH 여러 개가
뒤섞인 큰 그룹이 됨. 코드가 "그 그룹 안에 PW 1개, BH 1개일 때만 매칭"하도록 안전장치를
걸어놨었기 때문에, 이런 뒤섞인 그룹은 통째로 버려짐 — 정작 "리바이 단품"끼리는 명백히
서로 제일 잘 맞는 진짜 짝이었는데도 놓친 것. SNS 매칭(게시물 본문 텍스트, 정보량이 많음)엔
문제가 안 됐지만, 재고 상품명(짧고 반복되는 문구가 많음)엔 이 방식이 안 맞았음.

**해결**: Union-Find(그룹화) 대신 PW-BH 쌍마다 직접 자카드 유사도 점수(겹친 키워드 수 ÷
두 상품명 중 키워드가 더 많은 쪽 개수)를 매기고, 서로가 서로를 1순위로 고를 때만("상호
최선") 확정 매칭하는 방식으로 교체. 점수가 완전히 동률인 경우(예: BH가 같은 상품을
"박스 구성"/"단품 랜덤" 두 SKU로 중복 등록해서 PW 하나가 BH 둘과 똑같이 겹치는 경우)는
여전히 매칭하지 않음(잘못 짝짓는 것보다 안전).

**결과**: 실제 데이터 기준 14쌍 → 27쌍으로 개선. 남은 ~24개 불일치는 대부분 BH가 상품을
2개 SKU로 중복 등록한 구조적 차이(진짜 애매해서 의도적으로 안 묶은 것) + 일부 번역 차이
(PW "테노히라"/BH "손바닥", PW "에렌"/BH "엘런")로, 후자는 필요시 별칭 목록을 추가하면
더 개선 가능.
