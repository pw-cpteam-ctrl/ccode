## 2026-07-22

### 업데이트 받았는데 새로 추가된 기능이 오류를 내는 문제
- **무슨 문제/요청이 있었나**: 스크린샷 기능이 추가된 뒤 업데이트를 받았는데, 특정 부품
  파일을 찾을 수 없다는 오류가 뜨며 실행이 안 됨.
- **확인해보니**: 업데이트 과정은 원래 "코드만 새로 받아오고, 이미 설치돼 있는 부품들은
  다시 안 건드린다"는 방식으로 짜여 있었음(복사 과정에서 `node_modules` 폴더는 항상
  제외). 그런데 스크린샷 기능을 만들면서 새로운 부품(`html2canvas`)이 하나 추가됐는데,
  업데이트 과정이 이 새로 추가된 부품까지는 자동으로 설치해주지 않아서 생긴 문제였음.
- **어떻게 처리했나**: 업데이트 과정(`update.bat`) 마지막에, 새로 추가된 부품이 있으면
  자동으로 같이 설치하는 단계(`npm install`)를 추가.
- **결과**: 이후로는 업데이트 한 번으로 새로 추가된 부품까지 같이 설치되도록 함. (다음
  실제 업데이트 때 정상 설치되는지는 계속 지켜볼 부분)

### 리포트 내용을 노션에 자동으로 보내고 싶다
- **무슨 문제/요청이 있었나**: 팀에서 쓰는 노션 문서에 리포트 표를 매번 손으로 옮기지 않고
  자동으로 보낼 방법을 문의.
- **확인해보니**: 노션과 직접 연결하는 정식 방법(노션 API 연동)이 있지만, 이 방법을 쓰려면
  회사 노션 관리자가 미리 "이 연결을 허용한다"고 승인해줘야 하는 구조였음 — 실제로
  시도해보니 이 승인이 회사 정책상 막혀 있어서, 코드로는 우회할 수 없는 관리자 영역
  문제로 확인됨. 관리자에게 따로 요청하는 건 부담스럽다는 뜻을 밝혀서, 그 대신 다른
  방법을 찾기로 함.
- **어떻게 처리했나**: 우선 노션 쪽 승인 없이도 바로 쓸 수 있는 방법으로, 리포트 내용을
  노션에 붙여넣기 좋은 표 형태의 글로 자동으로 만들어서 복사 버튼 하나로 가져갈 수 있게
  만듦(`buildMarkdownExport()`). 이후 알아보니 개인(또는 회사) 노션 계정으로 직접 연결을
  새로 만드는 방법(내부 통합)은 관리자 승인 없이도 가능하다는 걸 확인해서, 이 방법으로
  실제 자동 전송 기능(`notion-export.js`)도 별도로 만듦 — 개인 계정으로 연결을 새로
  만들고, 리포트를 보낼 노션 페이지에 그 연결을 허용해주기만 하면 버튼 한 번으로 표가
  그 페이지에 만들어지는 방식.
- **결과**: 복사-붙여넣기 방식은 바로 확인됨. 자동 전송 방식은 처음에 페이지 위치를
  잘못 지정해서(다른 예시 페이지 주소를 그대로 씀) 안 됐었는데, 본인 계정 안의 올바른
  페이지로 다시 설정한 뒤 실제로 노션에 표가 만들어지는 것까지 화면으로 확인함.

### 노션 표에 색깔·그래프 등 시각적으로 보기 좋게 개선
- **무슨 문제/요청이 있었나**: 자동 전송되는 노션 표가 글자만 있어서 밋밋하다며, 자사/
  경쟁사를 색깔로 구분해달라는 요청. 이어서 색깔 구분만으로는 한눈에 비교가 잘 안 된다며,
  막대그래프처럼 크기를 눈으로 비교할 수 있는 형태로 바꿀 수 있는지 문의. 또 이왕이면
  노션에서 흔히 보이는 트윗 카드 형태(2칸 나열)로 넣을 수 있는지도 확인 요청.
- **확인해보니**: 노션 표 셀 하나에 배경색을 입히는 것과, 글자를 색깔 블록 여러 개로
  나눠 넣어서 막대그래프처럼 보이게 만드는 것 둘 다 가능함을 확인. 다만 트윗을 눌러서
  붙여넣을 때 노션이 자동으로 만들어주는 "진짜 카드" 형태는, 노션이 화면에서 사람이
  직접 붙여넣을 때만 특별히 만들어주는 것이라 프로그램을 통한 자동 전송 방식으로는
  똑같이 재현이 불가능하다는 것을 실제 테스트로 확인함(직접 시도해봤지만 일반 링크
  형태로만 들어가고 카드로는 안 바뀜).
- **어떻게 처리했나**: 자사는 파란색, 경쟁사는 빨간색 배경(`blue_background`/
  `red_background`)을 표에 입히고, 우세/경합/약세 결과도 색깔로 구분되게 추가. 이어서
  숫자 칸 옆에 자사/경쟁사 비율만큼 색깔 블록을 채운 막대그래프 칸(`buildBarCell()`)을
  새로 추가. 트윗 카드 형태는 재현이 안 된다는 점을 안내하고, 그 부분은 보류.
- **결과**: 실제로 노션에 전송해본 화면을 확인 — 색깔 구분과 막대그래프 모두 정상적으로
  보임. 매우 만족스럽다는 반응 확인.

### 노션 표 칸 이름 줄이기 + 칸 너비가 매번 초기화되는 문제
- **무슨 문제/요청이 있었나**: 표에 그래프 칸까지 늘어나며 자리가 좁아지자, 칸 이름을
  줄여 공간을 아껴달라는 요청. 이어서 노션에서 손으로 칸 너비를 맞춰놔도 다음에 다시
  보낼 때마다 원래대로 되돌아가는 게 불편하다며, 칸 너비를 아예 프로그램에서 지정할 수
  있는지 문의.
- **확인해보니**: 칸 이름은 글자 대신 짧은 그림 문자(이모지) 하나로도 표현 가능해서
  공간을 크게 아낄 수 있음을 확인. 칸 너비는, 노션이 표를 다시 만들 때마다 매번 "새 표"로
  인식해서 예전에 손으로 맞춘 너비 정보가 통째로 사라지는 게 원인이었음 — 그리고 칸
  너비 자체를 프로그램에서 직접 지정하는 기능은, 노션이 공식적으로 지원한다고 확실히
  확인되지 않은 부분이라 섣불리 손대면 오히려 지금 되던 것까지 깨질 위험이 있다고 판단.
- **어떻게 처리했나**: 칸 이름을 전부 짧은 그림 문자 하나로 교체(`FIELD_LABELS`를 이모지로
  변경). 칸 너비 문제는, 매번 새 표를 만드는 대신 처음 한 번만 표를 만들고(그때 페이지·
  표 블록 위치를 `notion-config.json`에 저장) 그다음부터는 그 표 안의 내용(행)만 지우고
  새로 채워 넣는 방식으로 구조를 바꿔서, 노션에 저장된 표 자체가 유지되게 함(표가
  유지되면 손으로 맞춘 너비도 같이 유지될 것으로 기대).
- **결과**: 칸 이름 압축은 바로 확인됨. 칸 너비 유지 방식은 "이상하면 그냥 안 쓰기로" 하는
  조건으로 우선 적용해두었고, 실제로 다음 전송부터 정말 같은 표가 갱신되고 너비가
  유지되는지는 아직 실기기에서 확인 전 — 확인되는 대로 이 기록에 결과를 이어서 남길 예정.

## 2026-07-21

### 트위터 로그인이 맞는 비밀번호도 계속 틀렸다고 거부하다가 결국 잠기는 문제
- **무슨 문제/요청이 있었나**: 로그인 화면에서 분명 맞는 비밀번호를 쳤는데 "틀렸다"는
  안내만 계속 뜨고, 다른 방법으로도 그 비밀번호가 맞다는 걸 재차 확인했는데도 마찬가지.
  결국 여러 번 시도하다 보니 "시도 횟수를 초과했다"며 계정이 한동안 잠김. "일단 기다렸다
  다시 해보자" 정도의 대응은 근본적인 해결이 아니라며, 방식 자체를 바꿔달라고 요청.
- **확인해보니**: 자동화 프로그램이 로그인할 때 쓰는 전용 내장 브라우저를 트위터 쪽에서
  "사람이 아닐 수도 있다"고 의심해서 맞는 비밀번호도 그냥 차단해버리고 있었던 것으로
  확인됨. 즉 비밀번호 문제가 전혀 아니라, 로그인하는 브라우저 자체가 의심받고 있었던 것.
- **어떻게 처리했나**: 자동화 전용 내장 브라우저 대신, 이 컴퓨터에 실제로 깔려 있는 평소
  쓰는 크롬을 그대로 빌려서 로그인하도록 방식을 바꿈(`chromium.launchPersistentContext`로
  시스템 크롬을 전용 프로필에서 실행). 다만 평소 쓰는 개인 크롬 창과 완전히 분리된,
  로그인 전용의 새 크롬 서랍(프로필)을 하나 따로 만들어서(`chrome-profile/{platform}`
  폴더) 그 안에서만 로그인하고 저장하게 함 — 평소 쓰던 크롬 기록·로그인 정보와는 전혀
  섞이지 않음. 혹시 이 컴퓨터에 크롬이 아예 안 깔려 있는 경우를 대비해서, 그럴 땐 예전
  방식(자동화 전용 내장 브라우저, `chromium.launch()`)으로 자동으로 대신 실행되도록
  안전장치도 같이 마련.
- **결과**: 실제로 다시 로그인을 시도해본 결과 더 이상 비밀번호가 거부되지 않고 정상적으로
  처리됨을 확인.

### 리포트에서 트위터/인스타 표만 이미지로 따로 저장하고 싶다
- **무슨 문제/요청이 있었나**: 리포트 화면 전체가 아니라, 트위터 표와 인스타 표를 각각
  따로 이미지 파일로 저장할 수 있는 버튼을 추가해달라는 요청. 오류만 없으면 된다는 조건.
- **확인해보니**: 화면의 특정 영역만 그림으로 떠내는 별도 부품(`html2canvas`)을 새로
  가져오면 구현 가능. 처음 만들어서 테스트해보니, 캡처 버튼 글자와 원형 비율 그래프
  (그러데이션 형태를 이 부품이 못 그려서 회색으로 깨져 보임)가 저장된 이미지에 그대로
  찍혀 나오는 문제가 있었음.
- **어떻게 처리했나**: 캡처하는 그 순간에만 캡처 버튼과 원형 그래프를 잠깐 안 보이게
  숨겼다가(`visibility:hidden`), 캡처가 끝나면 다시 원래대로 보이게 처리. 저장되는 파일
  이름이 엉뚱한 기본값으로 나오는 문제도 있어서, 파일을 내려받는 방식을 `canvas.toBlob()`
  + `URL.createObjectURL()`로 교체.
- **결과**: 실제로 버튼을 눌러 트위터 표/인스타 표 각각을 캡처해본 결과, 불필요한 버튼이나
  깨진 그래프 없이 깔끔한 이미지로 저장되는 것을 확인.

## 2026-07-20

### 집에서 팀 배포용 꾸러미 처음부터 만들기
- **무슨 문제/요청이 있었나**: 회사 컴퓨터가 아니라 집 컴퓨터에서, 아무것도 안 깔려 있는
  상태부터 팀에 나눠줄 배포판을 만들 수 있는지 문의. 깃(버전 관리 프로그램)도 안 깔려
  있었음.
- **확인해보니**: 처음부터 순서대로 안내하면 되는 작업이었음 — 깃 설치, 프로그램 실행에
  필요한 별도 프로그램 내려받아 폴더 안에 넣기, 필요한 부품들과 크롤링용 내장 브라우저까지
  그 폴더 안에 설치.
- **어떻게 처리했나**: 각 단계를 하나씩 순서대로 안내하면서 중간중간 나오는 화면/오류를
  같이 보면서 진행. 중간에 인터넷 저장 공간 문제로 다운로드가 멈추는 것처럼 보이는 순간이
  있었으나 실제로는 다운로드가 끝까지 된 것으로 확인.
- **결과**: 대시보드 화면이 뜨고 로그인 버튼을 눌러 실제 로그인 창이 뜨는 것까지 확인 —
  배포판 재료 준비가 끝남.

### 배포용 실행 파일을 열면 한글이 깨져서 실행이 안 되는 문제 (여러 차례 반복)
- **무슨 문제/요청이 있었나**: 대시보드를 켜는 실행 파일과 업데이트용 실행 파일을 열면
  화면에 알아볼 수 없는 글자가 뜨면서 실행이 안 됨. 파일이 저장되는 방식을 바꿔가며 여러
  번 다시 시도했지만 매번 비슷한 증상으로 실패.
- **확인해보니**: 실행 파일 안에 한글 문구가 들어 있으면, 그 파일을 저장하는 방식에 따라
  컴퓨터가 문장을 통째로 다른 명령으로 잘못 알아듣는 경우가 있었음. 저장 방식을 이리저리
  바꿔봐도 근본적으로 "한글이 들어있다"는 것 자체가 계속 말썽의 씨앗이었음. 또한 깃허브
  페이지에서 파일을 내려받아 저장하는 방식 자체도 실제로는 로컬 파일이 안 바뀌는 경우가
  반복돼서, 다시 받는 방식으로는 해결이 안 됨을 확인.
- **어떻게 처리했나**: 두 실행 파일 모두 한글 문구를 아예 빼고 영어로만 다시 작성. 또
  깃허브에서 파일을 다시 받는 방식 대신, 전체 내용을 그대로 복사해서 메모장에 붙여넣고
  저장하는 방식으로 바꿈.
- **결과**: 이후로는 실행 파일이 깨지지 않고 정상적으로 실행됨을 확인. 업데이트용 실행
  파일 이름도 한글에서 영어로 바꿈(더 안정적이라서).

### 로그인 완료를 눌러도 버튼이 계속 눌리지 않는 문제 (여러 차례 반복)
- **무슨 문제/요청이 있었나**: 트위터 로그인을 마치고 대시보드의 "로그인 완료했어요"
  버튼을 눌러도, 화면의 다른 버튼들이 계속 눌리지 않는 상태(마우스 커서가 금지 표시로
  바뀜)로 남는 문제. 한 번 고쳤다고 생각했는데도 다른 상황에서 똑같은 증상이 반복됨.
- **확인해보니**: 원인이 하나가 아니라 여러 개였음. (1) 로그인 완료 버튼 자체도 다른
  버튼들과 함께 잠기도록 걸려 있어서 정작 그 버튼을 못 누르는 경우 (2) 화면을 새로고침한
  뒤 다시 접속했을 때, 이미 끝난 작업인데도 화면이 그 사실을 못 알아채고 계속 잠긴 채로
  남는 경우 (3) 가장 근본적인 원인 — 로그인 세션을 저장한 뒤 로그인창(브라우저)을 닫는
  과정이 간혹 끝나지 않고 멈춰버려서, 저장은 이미 다 끝났는데도 뒷단 프로그램이 "아직
  진행 중"이라는 상태로 영원히 남아있던 것.
- **어떻게 처리했나**: (1)(2)는 화면 쪽 잠금/새로고침 로직을 고쳐서 해결. (3)은 로그인창을
  닫는 과정에 제한 시간을 두고, 그 시간이 지나면 저장이 끝났으니 그냥 다음 단계로 넘어가게
  해서 영원히 멈추는 일 자체를 없앰.
- **결과**: 실제로 다시 로그인해서 완료 버튼을 눌러본 결과 버튼이 정상적으로 풀리는 것
  확인. 이후로도 재발 신고가 없어서 근본 원인이 (3)이었던 것으로 판단.

### 경쟁사 게시물 하나가 오늘도 또 빠졌다 (헌터x헌터 반복 누락)
- **무슨 문제/요청이 있었나**: 실제로 존재하는 경쟁사의 평범한 게시물 하나가 이번에도
  수집 결과에서 계속 빠짐 — 예전에도 같은 상품에서 비슷한 문제가 있었다며 재차 확인 요청.
  처음엔 "그 게시물이 원래 안 올라온 것 아니냐"고 잘못 짐작했다가, 실제 게시물 화면을
  받아서 확인한 뒤 진짜 존재하는 정상 게시물임을 확인.
- **확인해보니**: 화면에 "재게시함" 같은 표시가 붙은 글은 원래 걸러내고, 그 대신 그 글이
  원래 있던 자리(자기가 처음 올린 자리)에서 나중에 자연스럽게 다시 잡히도록 설계돼 있었음.
  그런데 이 게시물은 그 계정이 자기 글을 계속 반복해서 재게시하고 있어서, "원래 있던 자리"
  자체가 항상 그 재게시 표시로 덮여버려 한 번도 안 걸리고 매번 통째로 누락되고 있었던 것.
- **어떻게 처리했나**: "재게시함" 표시가 붙어 있어도, 그게 그 계정이 자기 자신의 글을
  다시 올린 경우라면 걸러내지 않고 그대로 집계하도록 예외 처리 추가.
- **결과**: 사용자가 이 재게시 게시물도 원래 집계 대상이 맞다고 재확인해줌(이전에도 이미
  합의된 방침). 처리 후 이 계정의 게시물이 정상적으로 잡히는지는 다음 수집에서 확인 예정.

### 기간 지정 화면이 접혀 있어서 불편하고, 날짜 두 번 고르는 게 번거롭다
- **무슨 문제/요청이 있었나**: "특정 기간 지정" 영역이 평소엔 접혀 있어서 펼치는 손이
  하나 더 들어감. 그리고 시작일을 고르면 종료일도 보통 같은 날인 경우가 많은데 매번
  따로 또 골라야 하는 게 번거로움.
- **어떻게 처리했나**: 그 영역을 접었다 펴는 방식을 없애고 항상 펼쳐진 상태로 고정. 그리고
  시작일을 고르면 종료일 칸에도 같은 날짜가 자동으로 채워지도록 함(원하면 종료일만 따로
  다시 바꿀 수 있음).
- **결과**: 클릭 한 번과 날짜 선택 한 번이 줄어듦.

### 팀 배포판을 어디서 어떻게 나눠줄지 다시 설계
- **무슨 문제/요청이 있었나**: 배포판을 실제 릴리스로 올리려는데, 지금 쓰는 저장소를
  그대로 공개하면 내부 문서나 세세한 개발 기록까지 팀원이 다 보게 되는 게 부담스럽다는
  고민. 또 허브 페이지의 작은 카드 하나에 설치 안내를 다 욱여넣기엔 처음 보는 사람이
  이해하기 어렵다는 지적.
- **확인해보니**: 릴리스는 저장소 전체가 아니라 이 프로젝트 폴더만 압축해서 올리는 것이라
  다른 프로젝트에 영향이 없다는 것부터 확인. 다만 저장소 자체를 그대로 쓰면 내부 문서가
  같이 노출되는 건 맞음.
- **어떻게 처리했나**: 팀에게 나눠줄 몫만 담을 별도의 새 저장소를 따로 파서 그쪽으로
  릴리스를 올리기로 함. 그리고 허브 페이지의 카드는 다운로드 버튼만 남기고, 설치 방법·
  주의사항·문제 해결법은 전용 안내 페이지를 새로 만들어 그쪽으로 옮김.
- **결과**: 새 안내 페이지가 만들어졌고, 허브 페이지(개발용 브랜치와 실제 서비스용 양쪽
  다)에서 그 페이지로 연결되도록 반영함.

### 다운로드 안내 페이지 다듬기 — 버튼 배치, 개인정보·주의사항 안내
- **무슨 문제/요청이 있었나**: 다운로드 버튼과 "이미 설치했다면 여기로" 안내가 세로로
  나열돼 있어서 답답해 보인다며 나란히 두 칸으로 배치해달라는 요청. 그리고 "이미
  설치했다면" 링크에 밑줄이 왜 있는지 궁금해함. 또 잘 안 될 때 안내가 어떤 상황을 말하는
  건지 모호해서 아무 때나 따라 할까 걱정된다는 지적. 마지막으로 로그인 정보를 수집하지
  않는다는 안내가 빠져 있다는 점도 짚음.
- **확인해보니**: 밑줄은 그 링크 하나만 밑줄 없애는 처리를 깜빡해서 생긴 것이었음(의도한
  디자인이 아니라 실수). 잘 안 될 때 항목은 증상 설명 없이 대응만 나열돼 있어서 모호해
  보일 수 있었음.
- **어떻게 처리했나**: 다운로드 버튼과 "이미 설치했다면" 버튼을 나란히 두 칸으로 배치(왼쪽
  버튼 안에는 용량 안내도 작은 글씨로 같이 넣음). 잘 안 될 때 항목은 "이런 증상이면 → 이렇게
  하세요" 형태로 증상과 대응을 짝지어 나눠 다시 씀. 로그인 정보는 본인 컴퓨터에만 저장되고
  외부로 전송되지 않으며, 수집하는 건 공개된 게시물 실적일 뿐 개인정보가 아니라는 안내
  문구를 새로 추가.
- **결과**: 실제 화면 미리보기로 배치와 문구를 확인한 뒤 반영. 두 저장소(개발용/실서비스용)
  양쪽에 다 적용함.

### 압축 파일 대신 설치 파일 하나로 배포하는 방식으로 전환
- **무슨 문제/요청이 있었나**: 지금처럼 "압축 파일 받기 → 직접 풀기 → 실행 파일 찾아서
  누르기"가 단계가 많다며, 자동으로 설치되는 방식(예: 흔히 보는 설치 마법사)으로 바꿀 수
  없는지 문의. 그 뒤로 이어서, 나중에 업데이트할 때도 설치 파일을 다시 만들어야 하는지,
  프로그램을 컴퓨터의 어느 위치에 설치해야 하는지, 실행 아이콘이 따로 생기는 게 맞는지
  등 설치 방식 전반에 대해 여러 차례 질문.
- **확인해보니**: 프로그램을 하나로 합쳐 실행 파일 자체로 만드는 방법은 이 프로그램이 쓰는
  내장 브라우저 구조상 불가능함을 확인. 대신 지금 폴더 구조를 그대로 감싸서 "설치 마법사"
  형태로 만들어주는 방식이 있고, 이 방식이면 지금까지 만들어둔 것을 하나도 바꿀 필요가
  없음을 확인. 다만 설치 위치를 잘못 잡으면(관리자 전용 구역) 실행 중 저장이 막혀서
  "수집은 되는데 저장이 안 되는" 문제로 이어질 수 있다는 점도 미리 확인.
- **어떻게 처리했나**: 설치 마법사를 만드는 안내서(레시피 파일)를 새로 작성 — 설치 위치를
  개인 저장 공간으로 고정하고, 관리자 권한 없이 설치되도록 하고, 바탕화면과 시작 메뉴에
  "실행"과 "업데이트" 아이콘을 각각 만들도록 구성. 평소 코드 개선은 여전히 업데이트 아이콘
  하나로 끝나고, 설치 마법사를 다시 만들 일은 프로그램 구조 자체가 바뀌는 아주 드문 경우
  뿐이라는 점도 정리해서 안내.
- **결과**: 실제로 이 레시피로 설치 파일을 만들어본 결과, 다른 컴퓨터에서 실행했을 때
  설치 마법사가 정상적으로 뜨고, 완료 후 생긴 바탕화면 아이콘으로 대시보드가 실제로
  켜지는 것까지 확인. 처음 실행 시 뜨는 윈도우 방화벽 허용 팝업은 정상적인 것으로 확인.
  이후 다운로드 안내 페이지도 설치 마법사 방식에 맞게 안내문을 다시 씀(압축 풀기 안내
  삭제, 바탕화면 아이콘 안내로 교체) 및 다운로드 버튼을 실제 설치 파일 직접 연결로 교체.

### 배포용 안내 페이지가 오래도록 갱신되지 않는 문제
- **무슨 문제/요청이 있었나**: 안내 페이지를 고쳐서 반영했는데도 실제 화면에는 예전 내용이
  계속 보인다는 신고. 브라우저를 새로고침해도 그대로였음. 또 화면에 뜬 관리 목록에서 이전
  작업이 "취소됨" 표시로 나오는 것을 보고 이것도 오류인지 문의.
- **확인해보니**: 파일 자체는 정상적으로 바뀌어 있었음(직접 원본을 대조해서 확인). 안내
  페이지로 내보내는 절차가 계속 순서를 기다리는 상태로 멈춰 있었던 것. "취소됨" 표시는
  그 사이 여러 번 다시 고쳐서 올리다 보니, 이미 순서를 기다리던 오래된 내보내기 작업이
  더 최신 것으로 자동으로 대체되면서 생긴 정상적인 현상이었음(오류 아님). 최종적으로는
  깃허브 서비스 자체의 일부 기능(자동 내보내기)이 그 시점에 실제로 문제를 겪고 있었다는
  것을 깃허브의 공식 상태 안내를 통해 확인.
- **어떻게 처리했나**: 별도로 손댈 부분은 없었음 — 원인이 이쪽 설정이 아니라 깃허브 쪽
  서비스 문제였기 때문에, 서비스가 정상화되기를 기다리는 것으로 정리.
- **결과**: 서비스 쪽 문제로 확인됐고, 정상화되면 이미 반영해둔 내용이 자동으로 화면에
  나타날 것으로 예상.

### 대시보드 화면 없이 명령어로 직접 쓰는 게 나은지 문의
- **무슨 문제/요청이 있었나**: 팀원용으로 만든 화면(대시보드)을 거치지 않고, 본인이 쓸
  때는 그냥 명령어 창에서 바로 실행하는 게 더 나은지 문의.
- **확인해보니**: 화면의 각 버튼은 그 뒤에서 결국 명령어 하나씩을 그대로 실행하는 것에
  불과함을 확인 — 화면이 뭔가 추가로 처리해주는 게 아니라 그대로 전달만 하는 구조였음.
  로그인 완료 버튼도 마찬가지로, 명령어 창에서 로그인 마치고 그냥 엔터를 치는 것과 동일한
  효과.
- **어떻게 처리했나**: 각 화면 버튼에 대응하는 정확한 명령어를 정리해서 안내.
- **결과**: 실제 컴퓨터에 이미 필요한 프로그램이 설치돼 있는 것을 확인했고, 명령어로 바로
  실행하는 방식으로도 화면과 완전히 동일하게 동작함을 확인.

### 매번 날짜를 두 번 입력하는 게 번거롭다
- **무슨 문제/요청이 있었나**: 오늘 하루만 보고 싶을 때도 시작일·종료일을 매번 손으로
  입력해야 해서 번거로움 — "today"처럼 짧은 말로 대체할 수 있는지 물어봄.
- **확인해보니**: 오늘 날짜를 계산해서 시작·종료에 그대로 넣어주면 되는 간단한 방식으로
  가능. 다만 서버가 어느 나라 시간대로 돌아가든 상관없이 한국 시간 기준으로 "오늘"을
  계산해야 날짜가 어긋나지 않음.
- **어떻게 처리했나**: "today"라고만 입력하면 한국 시간 기준 오늘 날짜로 자동으로
  채워지게 함. 덧붙여, 예전부터 있던 설명 문구 중 "종료일은 포함 안 됨"이라고 적혀
  있던 게 실제 동작과 반대라는 것도 이번에 같이 발견해서 바로잡음 — 실제로는 종료일도
  포함되는 방식이었는데, 반대로 알고 있으면 하루치만 보려다가 실수로 이틀치를 수집했을
  수 있음.
- **결과**: 실제로 입력해서 오늘 날짜가 정확히 들어가는지 확인함.

### 미매칭 게시물 두 건, 직접 짝지어 달라는 요청
- **무슨 문제/요청이 있었나**: 리포트에 안 묶인 게시물 목록을 보고, 특정 번호끼리("이벤트는
  몇 번, 신제품 안내는 몇 번") 직접 짝지어 달라고 요청.
- **확인해보니**: 이미 만들어져 있던 "번호로 수동 짝짓기" 기능을 그대로 쓰면 됨.
- **어떻게 처리했나**: 알려준 번호에 맞는 게시물끼리 짝지어서 목록에 추가.
- **결과**: 반영 확인.

### 이 상품(신작 발표 게시물)은 왜 수집이 안 됐을까
- **무슨 문제/요청이 있었나**: 실제로 올라온 게 확실한 게시물인데 리포트에 전혀 안
  보인다는 문제 제기. 리트윗도 답글도 아니고 다른 정상 게시물이랑 형식이 똑같다고 함.
- **확인해보니**: 처음엔 "고정된 게시물이라 걸러졌을 것"이라고 짐작했으나 아니었고,
  다음엔 "답글이라 못 봤을 것"이라 짐작했으나 그것도 아니었음. 실제로 게시물 번호
  순서를 확인해보니, 이 게시물 바로 앞뒤에 있던 다른 게시물들은 정상적으로 수집됐다는
  걸 확인 — 수집 범위 밖으로 넘어간 게 아니라 이 게시물 하나만 화면에서 인식이 안 된
  것으로 추정. 이 게시물이 다른 정상 게시물과 다른 점은 상품 링크가 3개나 붙어있어서
  화면에서 유난히 길다는 것.
- **어떻게 처리했나**: (1) 화면을 아래로 넘기는 폭을 줄여서, 유난히 긴 게시물 하나를
  건너뛸 확률을 낮춤. (2) 그래도 확실한 원인을 못 찾아서, 게시물이 걸러질 때마다 그
  이유를 기록해두는 진단 기능을 추가 — 다음에 똑같은 문제가 또 생기면 왜 걸러졌는지
  바로 확인 가능하게.
- **결과**: 스크롤 폭을 줄인 뒤로는 이 게시물이 실제로 잡히기 시작함 — 다만 100% 원인
  확정은 아니고, 우연히 나아진 것일 수도 있어서 계속 지켜봐야 함.

### 실행하다가 실수로 옛날 기본 기간이 통째로 수집될 뻔함
- **무슨 문제/요청이 있었나**: 날짜를 안 넣고 그냥 실행했다가, 미리 정해둔 옛날 테스트용
  기본 기간으로 조용히 넘어가려는 걸 보고 직접 중단시킴. 날짜 안 넣으면 안내 문구부터
  뜨게 하는 게 낫겠다고 제안.
- **확인해보니**: 맞는 지적 — 날짜를 깜빡하고 실행했을 때 조용히 엉뚱한 기간이 수집되는
  사고로 이어질 수 있었음.
- **어떻게 처리했나**: 날짜(또는 "today")를 안 넣고 실행하면 무조건 에러 문구를 띄우고
  멈추도록 바꿈 — 옛날 기본 기간 자체를 없앰.
- **결과**: 날짜 없이 실행하면 안내 문구가 바로 뜨는 것 확인.

### 트위터가 갑자기 0건 수집됨
- **무슨 문제/요청이 있었나**: 위 사고 직후 재실행했더니 이번엔 트위터만 자사·경쟁사
  둘 다 0건으로 나옴. 로그인은 정상인데 왜 이런지 물어봄.
- **확인해보니**: 처음엔 "화면이 늦게 뜨는 문제"로 짐작했으나, 이후 재현 안 되고 다시
  정상적으로 수집됨 — 일회성 문제였던 것으로 보이나 완전히 확정은 못 함.
- **어떻게 처리했나**: 재현 안 되는 문제라 별도 수정 없이, 다시 실행해서 정상 수집되는지로
  확인.
- **결과**: 재실행 후 정상 수집 확인.

### 재고 추이 그래프가 며칠 전 시점에서 멈춰있음
- **무슨 문제/요청이 있었나**: 재고 스냅샷을 오늘도 찍고 며칠 전에도 찍었는데, 화면의
  추이 그래프는 며칠 전 시점에서 멈춰있음. 실적 수집할 때마다 재고 스냅샷도 자동으로
  같이 남겨줬으면 좋겠다고 요청.
- **확인해보니**: 재고 스냅샷을 새로 찍는 건 완전히 별도의 명령어였고, SNS 실적을
  수집하는 명령어는 이미 저장된 스냅샷을 읽기만 할 뿐 새로 찍지는 않고 있었음 — 그래서
  스냅샷 찍는 걸 깜빡하면 그래프가 그 시점에서 멈춰 보였던 것.
- **어떻게 처리했나**: SNS 실적 수집 명령어를 실행할 때마다 재고 스냅샷도 자동으로 하나
  같이 찍도록 연결. 재고 사이트 접속이 실패해도 SNS 리포트 저장 자체는 그대로 진행되게
  안전장치를 둠.
- **결과**: 실제 실행 로그로 스냅샷이 새로 찍히고 개수가 늘어나는 것 확인.

### 미매칭 항목을 채팅으로 일일이 알려주는 게 번거롭다 — 리포트에서 직접 처리하고 싶다
- **무슨 문제/요청이 있었나**: 안 묶인 게시물이 있을 때마다 매번 "몇 번 몇 번 짝지어줘"라고
  채팅으로 알려주고 리포트를 다시 받는 과정이 번거롭다며, 리포트 화면 안에서 직접 처리할
  수 있게 해달라고 요청. 서버 없이도 이런 반응형 화면을 만들 수 있지 않냐고 확인.
- **확인해보니**: 서버 없이 브라우저 안에서만 동작하는 방식으로 충분히 가능. 완전히
  새로운 게시물(스크래퍼가 아예 놓친 것)을 추가하려면 게시물 주소만으론 안 되고 내용을
  텍스트로 직접 가져와야 하는데, 게시물 상세 화면에서 전체 복사한 텍스트를 확인해보니
  그 안에 게시 시각과 좋아요·리트윗 숫자까지 정해진 순서로 다 들어있다는 걸 확인 —
  자동으로 읽어낼 수 있음.
- **어떻게 처리했나**: 리포트 화면에 이런 기능을 새로 만듦 — (1) 안 묶인 게시물마다
  "표에 추가" 버튼을 눌러 바로 표에 넣기 (2) 아예 새로운 게시물은 상세 화면에서 복사한
  텍스트를 붙여넣으면 본문·시각·좋아요·리트윗을 자동으로 읽어서 추가(자동 인식이 안
  되면 직접 입력하는 대체 방법도 마련). 추가한 내용은 그 리포트 화면 안에서만 보이는
  미리보기이고, 계속 남겨두려면 화면 하단에 나오는 저장 상자 내용을 복사해서 별도
  파일에 붙여넣으면 다음에 다시 만들 때도 계속 반영됨 — 이때 어느 상품과 같은 건지
  사람이 따로 지정할 필요 없이, 늘 쓰던 자동 인식이 다시 알아서 판단함.
- **결과**: 실제로 화면에서 눌러보면서 안 묶인 게시물 추가/새 게시물 붙여넣기 추가/잘못
  입력했을 때 안내 문구/직접 입력/저장 상자까지 전부 정상 동작하는 것 확인.

### 추가한 걸 실수로 잘못 넣었을 때 없앨 방법이 필요하다
- **무슨 문제/요청이 있었나**: 위 기능에 이어서, 표에 잘못 추가한 항목을 없애는 버튼도
  만들어달라고 요청.
- **확인해보니**: 간단히 추가 가능. 처음엔 버튼을 상품 이름 칸에 넣었다가, 실제로 눌러
  보니 칸이 좁아서 클릭이 막히는 걸 발견해서 자리를 옮김.
- **어떻게 처리했나**: 추가한 행마다 "삭제" 버튼을 붙여서(공간이 넉넉한 링크 칸 쪽으로),
  누르면 화면에서도 빠지고 저장 목록에서도 빠지게 함.
- **결과**: 실제로 눌러서 행이 없어지고 저장 상자도 다시 숨겨지는 것까지 확인.

### 예전엔 잘 잡히던 상품이 왜 이번엔 안 잡히지 + 매출 추정 숫자가 이상하다
- **무슨 문제/요청이 있었나**: 경쟁사 매출 추정 숫자가 이상하다는 의문 제기(나중에
  본인 착각이었다고 정정), 그리고 예전 리포트에선 분명 경쟁사와 비교되던 상품이 이번
  리포트에선 안 묶이고 미매칭으로 빠졌다며 원인을 봐달라고 요청.
- **확인해보니**: 매출 추정 숫자는 실제 오류가 아니라, 재고 데이터가 부족할 때 "현재
  재고를 가장 가까운 단위로 올려서 처음 한도였을 것"이라 가정하고 역산한 추정치였음
  (추정치라는 표시가 따로 붙어 있음). 처음엔 미매칭 문제도 화면 인식 문제로 짐작했으나,
  예전에 이 상품이 매칭됐던 리포트를 직접 다시 열어서 그 안에 실제로 어떤 게시물들이
  묶여 있었는지 대조해본 결과 — 전혀 다른 상품(다른 시리즈의 신작) 게시물이 함께 잘못
  묶여 있었던 걸 발견함. 자사가 여러 다른 상품 발표 게시물마다 똑같이 쓰는 문구
  ("초회특전 세트") 때문에, 서로 무관한 두 상품이 그 문구 하나로 잘못 엮였던 것.
- **어떻게 처리했나**: 그 문구를 상품 구분에서 제외하도록 목록에 추가해서 오묶음을 막음.
  겸사겸사 자사·경쟁사가 상품명을 살짝 다른 문자(영문 x와 곱셈 기호)로 써서 매칭이
  아예 안 되던 것도 같이 발견해서 통일되게 고침.
- **결과**: 두 상품 문구로 다시 확인해서 더 이상 안 겹치는 것 확인. 정리하면, "예전에
  실적이 좋아 보이던 숫자"는 사실 무관한 다른 상품 실적이 섞인 잘못된 숫자였고, "지금
  안 잡히는 것"은 데이터가 사라진 게 아니라 그 오묶음이 없어진 것 — 다만 진짜 경쟁사의
  해당 상품 게시물이 이번 수집에 실제로 있는지는 아직 별개로 확인이 필요함.

  다시 처음부터 수집하지 않고 그 결과로 엑셀 파일도 추가로 만들 수 없냐는 요청.
- **확인해보니**: 그 수집 당시엔 숫자(좋아요/리트윗) 포함 원본 데이터를 어디에도 남겨두지
  않고 글 내용 파일만 저장하는 구조였어서, 그 한 번의 수집 결과는 다시 수집하지 않는 이상
  복구가 불가능한 상태였음 — 이 부분은 솔직하게 그대로 안내함.
- **어떻게 처리했나**: 앞으로는 수집할 때마다 방식과 상관없이 항상 원본 수집 결과(좋아요·
  리트윗 숫자 포함)를 별도로 저장해두도록 바꾸고, 그 저장해둔 원본을 다시 읽어서 엑셀이나
  글 파일을 재수집 없이 다시 만들어주는 별도 명령어도 새로 만듦. 이번 요청 건 자체는 어쩔
  수 없이 한 번 더 수집해서 처리함.
- **결과**: 이제부터는 어떤 방식으로 수집하든 나중에 "이번엔 다른 형태로도 보고 싶다"고
  하면 다시 수집할 필요 없이 저장해둔 원본으로 바로 다시 만들어낼 수 있음.

### 엑셀에서 줄바꿈이 안 보이던 문제
- **무슨 문제/요청이 있었나**: 위 계정 단독 리포트의 엑셀 파일을 열어보니, 글 내용이 여러
  줄로 써 있던 게 텍스트 파일에서는 줄바꿈이 되어 보이는데 엑셀에서는 한 줄로 붙어서 보임 —
  엑셀 칸 안에서도 줄바꿈이 그대로 보이게 해달라는 요청.
- **확인해보니**: 엑셀 칸에 글을 넣을 때 줄바꿈 표시를 공백으로 바꿔서 넣고 있었고, 설령
  줄바꿈을 그대로 둬도 그 칸에 "줄바꿈 반영해서 보여주기" 설정을 켜두지 않으면 엑셀에서
  한 줄로 뭉쳐 보이는 것도 추가 원인이었음.
- **어떻게 처리했나**: 줄바꿈을 공백으로 바꾸지 않고 그대로 넣도록 하고, 그 칸에 "줄바꿈
  반영" 설정을 켬. 글이 여러 줄이면 칸 높이도 줄 수에 맞게 넉넉히 늘려서 잘려 보이지 않게 함.
- **결과**: 확인용 데이터로 다시 만들어 열어본 결과 엑셀 칸 안에서도 줄바꿈이 그대로 보이는
  것까지 확인함.

### 인스타그램 게시물 카드가 안 뜨던 문제 + 펼침 영역이 너무 넓던 문제
- **무슨 문제/요청이 있었나**: 리포트 안에서 트위터 게시물은 실제 카드 형태로 잘 보이는데,
  인스타그램 게시물은 인터넷이 연결돼 있어도 카드로 안 바뀌고 링크만 보임. 그리고 게시물
  목록을 펼쳤을 때 나오는 영역이 화면 폭을 다 차지해서 보기 불편함.
- **확인해보니**: 인스타그램 카드로 바꿔주는 외부 스크립트가 트위터 것보다 늦게 준비되는
  경우가 있는데, 목록을 펼친 시점에 그 스크립트가 아직 준비 안 됐으면 "이미 처리 끝났다"고
  잘못 표시해버려서 나중에 다시 펼쳐도 다시 시도를 안 하고 있었음. 그래서 인터넷 연결
  자체는 되는데도 카드로 안 바뀌는 것처럼 보였던 것.
- **어떻게 처리했나**: 목록을 펼칠 때마다 그 외부 스크립트가 준비됐는지 확인해서, 아직이면
  준비될 때까지 잠깐 기다렸다가 다시 시도하도록 고침. 그리고 펼쳤을 때 나오는 자사/경쟁사
  두 영역의 좌우 비율은 그대로 두고, 두 영역을 합친 전체 폭만 원래의 5분의 3 크기로 줄임.
- **결과**: 코드 상으로는 수정 완료. 실제 인터넷 환경에서 인스타그램 카드가 정상적으로
  뜨는지는 사용자 쪽에서 직접 다시 열어서 확인 필요.
