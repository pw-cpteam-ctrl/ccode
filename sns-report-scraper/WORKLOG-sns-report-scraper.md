# WORKLOG-sns-report-scraper.md

`WORKLOG-작성-규칙.md` 기준으로 작성 — 코드 용어 없이, 그날 있었던 요청과 처리 흐름을
있는 그대로 기록.

## 2026-07-13

### 예전 오류/시행착오 기록 보강
- **무슨 문제/요청이 있었나**: 오류가 생겼을 때 원인·해결 과정만 적혀 있고 "다음에 또
  이런 일이 안 생기게 하려면 뭘 조심해야 하는지"가 빠져 있다는 지적. 또 지금 기록된 것보다
  더 이전에도 실제로 겪은 문제(로그인 세션 관련 포함)가 있었을 텐데 그것도 챙겨서 남겨야
  하지 않냐는 요청.
- **확인해보니**: 기존 기록에는 "해결" 항목만 있고 "재발 방지"용 요약이 없었음. 또 지금
  기록에 없는 훨씬 이전 시점(7/4~7/9)의 실제 문제들 — 예를 들어 상품이 잘못 묶이던 문제,
  스크롤이 끝까지 안 되던 문제, 시간 표시가 잘못 나오던 문제, 재고 조회가 막히던 문제,
  이전에 실수로 파일을 지웠던 일, 로그인 세션이 다시 풀려서 재로그인해야 했던 일 등 —
  이 하나도 안 남아 있었음.
- **어떻게 처리했나**: 기존 기록 각 항목마다 "앞으로 이런 걸 조심하면 된다"는 요약을
  새로 추가함. 그리고 7/4~7/9 사이에 실제로 있었던 문제 11건을 새로 찾아서 같은 형식
  (무슨 문제 → 원인 → 해결 → 재발 방지)으로 정리해 추가함.
- **결과**: 기록 문서가 훨씬 두꺼워지고, 과거에 겪은 문제를 나중에 다시 찾아볼 때
  "그때 왜 그랬고 뭘 조심해야 하는지"까지 한 번에 볼 수 있게 됨.

### 수집 기간을 명령어에서 바로 지정하고 싶다는 요청
- **무슨 문제/요청이 있었나**: 7/10 하루치 실적만 보고 싶은데, 그러려면 파일을 직접 열어서
  날짜를 손으로 고쳐야 해서 번거롭다는 요청 — 명령어 실행할 때 날짜만 바로 넣으면
  그 기간만 수집되게 해달라는 것.
- **확인해보니**: 지금까지는 수집 기간이 파일 안에 고정값으로 박혀 있어서, 다른 기간을
  보려면 매번 그 파일을 열어 값을 수정하고 저장한 다음 실행해야 했음.
- **어떻게 처리했나**: 실행할 때 시작일과 종료일을 순서대로 적어주면 그 기간으로 바로
  돌아가도록 고침 (예: 7월 10일부터 11일까지 보고 싶으면 그 두 날짜를 명령어 뒤에 붙여서
  실행). 안 적으면 원래 설정된 기본 기간 그대로 동작.
- **결과**: 파일을 열어 고칠 필요 없이 명령어 한 줄로 원하는 기간을 바로 지정해서 실행할
  수 있게 됨.

### 엑셀 시트가 실행할 때마다 계속 쌓이던 문제
- **무슨 문제/요청이 있었나**: 테스트 삼아 여러 번 실행하다 보니 엑셀 파일 안에 시트가
  20~30개까지 계속 늘어나 있었음 — 같은 기간을 다시 돌려도 새 시트로 계속 추가되고 있었음.
- **확인해보니**: 실행할 때마다 "시트 이름이 이미 있으면 뒤에 번호를 붙여서 새로 만든다"는
  방식으로 짜여 있어서, 같은 기간으로 다시 실행해도 매번 새 시트가 하나씩 늘어나는 구조였음.
- **어떻게 처리했나**: 같은 수집 기간으로 다시 실행하면 그 기간의 기존 시트 하나만 최신
  내용으로 덮어쓰도록(교체) 바꾸고, 기간이 다르면 예전처럼 새 시트로 따로 남도록 함 —
  같은 기간을 몇 번을 다시 돌려도 시트가 하나만 유지되고, 다른 기간의 과거 기록은 그대로
  보존됨. 더불어 엑셀 파일이 다른 프로그램(엑셀 자체 등)에서 열려 있는 상태에서 저장을
  시도하면 저장이 실패할 수 있는데, 이 경우 잠깐 있다가 자동으로 몇 번 다시 시도하고, 그래도
  안 되면 "파일을 닫고 다시 실행해달라"는 안내 문구가 뜨도록 함(수집한 내용 자체는 사라지지
  않고 남아 있게 처리).
- **결과**: 같은 기간으로 반복 테스트해도 시트가 무한히 늘어나지 않고, 다른 기간 기록은
  그대로 남음. 실제로 같은 기간/다른 기간 두 경우를 각각 반복 실행해서 시트 개수가 의도한
  대로 동작하는지 확인함.

### 인스타그램 글 내용이 이상한 글자로 반복되던 문제 ("신지신지신지…")
- **무슨 문제/요청이 있었나**: 수집한 리포트를 열어보니 매칭 안 된 게시물 목록에서 원래
  글 내용 대신 특정 이름이 계속 반복되는 이상한 글자가 들어가 있었음.
- **확인해보니**: 인스타그램 게시물에서 원문을 뽑아올 때 "화면에 보이는 글자 조각 중 가장
  긴 것"을 무조건 원문이라고 판단하고 있었는데, 그 게시물 밑에 달린 팬 댓글(캐릭터 이름을
  반복해서 부르는 댓글)이 실제 원문보다 길이가 더 길어서 그게 원문으로 잘못 뽑힌 것이었음.
- **어떻게 처리했나**: 화면에 보이는 글자 조각들을 길이가 긴 순서대로 하나씩 확인하면서,
  "이 계정이 쓴 원문처럼 보이는 형태인지" 검사해서 맞는 걸 찾으면 그걸 쓰고, 끝까지 하나도
  안 맞으면 그때만 예전처럼 가장 긴 조각을 씀.
- **결과**: 실제 게시물 주소를 직접 열어서 원문과 비교해봤고, 이상한 반복 글자 없이 정상
  본문이 들어오는 걸 확인함.

### 서로 다른 상품 게시물들이 한 줄로 뭉쳐버리던 문제
- **무슨 문제/요청이 있었나**: 자사와 경쟁사 게시물 10건(각 5건씩, 트위터/인스타그램 둘 다)이
  원래는 서로 다른 상품인데 리포트에서 전부 하나의 상품 줄로 뭉쳐서 나왔음.
- **확인해보니**: 상품을 같은 것끼리 묶을 때 "게시물 문구에서 겹치는 단어가 2개 이상이면
  같은 상품으로 본다"는 방식을 쓰는데, 경쟁사가 신제품을 발표할 때마다 거의 매번 똑같이
  쓰는 정형화된 문구("원형", "최초", "추가", "추후", "색채", "조형" 등)가 있었음. 이 단어들이
  걸러지지 않고 그대로 비교 대상에 들어가다 보니, 서로 다른 캐릭터를 발표하는 게시물끼리도
  이 상용구 하나 때문에 "겹치는 단어 있음"으로 판정되어 줄줄이 하나로 엮여버린 것.
- **어떻게 처리했나**: 상품 구분에 의미 없는 상용구 목록에 이 단어들을 추가해서, 겹치는
  단어를 셀 때 더 이상 포함되지 않도록 함.
- **결과**: 사용자가 올려준 실제 수집 데이터로 직접 돌려서 확인한 결과, 트위터/인스타그램
  양쪽 모두 5개 상품이 각각 따로따로 구분되어 나왔고(예: 페르소나, 블리치, 초 가구야 공주,
  리본, 붕괴 스타레일), 매칭 안 된 게시물도 0건으로 확인됨.

### 인스타그램 좋아요 수가 계속 빈 값으로 나오던 문제 — 시도했다가 되돌린 건
- **무슨 문제/요청이 있었나**: 인스타그램 게시물의 좋아요/댓글 수가 계속 빈 값으로 나오는
  문제가 있었음.
- **확인해보니(1차 시도, 결과적으로 틀림)**: 몇 건은 성공하고 몇 건은 실패하는 걸 보고
  "가끔 뜨는 화면(로그인 유도 창 등)이 잠깐 가리는 것"이라고 짐작해서, 실패하면 화면을
  닫고 다시 시도하는 방식으로 고쳤음.
- **실제로 다시 써보니**: 오히려 이번엔 두 계정 게시물 전부(100%) 좋아요를 못 가져왔고,
  세 번씩 다시 시도하느라 실행 시간만 세 배로 늘어 아무 이득이 없었음. 처음 짐작이 틀렸던
  것으로 확인됨.
- **어떻게 처리했나**: 이 시도를 그대로 되돌려서 원래대로(한 번만 시도) 복구함. 좋아요 수가
  안 보이는 게 화면 인식 문제인지, 아니면 그 계정 자체가 좋아요 수를 숨기는 설정으로 되어
  있는 건지부터 먼저 확인이 필요한 상태.
- **결과**: 실행 시간을 낭비하던 시도는 원상 복구했고, 진짜 원인 확인(로그인한 상태로 직접
  열어봤을 때 좋아요 수가 보이는지)은 아직 사용자 쪽 확인 대기 중.

### 수집할 때마다 로그인 창(시크릿창처럼 인식)이 뜨는 문제
- **무슨 문제/요청이 있었나**: 자동 수집 프로그램이 마치 시크릿창(사생활 보호 모드)에서
  접속하는 것처럼 인식돼서 인스타그램이 자꾸 로그인 창을 띄운다는 문제 — 이전에도 여러 번
  얘기했었는데 아직 안 고쳐졌다는 지적.
- **확인해보니**: 예전에는 "자동화 프로그램임을 알려주는 신호" 중 딱 하나만 숨겨뒀었고,
  그 외에 자동화 프로그램인지 구분하는 데 쓰이는 다른 여러 특징(설치된 확장 기능 목록,
  화면 크기, 그래픽 카드 정보 등)은 그대로 노출돼 있었음.
- **어떻게 처리했나**: 그 외 특징들도 실제 사람이 쓰는 브라우저처럼 보이게 값을 채워 넣는
  작업을 추가로 함.
- **결과**: 아직 검증되지 않음 — 이 조치가 로그인 창 문제를 실제로 없애준다는 보장은 없고,
  기존에 이미 한 번 의심받은 적 있는 예전 로그인 정보를 계속 쓰면 이 조치를 해도 소용없을
  수 있어서, 로그인을 새로 한 번 다시 해서 테스트해봐야 하는 상태.

### 특정 계정 하나만 골라서 성과를 보고 싶다는 요청 (신규 기능)
- **무슨 문제/요청이 있었나**: 지금까지 만든 수집 기능을 바탕으로, 자사/경쟁사 비교가 아니라
  임의의 트위터 계정 하나(예시로 준 계정)를 지정하면 그 계정 게시물만 따로 모아서 성과를
  보고 싶다는 요청.
- **확인해보니**: 기존 기능은 항상 자사 vs 경쟁사 비교 형태로만 결과를 만들도록 되어 있어서,
  계정 하나만 단독으로 보는 용도로는 못 씀. 어떤 용도로 쓰고 싶은지 먼저 확인한 결과 "그
  계정 하나만의 성과를 단독으로 보고 싶다"는 것으로 확인됨(비교 목적 아님).
- **어떻게 처리했나**: 계정 이름 하나를 입력하면 그 계정 게시물만 모아서, 순위·합계·평균을
  보여주는 별도의 리포트를 만드는 기능을 새로 만듦. 로그인은 기존에 쓰던 내 계정 로그인
  상태를 그대로 재사용(새로 로그인할 필요 없음).
- **결과**: 예시로 준 계정으로 실제 실행해서 리포트가 정상적으로 만들어지는 것까지 확인함.

### 지표 없이 글 내용만 뽑고 싶다는 추가 요청
- **무슨 문제/요청이 있었나**: 위 계정 단독 리포트 기능에, 좋아요·리트윗 같은 숫자는 필요
  없고 그냥 "뭐라고 썼는지" 글 내용만 뽑고 싶을 때를 위한 별도 명령어를 추가해달라는 요청.
- **확인해보니**: 기존 방식은 항상 숫자 포함 순위표 형태로만 결과를 만들어서, 글 내용만
  보고 싶은 경우엔 필요 없는 정보까지 다 딸려 나옴.
- **어떻게 처리했나**: 명령어 끝에 별도 표시를 붙이면, 숫자 없이 글 내용 전체를 오래된
  순서대로 나열만 하는 파일이 따로 만들어지도록 함.
- **결과**: 요청한 형태로 텍스트 파일이 만들어지는 것까지 확인.

### 이미 수집한 결과로 엑셀도 같이 만들고 싶다는 요청
- **무슨 문제/요청이 있었나**: 방금 글 내용만 뽑는 방식으로 이미 한 번 수집을 끝냈는데,
  다시 처음부터 수집하지 않고 그 결과로 엑셀 파일도 추가로 만들 수 없냐는 요청.
- **확인해보니**: 그 수집 당시엔 숫자(좋아요/리트윗) 포함 원본 데이터를 어디에도 남겨두지
  않고 글 내용 파일만 저장하는 구조였어서, 그 한 번의 수집 결과는 다시 수집하지 않는 이상
  복구가 불가능한 상태였음 — 이 부분은 솔직하게 그대로 안내함.
- **어떻게 처리했나**: 앞으로는 수집할 때마다 방식과 상관없이 항상 원본 수집 결과(좋아요·
  리트윗 숫자 포함)를 별도로 저장해두도록 바꾸고, 그 저장해둔 원본을 다시 읽어서 엑셀이나
  글 파일을 재수집 없이 다시 만들어주는 별도 명령어도 새로 만듦. 이번 요청 건 자체는 어쩔
  수 없이 한 번 더 수집해서 처리함.

  수 있게 해달라고 요청. 서버 없이도 이런 반응형 화면을 만들 수 있지 않냐고 확인.
- **확인해보니**: 서버 없이 브라우저 안에서만 동작하는 방식으로 충분히 가능. 완전히
  새로운 게시물(스크래퍼가 아예 놓친 것)을 추가하려면 게시물 주소만으론 안 되고 내용을
  텍스트로 직접 가져와야 하는데, 게시물 상세 화면에서 전체 복사한 텍스트를 확인해보니
  그 안에 게시 시각과 좋아요·리트윗 숫자까지 정해진 순서로 다 들어있다는 걸 확인 —
  자동으로 읽어낼 수 있음.
- **어떻게 처리했나**: 리포트 화면에 이런 기능을 새로 만듦 — (1) 안 묶인 게시물마다
  "표에 추가" 버튼을 눌러 바로 표에 넣기 (2) 아예 새로운 게시물은 상세 화면에서 복사한
  텍스트를 붙여넣으면 본문·시각·좋아요·리트윗을 자동으로 읽어서 추가(자동 인식이 안
  되면 직접 입력하는 대체 방법도 마련). 추가한 내용은 그 리포트 화면 안에서만 보이는
  미리보기이고, 계속 남겨두려면 화면 하단에 나오는 저장 상자 내용을 복사해서 별도
  파일에 붙여넣으면 다음에 다시 만들 때도 계속 반영됨 — 이때 어느 상품과 같은 건지
  사람이 따로 지정할 필요 없이, 늘 쓰던 자동 인식이 다시 알아서 판단함.
- **결과**: 실제로 화면에서 눌러보면서 안 묶인 게시물 추가/새 게시물 붙여넣기 추가/잘못
  입력했을 때 안내 문구/직접 입력/저장 상자까지 전부 정상 동작하는 것 확인.

### 추가한 걸 실수로 잘못 넣었을 때 없앨 방법이 필요하다
- **무슨 문제/요청이 있었나**: 위 기능에 이어서, 표에 잘못 추가한 항목을 없애는 버튼도
  만들어달라고 요청.
- **확인해보니**: 간단히 추가 가능. 처음엔 버튼을 상품 이름 칸에 넣었다가, 실제로 눌러
  보니 칸이 좁아서 클릭이 막히는 걸 발견해서 자리를 옮김.
- **어떻게 처리했나**: 추가한 행마다 "삭제" 버튼을 붙여서(공간이 넉넉한 링크 칸 쪽으로),
  누르면 화면에서도 빠지고 저장 목록에서도 빠지게 함.
- **결과**: 실제로 눌러서 행이 없어지고 저장 상자도 다시 숨겨지는 것까지 확인.

### 예전엔 잘 잡히던 상품이 왜 이번엔 안 잡히지 + 매출 추정 숫자가 이상하다
- **무슨 문제/요청이 있었나**: 경쟁사 매출 추정 숫자가 이상하다는 의문 제기(나중에
  본인 착각이었다고 정정), 그리고 예전 리포트에선 분명 경쟁사와 비교되던 상품이 이번
  리포트에선 안 묶이고 미매칭으로 빠졌다며 원인을 봐달라고 요청.
- **확인해보니**: 매출 추정 숫자는 실제 오류가 아니라, 재고 데이터가 부족할 때 "현재
  재고를 가장 가까운 단위로 올려서 처음 한도였을 것"이라 가정하고 역산한 추정치였음
  (추정치라는 표시가 따로 붙어 있음). 처음엔 미매칭 문제도 화면 인식 문제로 짐작했으나,
  예전에 이 상품이 매칭됐던 리포트를 직접 다시 열어서 그 안에 실제로 어떤 게시물들이
  묶여 있었는지 대조해본 결과 — 전혀 다른 상품(다른 시리즈의 신작) 게시물이 함께 잘못
  묶여 있었던 걸 발견함. 자사가 여러 다른 상품 발표 게시물마다 똑같이 쓰는 문구
  ("초회특전 세트") 때문에, 서로 무관한 두 상품이 그 문구 하나로 잘못 엮였던 것.
- **어떻게 처리했나**: 그 문구를 상품 구분에서 제외하도록 목록에 추가해서 오묶음을 막음.
  겸사겸사 자사·경쟁사가 상품명을 살짝 다른 문자(영문 x와 곱셈 기호)로 써서 매칭이
  아예 안 되던 것도 같이 발견해서 통일되게 고침.
- **결과**: 두 상품 문구로 다시 확인해서 더 이상 안 겹치는 것 확인. 정리하면, "예전에
  실적이 좋아 보이던 숫자"는 사실 무관한 다른 상품 실적이 섞인 잘못된 숫자였고, "지금
  안 잡히는 것"은 데이터가 사라진 게 아니라 그 오묶음이 없어진 것 — 다만 진짜 경쟁사의
  해당 상품 게시물이 이번 수집에 실제로 있는지는 아직 별개로 확인이 필요함.
