# 새 프로젝트 시작 가이드

> 이 문서는 **사람(운영자)** 과 **AI(Claude Code)** 둘 다를 위한 문서다.
> 이 저장소(모노레포)에서 새 폴더로 새 도구를 만들 때, "코드는 완성됐는데
> 아무도 못 쓰는" 상태를 피하기 위한 체크리스트다. 특정 프로젝트 이야기가
> 아니라 이 레포 전체에 적용되는 구조적 원칙만 담는다.

## 이 저장소의 기본 구조 (전제)

- **모노레포(monorepo)**: 폴더 하나 = 프로젝트 하나. 여러 독립 도구가 한 저장소
  안에 같이 산다.
- **배포는 `main` 브랜치 기준**: 정적 페이지는 GitHub Pages, 서버 코드가 있는
  프로젝트는 Vercel 등 별도 플랫폼 — 어느 쪽이든 실제 배포에 반영되는 소스는
  `main`이다. feature 브랜치 없이 `main`에 직접 push하는 운영 방식이며,
  여러 세션(사람 혹은 AI)이 동시에 같은 `main`을 건드릴 수 있다.
- **루트 `index.html`은 허브(hub) 페이지**: 각 도구로 가는 링크 카드 모음.

이 전제를 모르고 작업하면 아래 체크리스트의 각 항목이 왜 필요한지 이해가
안 되므로, 새 프로젝트를 만들기 전 반드시 먼저 숙지할 것.

---

## 체크리스트 (이유 포함)

### 1. 결과물은 반드시 `main` 브랜치에 merge/push되어야 한다
로컬 브랜치나 별도 feature 브랜치에만 커밋이 있으면 배포 파이프라인
(GitHub Pages / Vercel 등)이 그 내용을 읽지 못한다. "로컬/브랜치에서는
동작하는데 실제 URL에는 반영이 안 된다"는 증상의 대부분은 이 문제다.
**AI 체크포인트**: 작업을 "완료"로 보고하기 전에 `git status`, `git log`로
커밋이 실제로 `main`에 올라갔는지(로컬 커밋만 있는 게 아닌지) 확인할 것.

### 2. 루트 `index.html`(허브)에 새 도구 카드를 반드시 추가한다
`/새폴더/index.html`을 아무리 잘 배포해도, 허브 페이지에 링크(`<a class="card"
href="새폴더/index.html">` 형태)를 추가하지 않으면 URL을 직접 아는 사람만
접근 가능하다. 즉 **배포 성공 ≠ 발견 가능(discoverable)**. 새 프로젝트를
만들면 반드시 이 카드 추가를 작업 목록에 포함시킬 것.

### 3. 폴더 단위로 완전히 독립시킨다 (경계/의존성 관리)
한 폴더의 코드가 다른 폴더의 파일을 import하거나 참조하지 않는다.
모노레포에서 프로젝트 간 결합(coupling)이 생기면, 한쪽을 옮기거나 삭제할 때
다른 쪽이 이유 없이 깨진다. 다른 프로젝트의 코드/디자인을 "참고"하는 것과
"의존"하는 것은 다르다 — 참고는 읽고 패턴만 가져오는 것, 의존은 그 파일을
직접 불러오거나 수정하는 것. 특정 프로젝트를 편집 범위에서 제외하라는
지시를 받으면, 그 프로젝트의 어떤 파일도 Read 이상의 목적(Edit/Write)으로
건드리지 않는다.

### 4. 같은 이름이 겹치는 파일은 프로젝트명을 접미사/접두사로 붙인다
`CHANGELOG.md`, `README.md`, `PLAN.md`처럼 흔한 파일명을 프로젝트 폴더
밖(루트)에 만들거나, 나중에 여러 프로젝트 문서를 한곳에서 검색할 상황이
생기면 이름 충돌로 헷갈린다. 예: `CHANGELOG-{프로젝트명}.md`. 이렇게 하면
grep/검색만으로도 어느 프로젝트 문서인지 즉시 구분된다.

### 5. `main`에 push하기 직전엔 항상 최신 상태를 먼저 받는다
여러 세션이 동시에 같은 `main`을 push할 수 있는 환경이므로, 내가 로컬에서
작업하는 사이 다른 커밋이 먼저 올라가 있을 수 있다. 그대로 push하면
non-fast-forward로 거절당한다.
**표준 순서**: `git fetch origin main` → `git rebase origin/main` →
`git push origin main`. 거절당했다고 강제 push(`--force`)로 덮어쓰지 말 것 —
다른 세션의 작업을 날릴 수 있다.

### 6. 배포 방식이 프로젝트 성격에 따라 다르다는 것을 먼저 정한다
- 빌드 과정 없는 정적 HTML/CSS/JS 도구 → GitHub Pages (push 후 자동 반영,
  단 캐시/전파 지연 있음).
- 서버 코드(API, 환경변수, LLM 호출 등)가 필요한 프로젝트 → Vercel 같은
  별도 플랫폼, 별도 배포 트리거와 환경변수 설정 필요.
새 프로젝트를 시작하기 전에 "정적 파일만으로 되는가, 서버 런타임이
필요한가"부터 결정해야 나중에 배포 구조를 갈아엎지 않는다. 모노레포에서
서버 배포 플랫폼을 쓸 경우, 다른 폴더의 변경사항 때문에 불필요하게
재배포되지 않도록 배포 스코프(예: ignore command)를 프로젝트별로 제한하는
것도 고려한다.

### 7. 배포 직후 "안 보인다" = 버그가 아니라 캐시/전파 지연일 수 있다
정적 호스팅은 push 즉시 반영이 아니라 수 분의 지연 + HTTP 캐시
(`cache-control` 헤더)가 걸릴 수 있다. 배포 헤더(`last-modified` 등)를 확인해
실제로 새 커밋이 반영됐는지부터 판단하고, 하드 리프레시 후 몇 분 기다려도
안 바뀔 때만 진짜 버그로 취급한다. 순서를 반대로 하면(바로 "버그다"라고
단정하면) 이미 잘 배포된 코드를 불필요하게 고치는 삽질을 하게 된다.

### 8. 클라이언트에 상태를 저장하는 도구는 스키마 변경 시 반드시 하위호환(fallback)을 챙긴다
브라우저 저장소(localStorage/IndexedDB 등)에 사용자 작업 상태를 저장하는
도구에 새 필드를 추가할 때, 예전에 저장된 데이터에는 그 필드가 없다.
새 필드를 무조건 있다고 가정하고 읽으면(`obj.field.something`) 예전 데이터를
불러오는 순간 런타임 에러가 나거나 화면이 깨진다.
**패턴**: `값 = 저장된값 ?? 기본값` (nullish coalescing) 형태로 항상 기본값을
깔아두고, 기존 필드는 지우지 말고 새 필드를 나란히 추가하는 방식(additive
schema change)을 기본으로 한다. 마이그레이션 스크립트 없이도 예전 데이터가
안전하게 로드되는지가 기준이다.

### 9. 새 기능은 "완료"라고 말하기 전에 실제 환경에서 동작을 검증한다
코드 리뷰/타입체크만으로 "됐다"고 하지 말고, 가능하면 브라우저 자동화
(예: Playwright)로 직접 조작해보고, 배포가 관련된 경우 라이브 URL까지
확인한다. 자동화가 불가능한 부분(네이티브 파일 선택창, 실제 로그인 세션
등)은 그 한계를 명확히 밝히고 가능한 범위(내부 로직 등)만이라도 모킹해서
검증한다.

> 코드는 완성됐는데 아무도 못 쓰는 상태를 피할 수 있다.**
