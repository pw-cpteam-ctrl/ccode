# 2026-07-16 (V2.0 다운로드 이미지 글자 잘림 근본 해결)

### 증상
- **미리보기 화면에선 멀쩡한데, 다운로드한 PNG에서만 글자가 잘림**
  (기술: 제품명 "페르소나3 리로드"→"페르소나3 리로", "명탐정 코난"→"명탐정", 오픈헤더 스토어명 "메가하우스 공식 스토어"→"메가하우스 공식 스토" 등 끝 글자 잘림 + 뱃지와 겹침)

### 진짜 원인 (한 줄 요약)
- **다운로드 엔진(html-to-image)이 텍스트를 화면보다 미세하게 넓게 그리는데, 글자 상자가 `overflow:hidden`으로 내용에 딱 맞게 잘려 있어 끝 글자가 잘려나감**
  (기술: 화면은 웹폰트 프리텐다드로 렌더 → 폭이 좁음 / 다운로드는 CORS로 폰트 임베드 실패 시 폴백폰트로 렌더 → 폭이 넓음. 상자는 화면 기준 폭으로 고정 → 다운로드에서 초과분이 잘림)

### ⚠️ 왜 6차까지 갔나 — 타 프로젝트 재발 방지용 상세 기록
> 이 버그는 "고쳤다"고 6번 보고했다가 6번 다시 잘렸다. **테스트 환경과 실제 사용자 환경이 달라서** 매번 헛다리를 짚은 것이 핵심 실패다. 아래를 순서대로 읽으면 같은 삽질을 피할 수 있다.

- **1차 — 폰트 로딩 타이밍 의심 (실패)**
  - 캡처 직전 `document.fonts.ready`를 기다리게 함. 그럴듯했지만 증상 그대로였음
  - (교훈: 테스트 환경에 웹폰트가 아예 안 깔려 있어서 폴백폰트로 통과된 것을 "해결됐다"고 오판. **폰트 없는 환경의 테스트 통과는 증거가 아니다**)

- **2차 — 오픈헤더 타이틀만 부분 해결 (부분 성공)**
  - 헤더 타이틀/날짜를 한 줄 고정(nowrap) + 넘치면 글자 크기 자동 축소. 헤더는 고쳐졌으나 그리드 제품명은 여전
  - (교훈: 증상이 여러 곳에서 나면 한 곳만 고치고 "됐다" 하지 말 것)

- **3차 — 제품명 자동 축소 추가했으나 측정이 거짓말 (실패)**
  - `scrollWidth > 여유폭`이면 글자 축소하도록 함. 코드상 측정은 "안 잘림"으로 나왔지만 실제로는 여전히 잘림
  - (교훈: 제품명이 `white-space:normal`이라 넘칠 때 **가로로 넘치는 대신 아래로 줄바꿈** → `scrollWidth`가 작게 나와 "안 넘친다"고 거짓 판정. **측정값(clipped:false)을 믿지 말고 실제 렌더 결과 이미지를 눈으로 봐라**)

- **4차 — 제품명 nowrap 전환 + 폰트 로드 후 재조정 (실패)**
  - 한 줄 고정으로 넘침이 측정되게 하고, 폰트 로드 완료 후 크기 재계산. 그래도 다운로드에서 잘림
  - (교훈: 여기서 "미리보기 ≠ 다운로드"라는 사용자의 반복 지적이 결정적 단서였음. 화면 기준으로 아무리 맞춰도 다운로드 렌더가 다르면 소용없다)

- **5차 — 실제 폰트를 테스트 환경에 심어서 재현 성공 (원인 확정)**
  - 사용자가 올린 실제 작업파일(JSON)을 그대로 툴에 주입 + 프리텐다드/페이퍼로지 폰트를 강제로 로드시켜, **미리보기와 다운로드를 같은 페이지에서 나란히 생성해 비교**
  - 미리보기는 완벽, 다운로드만 잘림 → 원인이 html-to-image 렌더 차이임을 눈으로 확정
  - (교훈: **재현이 안 되면 사용자 환경과 똑같은 조건(실제 데이터 + 실제 폰트)을 강제로 만들어라.** 이 한 번의 제대로 된 재현이 앞의 다섯 번보다 값졌다)

- **6차 — 근본 수정 (성공)**
  - 제품명 상자의 강제 잘림(`overflow:hidden`) 제거 → 다운로드가 살짝 넓게 그려도 안 잘림
  - 뱃지는 **원래대로 이름 옆(인라인)에 유지**하되, 이름 상자에 10% 여유 폭을 확보해 뱃지를 그만큼 오른쪽으로 밀어둠 → 다운로드가 글자를 넓게 그려도 그 여유 안에 들어와 뱃지 밑으로 안 삐져나감
  - 이름이 길면 글자 크기를 반응형으로 자동 축소(export 여유 마진 포함), 옆에 두기엔 너무 작아질(30px 미만) 극단적 경우에만 뱃지를 윗줄로 이동
  - 스토어명은 헤더 폭의 38%로 상한 + 폰트 로드 후 재조정
  - 실제 폰트 환경에서 다운로드 결과물까지 브라우저로 최종 검증
  - (추가 교훈: html-to-image는 **요소 위치는 화면 기준으로 두고 글자만 넓게 그려서**, 옆 요소 밑으로 텍스트가 삐져나가 겹침 → 옆 요소가 있으면 텍스트 상자에 미리 여유 폭을 확보해 둘 것)

- **7차 — 버그 고치다 기존 로직까지 훼손한 것을 사용자 지적으로 재점검 (성공)**
  - 6차에서 "이름이 길면 뱃지를 무조건 윗줄로" 규칙을 넣었는데, 이건 **원래 있던 반응형 동작을 멋대로 바꿔버린 것**이었음 (원래: 글자 크기에 따라 옆에 들어가면 옆에, 안 되면 위로)
  - 사용자가 "왜 또 거지같이 바꿔놨냐, 원래 그런 설정이었다"고 지적 → 원래 동작으로 복원
  - **뱃지는 이름 옆(인라인) 유지가 기본**, 이름이 길면 글자 크기를 반응형 축소해 옆에 맞춤, 옆에 두기엔 너무 작아질(30px 미만) 극단적 경우에만 윗줄로
  - export 겹침은 뱃지를 올리는 대신 **이름 상자에 여유 폭 10% 확보**로 해결
  - (교훈: **버그를 고칠 때 그 근처의 잘 작동하던 로직을 새 규칙으로 덮어쓰지 말 것.** 기존 동작은 사용자가 의도해 만든 사양일 수 있다. 고정 임계값(예: "80% 넘으면")으로 기존 반응형 로직을 대체하면, 증상은 사라져도 사용자가 쓰던 동작이 사라진다)

### 교과서 (다른 이미지 생성 도구에 그대로 적용)
- **화면 캡처/이미지 export는 화면 렌더와 다르다** — 특히 웹폰트가 CORS로 임베드 안 되면 폴백폰트로 폭이 달라져 잘림. **export 검증은 반드시 export 결과물 자체를 실제 폰트 환경에서 확인**
  (기술: html-to-image·dom-to-image·html2canvas 계열 공통. `getFontEmbedCSS` 실패 로그를 흘려보지 말 것)
- **"딱 맞게 자르는" UI(overflow:hidden + 콘텐츠 폭 상자)는 export의 미세 폭 증가에 취약** — 여유 마진 + 강제 잘림 제거로 견고하게
- **측정값과 실제 렌더가 다를 수 있다** — `scrollWidth`류 측정은 줄바꿈·overflow 설정에 따라 거짓음성. 스크린샷으로 눈으로 확인
- **테스트 환경 통과 ≠ 사용자 환경 통과** — 폰트·데이터가 다르면 재현 자체가 안 된다. 사용자 실제 데이터와 폰트를 강제 주입해 동일 조건을 만들 것
- **버그 수정이 기존 동작을 훼손하지 않게 할 것** — 증상만 없애려고 근처의 잘 작동하던 로직을 새 규칙으로 덮어쓰면, 사용자가 의도해 만든 사양이 사라진다. 고칠 땐 "이 동작이 원래 의도된 것인지"부터 확인하고, 기존 로직은 최대한 보존한 채 최소 침습으로 고칠 것

### 🔁 사용자가 같은 증상을 반복해서 호소할 때 (대응 원칙)
> 이번에 "고쳤다"→"또 잘린다"가 6번 반복되며 사용자를 지치게 했다. 반복 신고는 **내 진단이 틀렸다는 강한 신호**다. 아래 순서로 임한다.

- **1. "저번 수정이 덜 됐나 보다"라고 단정하지 말 것** — 같은 부분만 또 들여다보는 것이 가장 흔한 함정. 재신고는 "그 부분이 원인이 아닐 수 있다"는 뜻으로 받아들이고, **인접 코드 전체를 처음부터 다시 의심**한다
- **2. 이미 정상인 신호부터 확인** — 예: "글자가 잘린다"는데 글자 내용·개수·위치는 정상이라면, "글자를 그리는 코드는 정상, 자르는(크기/폭) 코드가 범인"이라는 뜻. 정상 신호가 가리키는 반대편을 볼 것
- **3. 사용자의 표현을 문자 그대로 신뢰할 것** — 이번엔 "미리보기랑 다운로드가 다르다"는 반복된 지적이 정답이었는데, 내가 계속 미리보기(화면)만 검증해서 헛다리를 짚었다. 사용자가 구분해서 말하는 조건(A는 되고 B는 안 됨)은 그 자체가 원인의 위치를 알려주는 단서다
- **4. 스크린샷·측정값을 맹신하지 말고 재현하라** — 간접 증거(측정 `clipped:false`, 화면 스샷)로 판단이 안 서면, **사용자 실제 데이터 + 실제 폰트로 동일 조건을 강제 재현**해 결과물 자체를 눈으로 볼 것. 이번에도 실제 폰트를 심어 재현한 5차에서야 원인이 보였다
- **5. 못 고쳤으면 솔직히, 왜 놓쳤는지까지 설명** — "이번엔 됐다"를 남발하지 말고, 검증한 조건(무엇을·어떤 환경에서 확인했는지)을 함께 밝혀 신뢰를 회복할 것
