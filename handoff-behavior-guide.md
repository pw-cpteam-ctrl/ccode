## 작업 시 행동 지침

> 이 레포의 모든 프로젝트(폴더별로 분리된 각 도구)에서 공통으로 참고하는
> 행동 지침. 프로젝트별 기능 스펙은 각 폴더의 `PLAN.md`/`README.md`를 참고.

1. **데이터 유실 관련은 최우선으로 다룰 것** — 저장/캐싱 로직을 바꿀 땐 "새로고침해도 안전한가"를 항상 먼저 검증하고, 팀 공유(백업/인수인계) 시나리오도 같이 고려할 것. 모든 입력 요소(슬라이더·텍스트박스 등)는 **값 변경 → 화면 반영 → 저장** 3단계가 빠짐없이 있는지 확인할 것.

2. **"완료했다"고 말하기 전 실제 브라우저 동작으로 검증할 것** — 코드 리뷰만으로 끝내지 말고 Playwright 등으로 직접 조작해보고, 가능하면 배포 후 라이브 반영까지 확인한 뒤 보고할 것. 단, 네이티브 파일 선택창·실제 로그인 세션처럼 Playwright로 스크립트 불가능한 부분은 그 한계를 명확히 밝히고, 가능한 내부 로직만이라도 모킹해서 검증할 것.

3. **요청받은 내용과 실제 작업 내용이 정확히 일치하는지 스스로 재확인할 것** — 애매하면 "이건 이런 의미로 요청하신 게 맞나요"라고 먼저 확인.

4. **작업 시작 전 비용/시간이 크게 갈리는 선택지가 있으면 먼저 비교해서 제시할 것.**

5. **버그가 발생하면 얼버무리지 말고 정확한 기술적 원인을 설명할 것** — "왜 미리 못 잡았는지"까지 포함해서.

6. **모든 소통(채팅, 진행 멘트, 커밋 메시지, 코드 주석)은 한국어로 통일할 것.**

7. **git 커밋/푸시 등 되돌리기 번거로운 작업은 항상 사전 확인 후 진행할 것.** 이 레포는 별도 브랜치 없이 **`main`에 직접 push**하는 방식으로 운영하니, 임의로 새 브랜치를 만들지 말 것.

8. **기존에 잘 작동하는 코드는 최대한 재사용하고, 기능 범위가 불필요하게 커지지 않도록 스코프를 관리할 것.**

9. **사용자·참가자 데이터가 포함된 로그·결과를 저장할 땐 민감정보 포함 여부를 먼저 판단할 것** — 필요하면 비공개 저장소 사용 여부, 저장 범위(원본 포함 vs 결과만)를 먼저 확인하고 진행할 것.

10. **여러 프로젝트를 병행할 땐 폴더 + 계획 문서(`PLAN.md` 등)로 분리해서, 사용자가 준비해야 하는 막힌 부분과 지금 가능한 부분을 구분해 먼저 진행할 것.**

11. **사용자는 기능 자체보다 UI/인터페이스의 조작 편의성을 중요하게 여긴다** — 슬라이더/드래그 핸들/버튼 배치처럼 직접 만지는 조작감, 클릭 수, 시각적 피드백까지 기능 구현과 동일한 비중으로 신경 쓸 것. "동작은 하지만 쓰기 불편한" 상태로 완료 보고하지 말 것.

12. **사용자가 로컬에서 직접 실행할 명령어를 안내할 때는, 필요한 명령어를 실행 순서대로 전부 한 번에 줄 것** — 예를 들어 서버 코드가 있는 프로젝트에서 로컬 코드를 최신화해야 `git pull` 없이 다음 명령(`npm run dev`, `node run.js` 등)만 안내하면 사용자가 그 사이의 필수 단계를 빠뜨리게 된다. 사용자는 각 명령어의 목적을 몰라도 되니, 순서대로 복사해서 그대로 실행하면 되는 완전한 한 세트로 줄 것. 명령어를 나눠서 여러 번에 걸쳐 안내하지 말 것.

## LLM 연동 도구에 한해 (조건부 — 해당 프로젝트에서만 참고)

13. **LLM을 붙이는 도구는 stateless로 설계할 것** — 대화 기억에 의존하면 규칙이 서서히 어긋나는 문제(컨텍스트 드리프트)가 생기니, 매 요청을 완전히 새로 시작하게 만들고 규칙 변경은 파일 수정으로만 반영할 것. 일회성 지시(이번 건 한정)와 영구 규칙은 입력창부터 분리해서 받을 것.

14. **판단이 필요한 부분만 LLM에 맡기고, 검증 가능한 로직은 코드로 처리할 것** — 예: 무작위 추첨처럼 "왜 그 결과가 나왔는지 설명 가능해야 하는" 작업은 LLM이 아니라 코드가 담당해서 신뢰성을 확보할 것.
