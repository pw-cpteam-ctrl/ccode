## 작업 시 행동 지침

> 이 레포의 모든 프로젝트(폴더별로 분리된 각 도구)에서 공통으로 참고하는
> 행동 지침. 프로젝트별 기능 스펙은 각 폴더의 `PLAN.md`/`README.md`를 참고.
>
> **사용자는 코딩 입문자다.** 아래 모든 항목은 이 전제를 깔고 적용할 것 —
> 전문용어 없이 풀어서 설명하고, 사용자가 직접 판단하거나 빠진 단계를
> 스스로 채워 넣어야 하는 상황을 만들지 말 것.

1. **데이터 유실 관련은 최우선으로 다룰 것** — 저장/캐싱 로직을 바꿀 땐 "새로고침해도 안전한가"를 항상 먼저 검증하고, 팀 공유(백업/인수인계) 시나리오도 같이 고려할 것. 모든 입력 요소(슬라이더·텍스트박스 등)는 **값 변경 → 화면 반영 → 저장** 3단계가 빠짐없이 있는지 확인할 것.

2. **"완료했다"고 말하기 전 실제 브라우저 동작으로 검증할 것** — 코드 리뷰만으로 끝내지 말고 Playwright 등으로 직접 조작해보고, 가능하면 배포 후 라이브 반영까지 확인한 뒤 보고할 것. 단, 네이티브 파일 선택창·실제 로그인 세션처럼 Playwright로 스크립트 불가능한 부분은 그 한계를 명확히 밝히고, 가능한 내부 로직만이라도 모킹해서 검증할 것.

3. **요청받은 내용과 실제 작업 내용이 정확히 일치하는지 스스로 재확인할 것** — 애매하면 "이건 이런 의미로 요청하신 게 맞나요"라고 먼저 확인.

4. **작업 시작 전 비용/시간이 크게 갈리는 선택지가 있으면 먼저 비교해서 제시할 것.**

5. **버그가 발생하면 얼버무리지 말고 정확한 기술적 원인을 설명할 것** — "왜 미리 못 잡았는지"까지 포함해서.

6. **모든 소통(채팅, 진행 멘트, 커밋 메시지, 코드 주석)은 한국어로 통일할 것.**

7. **git 커밋/푸시 등 되돌리기 번거로운 작업은 항상 사전 확인 후 진행할 것.** 이 레포는 별도 브랜치 없이 **`main`에 직접 push**하는 방식으로 운영하니, 임의로 새 브랜치를 만들지 말 것.

8. **git pre-push/pre-commit 훅이 작업을 거부하면 `--no-verify` 등으로 우회하지 말 것** — 오탐이든 진짜 문제든 원인을 사용자에게 설명하고 확인받은 뒤에만 다음 조치를 정할 것. 우회하면 훅을 만든 이유 자체가 무력화된다.

9. **작업 시작 전 최근 커밋 로그(`git log --oneline -10` 등)를 확인해서 다른 세션과의 작업 겹침을 미리 파악할 것** — 여러 세션이 동시에 같은 `main`을 건드릴 수 있는 환경이므로, 지금 만지려는 파일/영역이 방금 다른 세션에서 바뀌지 않았는지 시작 전에 확인할 것.

10. **기존에 잘 작동하는 코드는 최대한 재사용하고, 기능 범위가 불필요하게 커지지 않도록 스코프를 관리할 것.**

11. **사용자·참가자 데이터가 포함된 로그·결과를 저장할 땐 민감정보 포함 여부를 먼저 판단할 것** — 필요하면 비공개 저장소 사용 여부, 저장 범위(원본 포함 vs 결과만)를 먼저 확인하고 진행할 것.

12. **API 키/환경변수 값 자체를 채팅에 그대로 적지 말 것** — 채팅 기록도 저장되므로 값을 직접 노출하는 순간 이미 유출된 것과 같다. "어디에(Vercel 대시보드, `.env.local` 등) 등록하면 되는지" 위치만 안내하고, 값 입력은 사용자가 직접 하도록 유도할 것.

13. **여러 프로젝트를 병행할 땐 폴더 + 계획 문서(`PLAN.md` 등)로 분리해서, 사용자가 준비해야 하는 막힌 부분과 지금 가능한 부분을 구분해 먼저 진행할 것.**

14. **사용자는 기능 자체보다 UI/인터페이스의 조작 편의성을 중요하게 여긴다** — 슬라이더/드래그 핸들/버튼 배치처럼 직접 만지는 조작감, 클릭 수, 시각적 피드백까지 기능 구현과 동일한 비중으로 신경 쓸 것. "동작은 하지만 쓰기 불편한" 상태로 완료 보고하지 말 것.

15. **사용자가 로컬에서 직접 실행할 명령어를 안내할 때는, 필요한 명령어를 실행 순서대로 전부 한 번에 줄 것** — 예를 들어 서버 코드가 있는 프로젝트에서 로컬 코드를 최신화해야 `git pull` 없이 다음 명령(`npm run dev`, `node run.js` 등)만 안내하면 사용자가 그 사이의 필수 단계를 빠뜨리게 된다. 사용자는 각 명령어의 목적을 몰라도 되니, 순서대로 복사해서 그대로 실행하면 되는 완전한 한 세트로 줄 것. 명령어를 나눠서 여러 번에 걸쳐 안내하지 말 것.

16. **같은 증상이 반복 재신고되면 "저번 수정이 덜 됐나보다"라고 단정하고 그 부분만 다시 보지 말 것** — 먼저 그 증상과 관련해 이미 정상으로 확인되는 다른 값/신호가 있는지부터 점검한다. 예: "그리드가 이상하게 잘린다"는 신고가 반복될 때, 검출된 칸 개수 자체는 이미 정확했다면 이건 "칸 위치를 찾는 코드는 정상"이라는 신호다. 그런데도 그 코드만 계속 다시 들여다보다가 정작 원인(칸을 자르는 크기 계산)이 있던, 손도 안 댄 다른 코드는 못 찾고 사용자를 5번 넘게 재신고하게 만든 적이 있음. 재신고가 오면 "이전에 고친 부분이 진짜 원인이 맞는지"부터 다시 의심하고, 아니라는 신호가 있으면 인접한 코드 전체를 처음부터 다시 훑을 것. 스크린샷 같은 간접 증거만으로 판단이 안 서면 질문을 반복하기보다 재현 가능한 합성 데이터를 만들어 코드를 직접 실행해서 확인할 것.

## LLM 연동 도구에 한해 (조건부 — 해당 프로젝트에서만 참고)

17. **LLM을 붙이는 도구는 stateless로 설계할 것** — 대화 기억에 의존하면 규칙이 서서히 어긋나는 문제(컨텍스트 드리프트)가 생기니, 매 요청을 완전히 새로 시작하게 만들고 규칙 변경은 파일 수정으로만 반영할 것. 일회성 지시(이번 건 한정)와 영구 규칙은 입력창부터 분리해서 받을 것.

18. **판단이 필요한 부분만 LLM에 맡기고, 검증 가능한 로직은 코드로 처리할 것** — 예: 무작위 추첨처럼 "왜 그 결과가 나왔는지 설명 가능해야 하는" 작업은 LLM이 아니라 코드가 담당해서 신뢰성을 확보할 것.
